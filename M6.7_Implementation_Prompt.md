# OpenFEP M6.7 Implementation Prompt

## Milestone

M6.7: Closed-Loop Evaluation

## Goal

Prove that M6 is more than observability: outcome feedback must influence next-turn cognitive state, affective state, prompt capsule, memory update signal, or meta-control guidance in a measurable and deterministic way.

M6.7 must use trace-based evaluation and ablation, not informal examples.

## Work Principles

- Closed-loop claims require current-run evidence.
- Correlation is not enough; use A/B or ablation comparisons.
- Existing M5 dialogue behavior must remain stable under default settings.
- Evaluation scenarios should be small, deterministic, and explainable.

## Required Code Changes

### 1. Add closed-loop evaluation script

Add:
- `scripts/run_m67_closed_loop_evaluation.py`

The script must run deterministic scenarios:
- low-margin ambiguity
- high-conflict dialogue
- affective recovery after repair
- memory interference
- prompt overload
- outcome failure

For each scenario, compare:
- M6 loop enabled
- ablated loop disabled or guidance disabled

Output:
- `artifacts/m67_closed_loop_evaluation.json`
- optional trace files under `artifacts/m67_traces/`

### 2. Add scenario fixtures

Add:
- `segmentum/dialogue/m67_scenarios.py` or test-local fixtures

Each scenario must define:
- input turns
- seed
- expected trigger
- expected state/guidance/prompt delta
- expected affective-state delta where relevant
- forbidden false-positive claims

### 3. Add evaluation checks

Metrics should include:
- next-turn gap state change
- next-turn affective state change
- next-turn meta-control guidance change
- prompt capsule guidance field change
- memory update signal or outcome label propagation
- selected action stability or intentional guidance effect
- trace explainability fields present
- persona-scoped conscious artifact isolation
- `Self-consciousness.md` slow-update evidence if consolidation is enabled

### 4. Preserve default behavior

If M6 loop is not explicitly enabled, M5.3/M5.6 behavior should remain compatible.

## Boundary Constraints

- Do not claim causal closure without ablation.
- Do not require LLM calls for acceptance.
- Do not tune scenarios by changing core policy scoring.
- Do not hide failed scenarios; artifact must report pass/fail per scenario.
- Do not let trace-only changes count as closed-loop behavior unless state, prompt, memory, or guidance changes next turn.
- Do not use stale artifacts as acceptance evidence.

## Required Tests

Add:
- `tests/test_m67_closed_loop_evaluation.py`

Cover:
- each scenario fixture is deterministic
- outcome event can influence next-turn state or guidance
- outcome event can influence next-turn affective state
- cognitive state can influence next-turn prompt capsule
- low-margin ambiguity increases clarification/lower assertion guidance
- high-conflict scenario increases repair/control guidance
- affective recovery scenario increases social safety or warmth after a positive repair outcome
- prompt overload records omitted signals
- ablation differs from enabled loop in at least one expected field
- two personas with the same display name do not share `Self-consciousness.md`
- `Conscious.md` changes do not count as closed-loop behavior unless state, prompt, memory, or guidance changes
- artifact schema is stable

## Acceptance Criteria

- Outcome event influences next-turn `CognitiveStateMVP`, affective state, memory signal, prompt capsule, or meta-control guidance.
- `CognitiveStateMVP` influences next-turn prompt capsule.
- Affective state influences next-turn prompt capsule through compressed guidance, not raw emotional speculation.
- Meta-control guidance changes under low margin, high conflict, or failure.
- Prompt overload causes omitted signals to be recorded.
- Evaluation uses ablation to support closed-loop claims.
- Conscious artifacts are persona/session scoped and cannot be used as the only proof of closure.
- `Self-consciousness.md` updates, if tested, require slow-consolidation evidence rather than a single-turn rewrite.
- Existing M5.3 dialogue tests remain stable.
- Trace can explain why a path was selected and how feedback changed the next turn.

## Deliverables

- `scripts/run_m67_closed_loop_evaluation.py`
- scenario fixtures
- `tests/test_m67_closed_loop_evaluation.py`
- `artifacts/m67_closed_loop_evaluation.json`
- optional `artifacts/m67_traces/*.jsonl`

2. Which ablation deltas prove closed-loop behavior.
3. Which fields changed on the next turn.
4. Which tests passed.
