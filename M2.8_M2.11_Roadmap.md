# OpenFEP M2.8-M2.11 Roadmap

## 总体目标

M2.8 到 M2.11 的任务不是继续扩张复杂世界本身，而是为进入 M3 做稳健准备：先补齐主循环中的认知底盘，再建立最小多环境与迁移基准，再用统计验证证明人格/叙事机制真的起作用，最后把实验、artifact、回归和兼容性全部固化。  

路线顺序：

1. M2.8：补闭环  
2. M2.9：补环境  
3. M2.10：补证据  
4. M2.11：补工程与准入  

---

## M2.8: Attention Bottleneck

### 标题

M2.8: Attention Bottleneck（注意力瓶颈）  
“此刻什么最值得被处理？”

### FEP 对应概念

精度加权与选择性信息通道（Precision Weighting / Attentional Selection）。

### 为什么是这一步

当前系统已经具备感知、行动、记忆、sleep、人格等部件，但还没有真正位于主循环中的“资源限制层”。没有注意力瓶颈，Agent 更像是对所有信号统一处理，而不是在有限带宽下选择什么进入有效访问。  

这一步的意义不是做“更智能的感知”，而是补上 `感知 -> 注意 -> 行动 -> 学习` 链条里缺失的中间层，为后续的功能性意识访问、跨环境稳定性和复杂世界测试提供必要结构。

### 工程内容

- 新增 `segmentum/attention.py`
- 在 `segmentum/agent.py` 的主感知路径中接入 `AttentionBottleneck`
- 为 attention 增加 on/off 开关、trace、序列化、回放兼容
- 将 attention trace 写入 diagnostics 和 runtime trace
- 新增 benchmark 脚本对 attention on/off 进行同 seed 对照

### 代码文件清单

- `segmentum/attention.py`
- `segmentum/agent.py`
- `segmentum/runtime.py`
- `segmentum/types.py`
- `tests/test_m28_attention.py`
- `scripts/run_m28_attention_benchmark.py`

### 测试文件清单

- `tests/test_m28_attention.py`
  - salience scoring deterministic
  - top-k selection stable
  - filtered observation behavior correct
  - serialization round-trip
- 必须保证现有回归测试继续通过，尤其是：
  - `tests/test_runtime.py`
  - `tests/test_determinism.py`
  - `tests/test_baseline_regressions.py`

### Artifact 文件

- `artifacts/m28_attention_on_off.json`
- 可选补充：
  - `artifacts/m28_attention_trace_sample.json`

### 完成门槛

- attention on/off 可以显式切换
- 同 seed 结果可复现
- top-k 关键通道命中率 >= 0.80
- attention-on 条件下 conditioned prediction error 相对 off 下降 >= 10%
- survival performance 不低于 off 条件的 95%
- 全量测试继续通过

### 风险点

- 主循环接入可能破坏现有 determinism
- attention 如果做得太强，可能造成关键信号漏检
- attention 如果只是 trace 层记录而不影响主计算路径，会沦为“概念存在”

## M2.9: Minimal Narrative Worlds & Transfer

### 标题

M2.9: Minimal Narrative Worlds & Transfer（最小叙事世界与跨环境迁移）  
“我学到的东西能带到别处吗？”

### FEP 对应概念

生成模型的情境泛化与迁移（Generalization / Transfer Across Contexts）。

### 为什么是这一步

当前系统的大多数能力是在单一 toy 环境里验证的，这不足以说明记忆、人格、叙事先验是可迁移的结构，还是仅仅在一个环境中形成的局部适应。  

进入 M3 之前，不需要先做复杂世界细节，但必须先建立最小多环境基准，让系统至少能被跨情境测试。

### 工程内容

- 新增 `segmentum/narrative_world.py`
- 在 `data/worlds/` 下增加至少 3 个最小 world config
- 扩展 `segmentum/runtime.py`，支持 world config 加载与运行
- 通过 `NarrativeIngestionService` 将世界事件自动转成 narrative episodes 注入 agent
- 建立第一版 transfer benchmark

### 代码文件清单

- `segmentum/narrative_world.py`
- `segmentum/runtime.py`
- `segmentum/narrative_ingestion.py`
- `data/worlds/foraging_valley.json`
- `data/worlds/predator_river.json`
- `data/worlds/social_shelter.json`
- `tests/test_m29_narrative_world.py`
- `tests/test_m29_transfer.py`
- `scripts/run_m29_transfer_benchmark.py`

### 测试文件清单

- `tests/test_m29_narrative_world.py`
  - world config round-trip
  - event schedule deterministic
  - action effects deterministic
  - generated episodes schema valid
- `tests/test_m29_transfer.py`
  - train/eval protocol valid
  - fresh vs transfer baseline comparable
  - transfer metrics schema stable

### Artifact 文件

- `artifacts/m29_world_rollout_foraging_valley.json`
- `artifacts/m29_world_rollout_predator_river.json`
- `artifacts/m29_world_rollout_social_shelter.json`
- `artifacts/m29_transfer_benchmark.json`

### 完成门槛

- 至少 3 个 world config 可稳定运行
- 至少 2 个迁移方向跑通
- 迁移后 agent 相对 fresh baseline 至少满足以下之一：
  - survival score 提升 >= 8%
  - conditioned prediction error 降低 >= 10%
  - first-50-cycle regret 降低 >= 10%
- 所有实验可固定 seed 复现

### 风险点

- 如果世界设计过复杂，会提前把 M3 的工作挤进来
- 如果世界设计过简单，迁移结果会失去解释力
- 如果事件注入与 runtime 主循环耦合过深，后续维护成本会变高

## M2.10: Personality & Narrative Validation

### 标题

M2.10: Personality & Narrative Validation（人格与自我叙事验证）  
“我是否真的形成了稳定的行为风格？”

### FEP 对应概念

慢变量人格参数、自我模型一致性、长期行为约束（Stable Latent Traits / Self-Model Coherence）。

### 为什么是这一步

当前系统已有 `PersonalityProfile`、`NarrativePriors`、`IdentityNarrative`，但证据主要还是案例型和单场景断言。  

进入 M3 之前，必须把“看起来像人格”和“看起来像自我叙事”提升为“可复现实验中有统计支持的行为结构”，否则后续在复杂世界中看到的差异都难以解释。

### 工程内容

- 定义固定 personality profile 协议
- 在多个最小世界中进行重复 rollout
- 采集行为指标：
  - caution_rate
  - exploration_rate
  - seek_contact_rate
  - action_entropy
  - survival_score
- 实现最小 ANOVA / effect size 计算
- 增加长程一致性测试
- 验证 narrative/self state 是否持续影响行为，而不仅仅记录在日志里

### 代码文件清单

- `segmentum/self_model.py`
- `segmentum/agent.py`
- `tests/test_m210_personality_validation.py`
- `tests/test_m210_longitudinal_stability.py`
- `scripts/run_m210_personality_anova.py`
- `scripts/run_m210_stability_benchmark.py`

### 测试文件清单

- `tests/test_m210_personality_validation.py`
  - profile factory deterministic
  - metrics extraction deterministic
  - ANOVA helper correctness
  - group differences detectable
- `tests/test_m210_longitudinal_stability.py`
  - long-run personality no collapse
  - narrative bias persists across repeated runs
  - same profile under same seed remains stable

### Artifact 文件

- `artifacts/m210_personality_anova.json`
- `artifacts/m210_longitudinal_stability.json`
- `artifacts/m210_profile_behavior_summary.json`

### 完成门槛

- 至少 3 个行为指标满足 `p < 0.05`
- 至少 2 个指标 effect size 达到中等水平，如 `eta_squared >= 0.06`
- 相同人格 profile 在重复实验中的行为分布稳定
- 长程运行后不存在明显人格塌缩或单策略锁死
- narrative/self state 对行为的偏置作用可重复测得

### 风险点

- 统计显著但行为不可解释
- profile 设计不合理导致结果被环境主效应吞没
- 人格参数可能只在短期有效，长期会漂回中性或塌缩

## M2.11: Pre-M3 Readiness Hardening

### 标题

M2.11: Pre-M3 Readiness Hardening（M3 前工程固化与准入）  
“这个系统已经稳到可以放进更复杂世界了吗？”

### FEP 对应概念

模型稳健性、可复现性、结构化更新纪律（Robustness / Reproducibility / Controlled Scaling）。

### 为什么是这一步

如果没有统一实验入口、artifact schema、快照兼容性和长程回归基线，M3 一旦开始扩世界与社会互动，系统很快会失去可比较性。  

这一里程碑不追求新增认知能力，而是把 M2.8-M2.10 的成果固定下来，确保后续扩展时有稳定底盘。

### 工程内容

- 统一 benchmark 脚本入口
- 固化 artifact schema
- 增加 snapshot 向后兼容检查
- 增加长程 soak regression
- 增加失败模式回归
- 形成 Pre-M3 readiness checklist 和最终准入结论

### 代码文件清单

- `scripts/generate_pre_m3_acceptance_artifacts.py`
- `scripts/run_pre_m3_regressions.py`
- `tests/test_m211_artifact_schema.py`
- `tests/test_m211_snapshot_compatibility.py`
- `tests/test_m211_pre_m3_readiness.py`
- `Pre_M3_Readiness_Checklist.md`

### 测试文件清单

- `tests/test_m211_artifact_schema.py`
  - all benchmark outputs share the same schema
- `tests/test_m211_snapshot_compatibility.py`
  - old snapshots load without failure
  - missing new fields get safe defaults
- `tests/test_m211_pre_m3_readiness.py`
  - readiness gates reflect actual metrics
  - no stale artifact can falsely mark system ready

### Artifact 文件

- `artifacts/pre_m3_attention_summary.json`
- `artifacts/pre_m3_transfer_summary.json`
- `artifacts/pre_m3_personality_summary.json`
- `artifacts/pre_m3_readiness_report.json`

### 完成门槛

- 所有核心 benchmark 可一条命令复现
- artifact 格式统一，可自动比较
- 快照向后兼容策略稳定
- 长程 soak regression 稳定通过
- readiness report 明确结论：
  - attention 主循环已成立
  - 多环境运行已成立
  - transfer benchmark 已成立
  - personality / narrative 统计证据已成立
  - 系统可稳健进入 M3

### 风险点

- 工程收口容易被低估，最后变成大量临时脚本
- artifact schema 如果不统一，M3 前的比较都会失真
- readiness gate 如果设计太松，会导致过早进入 M3

## 跨里程碑依赖关系

- M2.8 是底层依赖，必须先完成
- M2.9 依赖 M2.8 的主循环稳定性
- M2.10 最好在 M2.9 的最小多环境基础上做统计验证
- M2.11 依赖前面三者都已有可复现实验与 artifact

---

## Pre-M3 准入门槛

进入 M3 之前，建议至少满足以下条件：

- 注意力瓶颈已接入主循环，并有 on/off 对照 artifact
- 至少 3 个最小 world config 可稳定运行
- 至少 2 个跨环境迁移任务完成并有量化收益
- 人格差异存在统计显著证据
- 长程运行不出现明显人格塌缩或策略单一化
- 自我叙事 / narrative priors 对行为存在在线约束证据
- 所有核心 benchmark 可固定 seed 复现
- 全量测试保持绿色

---

## 一句话总结

M2.8-M2.11 的职责不是把世界做得更像现实，而是把 Agent 先打磨成“值得放进更复杂世界里测试”的系统。  
完成这四个里程碑后，再进入 M3 做更细的虚拟世界、长期交互和更强的类人一致性，会稳得多。
