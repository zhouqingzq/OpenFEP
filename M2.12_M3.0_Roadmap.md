# OpenFEP M2.12-M3.0 Roadmap

## Overall Goal

From `M2.12` onward, the project should stop optimizing only for a stronger toy-world prototype and start closing the gap toward a long-running digital organism with:

- persistent multi-source perception,
- scarce attention and functional conscious access,
- autonomous homeostatic self-maintenance,
- unified self-narrative and identity commitments,
- socially grounded cross-context consistency,
- open-world action through controlled tool and environment adapters,
- strict milestone-by-milestone audit acceptance.

The ordering below is deliberate:

1. `M2.12` first opens the I/O bottleneck.
2. `M2.13` turns access into a real global control surface.
3. `M2.14` makes survival maintenance continuous instead of reactive.
4. `M2.15` binds long-term behavior to an explicit self narrative.
5. `M2.16` introduces persistent others and social continuity.
6. `M2.17` opens constrained real-world action loops.
7. `M2.18` hardens lifelong learning and anti-collapse safeguards.
8. `M3.0` validates the whole stack under a closed-loop life trial.

All milestones must use the strict audit process defined in `Strict_Audit_Framework.md`.

## M2.12: Embodied I/O Bus and World Adapters

### Title

`M2.12: Embodied I/O Bus and World Adapters`

`What can this organism actually sense and actually affect?`

### FEP Mapping

Generative-model grounding through richer observation/action channels.

### Why this milestone

The current runtime mainly senses a six-channel simulated world plus optional host-process telemetry. That is enough for prototype predictive coding, but not for the target organism-level architecture. We need a unified I/O substrate before we can claim persistent perception, action, or survival in anything broader than the toy world.

### Engineering scope

- Introduce a typed `PerceptionBus` and `ActionBus` abstraction.
- Add adapter interfaces for:
  - simulated world input,
  - host telemetry input,
  - narrative event input,
  - future tool/event adapters.
- Add action-effect acknowledgment so outputs are measured, not just emitted.
- Separate raw signals, normalized signals, and inferred latent causes in traces.
- Add explicit channel provenance and confidence.

### Files to expect

- `segmentum/io_bus.py`
- `segmentum/runtime.py`
- `segmentum/types.py`
- `segmentum/environment.py`
- `segmentum/narrative_world.py`
- `tests/test_m212_io_bus.py`
- `tests/test_m212_adapter_roundtrip.py`
- `scripts/generate_m212_acceptance_artifacts.py`

### Artifacts

- `artifacts/m212_io_bus_schema.json`
- `artifacts/m212_adapter_trace_sample.json`
- `reports/m212_acceptance_report.json`

### Acceptance gates

- At least 4 input sources can be ingested through one unified bus schema.
- At least 3 output channels produce effect acknowledgments in trace records.
- The same seed and adapter config are replayable deterministically.
- Missing adapters degrade gracefully without crashing the lifecycle.
- All new schema fields survive snapshot round-trip.

### Audit focus

- False embodiment: adapters must affect runtime behavior, not only logging.
- Hidden coupling: adapter failures must be attributed and quarantined.
- Trace sufficiency: every bus event must be reconstructible post-hoc.

## M2.13: Global Access Workspace

### Title

`M2.13: Global Access Workspace`

`Which contents become globally available for control, report, and learning?`

### FEP Mapping

Precision-gated broadcast and functional conscious access.

### Why this milestone

Current attention traces and inner speech exist, but conscious access is not yet a first-class global control structure. To reach the stated goal, selected contents must be able to influence policy, narration, memory write priority, and self-maintenance in one shared bottleneck.

### Engineering scope

- Add `GlobalWorkspaceState` with capacity limits and broadcast winners.
- Make attention outputs compete for workspace admission.
- Gate policy explanation, episodic write strength, and self-report through workspace access.
- Distinguish:
  - latent processing,
  - attended processing,
  - globally accessible contents.
- Add workspace persistence and replay.

### Files to expect

- `segmentum/workspace.py`
- `segmentum/attention.py`
- `segmentum/agent.py`
- `segmentum/runtime.py`
- `tests/test_m213_global_workspace.py`
- `tests/test_m213_access_causality.py`
- `scripts/generate_m213_acceptance_artifacts.py`

### Artifacts

- `artifacts/m213_workspace_broadcast_trace.jsonl`
- `artifacts/m213_access_ablation.json`
- `reports/m213_acceptance_report.json`

### Acceptance gates

- Workspace capacity constraints are enforced deterministically.
- Removing workspace broadcast degrades at least 2 downstream functions:
  - policy selection quality,
  - self-report fidelity,
  - episodic storage prioritization,
  - surprise attribution quality.
- Broadcast contents are visible in trace, explanation, and learning records.
- Non-broadcast contents do not appear in conscious report.

### Audit focus

- No fake workspace: access must causally change decisions.
- No unlimited bandwidth: capacity and competition must bind.
- No report leakage: unbroadcast contents cannot appear in self-report.

## M2.14: Autonomous Homeostasis and Survival Scheduler

### Title

`M2.14: Autonomous Homeostasis and Survival Scheduler`

`Can the organism maintain itself without an external operator telling it what matters now?`

### FEP Mapping

Active inference for self-maintenance under scarce resources.

### Why this milestone

The current agent already tracks energy, fatigue, stress, temperature, and sleep triggers. But it still behaves like a cycle-based benchmark subject, not a continuously self-maintaining organism. We need a scheduler that continuously arbitrates maintenance, exploration, repair, and rest.

### Engineering scope

- Add a persistent maintenance agenda and interrupt system.
- Promote sleep, repair, trace compaction, and context-budget protection into first-class survival acts.
- Add multi-timescale homeostatic debt:
  - acute,
  - short-horizon,
  - chronic.
- Add explicit self-maintenance policies for memory pressure, token pressure, and runtime degradation.
- Introduce bounded background loops for maintenance tasks.

### Files to expect

- `segmentum/homeostasis.py`
- `segmentum/runtime.py`
- `segmentum/agent.py`
- `segmentum/self_model.py`
- `tests/test_m214_homeostasis_scheduler.py`
- `tests/test_m214_maintenance_interrupts.py`
- `scripts/generate_m214_acceptance_artifacts.py`

### Artifacts

- `artifacts/m214_homeostasis_timeline.json`
- `artifacts/m214_interrupt_recovery.json`
- `reports/m214_acceptance_report.json`

### Acceptance gates

- The agent can defer low-priority tasks to preserve survival budgets.
- At least 3 maintenance loops can trigger without operator input.
- Under induced resource stress, survival duration improves relative to no-scheduler baseline.
- Chronic debt accumulation is observable and partially reversible.
- Long-run execution avoids collapse into a single repetitive maintenance action.

### Audit focus

- Maintenance must not be scripted one-off behavior.
- Scheduler must trade off across competing debts, not only fire thresholds.
- Recovery must be measurable after stress injections.

## M2.15: Unified Self-Narrative and Identity Commitments

### Title

`M2.15: Unified Self-Narrative and Identity Commitments`

`What kind of being am I, and how does that constrain my future action?`

### FEP Mapping

Long-timescale self-model coherence and identity-stabilized policy priors.

### Why this milestone

There is already personality modulation and identity narrative updating, but the current narrative is still mostly descriptive. The next step is to make self-narrative normative: identity claims should constrain action, memory protection, and inconsistency repair.

### Engineering scope

- Add explicit identity commitments and violation detection.
- Separate:
  - autobiographical summary,
  - trait self-model,
  - current chapter,
  - binding commitments.
- Add self-inconsistency prediction error and repair policies.
- Require decision explanations to reference current commitments when relevant.
- Add chapter transition logic with explicit triggers and evidence.

### Files to expect

- `segmentum/self_model.py`
- `segmentum/agent.py`
- `segmentum/metacognitive.py`
- `tests/test_m215_identity_commitments.py`
- `tests/test_m215_narrative_coherence.py`
- `scripts/generate_m215_acceptance_artifacts.py`

### Artifacts

- `artifacts/m215_identity_commitment_trace.json`
- `artifacts/m215_self_inconsistency_repair.json`
- `reports/m215_acceptance_report.json`

### Acceptance gates

- Identity commitments alter action ranking in relevant contexts.
- Self-inconsistency events are explicitly traced and classified.
- The same long-run seed produces coherent chapter progression.
- Narrative summaries remain supported by stored evidence rather than free text only.
- Protected identity-critical memories are preserved across compaction.

### Audit focus

- No decorative narrative: commitments must change behavior.
- No unsupported claims: all self-summary statements need evidence links.
- No silent identity drift: large identity shifts require explicit chapter evidence.

## M2.16: Persistent Others and Social Continuity

### Title

`M2.16: Persistent Others and Social Continuity`

`Can this organism form stable models of other agents across encounters?`

### FEP Mapping

Hierarchical social prediction and cross-episode agent modeling.

### Why this milestone

A human-like digital organism cannot be internally coherent while treating the social world as anonymous noise. This milestone introduces persistent social objects, reputational memory, attachment expectations, and social surprise.

### Engineering scope

- Add `OtherModel` and `SocialMemory`.
- Track counterpart identity, trust, threat, reciprocity, and predictability.
- Let social models affect both action policy and self-narrative.
- Add partner continuity across episodes and worlds.
- Add social rupture and repair mechanics.

- Social expectations transfer across repeated encounters.

## M2.17: Open-World Action and Self-Maintenance Governance

### Title

`M2.17: Open-World Action and Self-Maintenance Governance`

`Can the organism act outside the toy world without breaking itself?`

### FEP Mapping

Controlled active inference over external tools and environments.

### Why this milestone

To move beyond simulation, the agent needs constrained open-world actions. But open action without governance turns the system into an unsafe script runner. This milestone should open real action channels together with policy guardrails, budgeting, and rollback discipline.

### Engineering scope

- Add controlled tool adapters with capability descriptors and costs.
- Add action authorization policies:
  - allowed,
  - review-required,
  - denied.
- Require ex-ante predicted effects and ex-post acknowledgments for every external act.
- Add self-maintenance governance for context, file, and network budgets.
- Add structured rollback or repair attempts for failed external actions.

### Files to expect

- `segmentum/action_registry.py`
- `segmentum/runtime.py`
- `segmentum/self_model.py`
- `segmentum/governance.py`
- `tests/test_m217_governed_action.py`
- `tests/test_m217_external_effect_ack.py`
- `scripts/generate_m217_acceptance_artifacts.py`

### Artifacts

- `artifacts/m217_governance_decisions.json`
- `artifacts/m217_external_action_trace.jsonl`
- `reports/m217_acceptance_report.json`

### Acceptance gates

- Every external action has a declared capability, budget cost, and acknowledgment record.
- Unsafe actions are blocked deterministically by policy.
- Failed actions produce bounded recovery behavior instead of silent drift.
- Governance traces explain why an action was allowed or denied.
- Tool use improves at least one benchmark without increasing uncontrolled failure rate.

### Audit focus

- No hidden side effects outside governed channels.
- No external action without acknowledgment.
- No policy bypass through indirect helper paths.

## M2.18: Lifelong Learning, Anti-Collapse, and Continuity Preservation

### Title

`M2.18: Lifelong Learning, Anti-Collapse, and Continuity Preservation`

`Can the organism keep learning for a long time without forgetting who it is?`

### FEP Mapping

Slow-weight consolidation, continuity protection, and anti-catastrophic collapse.

### Why this milestone

Once open-world action and persistent social models exist, long-run continuity becomes the main risk. The organism needs lifelong learning without catastrophic forgetting, value inversion, action collapse, or narrative fragmentation.

### Engineering scope

- Add continuity-preservation scores and collapse detectors.
- Add memory retirement, rehearsal, and protected identity anchors.
- Add drift budgets for personality, narrative, and policy priors.
- Add explicit anti-collapse interventions when overcompression or overfitting is detected.
- Add long-run restart consistency and continuity audits.

### Files to expect

- `segmentum/memory.py`
- `segmentum/self_model.py`
- `segmentum/runtime.py`
- `tests/test_m218_lifelong_learning.py`
- `tests/test_m218_continuity_preservation.py`
- `scripts/generate_m218_acceptance_artifacts.py`

### Artifacts

- `artifacts/m218_lifelong_soak_summary.json`
- `artifacts/m218_continuity_preservation_trace.json`
- `reports/m218_acceptance_report.json`

### Acceptance gates

- Long-run performance does not collapse into a single dominant action pattern.
- Identity-critical evidence remains recoverable after extended consolidation.
- Restart consistency remains within declared tolerance.
- Personality and self-narrative drift stay within bounded audited windows unless chapter-shift conditions are met.
- Anti-collapse interventions trigger before irreversible degradation in benchmark stress scenarios.

### Audit focus

- No hidden forgetting of protected memories.
- No silent value inversion.
- No soak-run success based on stale artifacts or partial replay.

## M3.0: Closed-Loop Life Trial

### Title

`M3.0: Closed-Loop Life Trial`

`Does the whole organism persist, adapt, explain itself, and remain itself across changing conditions?`

### FEP Mapping

End-to-end hierarchical active inference under long-horizon open conditions.

### Why this milestone

This is the first milestone that should be evaluated as an organism-level integration trial, not a component release.

### Engineering scope

- Run a long-horizon mixed-environment trial combining:
  - embodied inputs,
  - global workspace,
  - homeostasis,
  - self-narrative,
  - persistent others,
  - governed open-world action,
  - lifelong learning.
- Add explicit scenario phases:
  - stable baseline,
  - environmental shift,
  - social rupture,
  - resource deprivation,
  - recovery/reorganization.
- Produce a single machine-readable readiness and organism-trial report.

### Files to expect

- `evals/m30_life_trial_evaluation.py`
- `scripts/run_m30_life_trial.py`
- `tests/test_m30_life_trial.py`
- `reports/m30_life_trial_report.md`
- `reports/m30_life_trial_metrics.json`

### Artifacts

- `artifacts/m30_life_trial_trace.jsonl`
- `artifacts/m30_phase_transition_summary.json`
- `reports/m30_acceptance_report.json`

### Acceptance gates

- The organism completes the trial without uncontrolled collapse.
- It adapts across at least 3 distinct context shifts while preserving continuity.
- It can produce bounded self-reports grounded in workspace-accessible evidence.
- Social, narrative, and maintenance states remain behaviorally coupled.
- Independent audit concludes `PASS` with no unresolved critical findings.

### Audit focus

- End-to-end claims must be supported by trace evidence.
- Success cannot depend on a single lucky seed.
- Recovery must be real, not merely a logging artifact.

## Delivery Discipline for Every Milestone

Every milestone from `M2.12` onward must ship with:

- code,
- tests,
- acceptance artifacts,
- a milestone-specific acceptance report,
- strict audit evidence,
- a rollback or quarantine plan for failed states,
- an updated readiness checklist for the next milestone.

No milestone is considered complete if any of the above is missing.
