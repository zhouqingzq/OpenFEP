# OpenFEP M13.0 Work Prompt

## Milestone

M13.0: MVP-Local Behavioral Pull

## Goal

Add the first M13 drive layer to the real UI chat path. This milestone does
not unify `mvp_loop.py` with the research `CognitiveLoop` stack. Instead, it
creates a small, deterministic, auditable M13 adapter inside the MVP runtime
so Hu Tao can express path traction through the current user-facing chat path.

The target behavior is:

```text
cue -> recalled memory / habit evidence -> candidate reply action
-> behavioral_pull estimate
-> control_guidance patch
-> thinking prompt constraint
-> post-turn auditable state update
```

M13.0 should make repeated, outcome-supported reply paths easier to choose
without turning prompt text into the only cognition layer.

## Architecture Position

M13.0 is an MVP-local bridge:

```text
MVPDialogueRuntime.run_turn
-> conscious plan
-> memory_dynamics guidance
-> retrieval / evidence judgment
-> M13DriveEvaluator
-> M13DriveEvent + control_guidance patch
-> thinking prompt
-> post-reply M13 state patch
-> MVPStateStore save + conversation_log audit
```

This is intentionally smaller than the ideal post-M8 architecture:

```text
CognitiveEventBus -> CognitiveLoop -> CognitiveStateMVP
```

The bridge is acceptable only if it keeps the same ownership discipline:

- M13 modules may calculate and propose.
- The MVP runtime remains the local orchestrator for this path.
- Durable drive state writes must be explicit patches with source, reason,
  confidence, turn id, and previous/new value summaries.
- Prompt-facing output must be compressed guidance, never raw drive state.

## Non-Goals

- Do not unify Path A (`conversation_loop.py`) and Path B (`mvp_loop.py`).
- Do not claim Hu Tao has subjective consciousness, real craving, or real
  pleasure.
- Do not implement boredom, pleasure proxy, tolerance, or UI idle activity
  here.
- Do not replace M11, M12.0, M12.1, or M12.2.
- Do not expose internal terms like "behavioral_pull", "policy precision",
  "free energy", or "candidate cost" in visible replies.
- Do not let the LLM invent behavioral-pull scores.

## Required Work

### 1. Add M13 State Surface To MVP Files

Extend the MVP JSON state defaults with an owned state section such as:

```text
m13_drive_state:
  version
  path_patterns_by_action
  recent_action_trace
  traction_by_action
  cue_precision_by_topic
  relation_path_precision
  last_patch_id
```

Minimum per-action pattern fields:

```text
action
user_id
support_count
success_proxy_count
failure_proxy_count
habit_precision
mean_control_cost_discount
last_seen_turn
source_evidence_ids
status
```

State constraints:

- keep bounded lists; no raw transcript dumps
- retain only compact source refs and summaries
- support JSON load/save through `MVPStateStore`
- be backward-compatible when the file is missing
- key path traction by `(action, user_id)` unless a field is explicitly
  documented as persona-global
- keep topic and retrieval fingerprints separate from relationship-specific
  traction so one user cannot silently train another user's relational path

Hard state bounds:

```text
recent_action_trace: 30 rows
path_patterns_by_action: 200 rows, evict by oldest last_seen_turn
recent_topic_fingerprints: 24 rows
pending_settlements: 8 rows
tolerance_by_path: 64 rows
source_evidence_ids per pattern: 8 refs
```

If a later milestone needs larger state, it must justify the bound change in
its work prompt and add a state-size regression test.

### 3. Implement Behavioral Pull Candidate Evaluation

Build a deterministic evaluator from:

- current `user_text`
- `conscious_plan`
- `memory_dynamics`
- retrieved memories
- `response_style_prior`
- learned conversation habits
- relationship value context
- prior M13 drive state

Candidate actions should come from existing surfaces when available:

```text
answer
ask_question
empathize
clarify
disagree
deflect
self_disclose
abstract_share
truthful_refusal
```

For each candidate, compute bounded components:

```text
habit_precision
cue_precision
relation_precision
memory_support
recent_success_proxy
repetition_penalty
control_cost_discount
behavioral_pull
```

Suggested formula shape:

```text
behavioral_pull =
  habit_precision * w_habit
+ cue_precision * w_cue
+ relation_precision * w_relation
+ memory_support * w_memory
+ recent_success_proxy * w_success
- repetition_penalty * w_repetition
```

All weights must live in a small constants object with rationale comments.
No magic numbers should be scattered through `mvp_loop.py`.

### 4. Patch Prompt-Safe Control Guidance

M13.0 may patch `memory_dynamics.control_guidance` before
`build_thinking_prompt`.

Allowed prompt-facing guidance:

```text
drive_guidance:
  preferred_reply_actions
  discouraged_reply_actions
  action_tendency_reason
  caution
  ordinary_language_policy
```

The guidance must be framed as response tendencies, not as visible inner
states. The thinking prompt may receive:

```text
"A recently useful response pattern is available, but do not repeat it
mechanically if the current user need differs."
```

It must not receive:

```text
"behavioral_pull=0.82"
"habit_precision increased"
"I am compelled to do this"
```

M13 drive guidance is advisory-only. It may raise or lower tendencies, but it
does not own final action selection. The final `reply_action` is still chosen
by the existing thinking/generation path under evidence, identity, safety, and
reply-contract constraints.

Merge precedence is strict:

```text
safety / privacy gates
> evidence_judgment.allowed_reply_actions
> sharing_policy.allow_direct_disclosure / allow_abstract_sharing
> identity_anchored_action and M12/M12.2 reply-policy constraints
> reply_contract hard rules
> M13 drive_guidance preferences
> ordinary style and habit tendencies
```

Implementation requirement:

```text
preferred_reply_actions =
  preferred_reply_actions
  intersect allowed_reply_actions
  intersect actions allowed by sharing_policy and reply_contract
```

M13 must never enable an action blocked by safety, privacy, identity binding,
evidence judgment, or the reply contract.

### 5. Post-Turn State Update

After reply generation and validation, update M13 drive state using proxy
outcomes:

Positive proxies:

- reply validation unchanged
- post-reply observer did not require repair
- expectation result confirmed
- user-facing reply action matched a high-pull candidate and no safety repair
  was triggered
- memory write candidate was accepted with evidence

Negative proxies:

- reply validation had to rewrite the reply
- post-reply observer required repair or self-correction
- safety layer blocked or altered high-risk content
- expectation result violated
- explicit user correction in the next turn, when available

M13.0 only uses current-turn proxies. Stronger next-turn outcome settlement is
deferred to M13.2.

Proxy update constraints:

```text
minimum_success_proxy_confidence: 0.60
maximum_single_turn_habit_precision_delta: 0.05
maximum_single_turn_control_discount_delta: 0.04
decay_for_unseen_patterns_per_turn: small, documented in constants
rollback_window: at least the most recent accepted positive update
```

If the next turn or post-reply observer shows that a positive proxy was wrong,
the implementation must be able to reverse or dampen the most recent
confidence increase rather than permanently encoding a bad habit.

### 6. Deterministic Topic Fingerprint

Define one shared deterministic topic fingerprint helper for all M13
milestones. It must not call an LLM.

Input sources, in priority order:

```text
conscious_plan.memory_search_keywords
memory_dynamics.recall_query.semantic_terms
entity_binding.target_person
entity_binding.pronoun_bindings values
retrieved memory keywords
rough current user_text terms
```

Algorithm:

```text
1. collect candidate terms from the sources above
2. lowercase / casefold
3. trim whitespace and punctuation
4. drop empty terms and common stop terms
5. de-duplicate preserving first source-priority order
6. keep at most 8 terms
7. join with "|"; if empty, use "topic:unknown"
```

The fingerprint is a compact engineering key, not a semantic truth claim.
Tests must cover stability across source ordering and missing fields.

## Integration Points

Preferred files:

```text
segmentum/dialogue/runtime/m13_drive.py
segmentum/dialogue/runtime/mvp_loop.py
tests/test_m13_0_behavioral_pull.py
```

`mvp_loop.py` should remain orchestration glue. Most scoring, normalization,
patch creation, and summarization belongs in `m13_drive.py`.

M13.0 should also publish M13 dict events into the same MVP-local bus list used
by `run_turn`. If a `CognitiveEventBus` is already present for M12 events in
the same turn, mirror or adapt M13 events into that bus as trace-compatible
events when practical. This is not required to unify Path A and Path B, but it
reduces future merge debt.

Add a short technical-debt note near the M13 module boundary:

```text
M13 is MVP-local and intentionally parallel to Path A drive/process-valence
surfaces. Future architecture work should reconcile names and state ownership
with segmentum/drives.py rather than treating both as independent permanent
systems.
```


The multi-turn scenario should run 8-10 deterministic turns with repeated
same-class requests and assert that path traction rises while the implementation
still records M13 guidance as advisory-only.

## Acceptance Criteria

M13.0 is complete when the MVP chat path can compute and log behavioral pull
from bounded memory and habit evidence, use it to influence reply-action
guidance, and persist a compact audited drive-state patch without exposing
internal scores or claiming subjective compulsion.
