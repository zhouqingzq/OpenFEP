# OpenFEP M13.0 Work Prompt

## Milestone

M13.0: MVP-Local Behavioral Pull

## Goal

Add the first M13 drive layer to the real UI chat path. This milestone does
not unify `mvp_loop.py` with the research `CognitiveLoop` stack. Instead, it
creates a small, deterministic, auditable M13 adapter inside the MVP runtime
so Hu Tao can express path traction through the current user-facing chat path.

The target behavior is:

```text
cue -> recalled memory / habit evidence -> candidate reply action
-> behavioral_pull estimate
-> control_guidance patch
-> thinking prompt constraint
-> post-turn auditable state update
```

M13.0 should make repeated, outcome-supported reply paths easier to choose
without turning prompt text into the only cognition layer.

## Architecture Position

M13.0 is an MVP-local bridge:

```text
MVPDialogueRuntime.run_turn
-> conscious plan
-> memory_dynamics guidance
-> retrieval / evidence judgment
-> M13DriveEvaluator
-> M13DriveEvent + control_guidance patch
-> thinking prompt
-> post-reply M13 state patch
-> MVPStateStore save + conversation_log audit
```

This is intentionally smaller than the ideal post-M8 architecture:

```text
CognitiveEventBus -> CognitiveLoop -> CognitiveStateMVP
```

The bridge is acceptable only if it keeps the same ownership discipline:

- M13 modules may calculate and propose.
- The MVP runtime remains the local orchestrator for this path.
- Durable drive state writes must be explicit patches with source, reason,
  confidence, turn id, and previous/new value summaries.
- Prompt-facing output must be compressed guidance, never raw drive state.

## Non-Goals

- Do not unify Path A (`conversation_loop.py`) and Path B (`mvp_loop.py`).
- Do not claim Hu Tao has subjective consciousness, real craving, or real
  pleasure.
- Do not implement boredom, pleasure proxy, tolerance, or UI idle activity
  here.
- Do not replace M11, M12.0, M12.1, or M12.2.
- Do not expose internal terms like "behavioral_pull", "policy precision",
  "free energy", or "candidate cost" in visible replies.
- Do not let the LLM invent behavioral-pull scores.

## Required Work

### 1. Add M13 State Surface To MVP Files

Extend the MVP JSON state defaults with an owned state section such as:

```text
m13_drive_state:
  version
  path_patterns_by_action
  recent_action_trace
  traction_by_action
  cue_precision_by_topic
  relation_path_precision
  last_patch_id
```

Minimum per-action pattern fields:

```text
action
support_count
success_proxy_count
failure_proxy_count
habit_precision
mean_control_cost_discount
last_seen_turn
source_evidence_ids
status
```

State constraints:

- keep bounded lists; no raw transcript dumps
- retain only compact source refs and summaries
- support JSON load/save through `MVPStateStore`
- be backward-compatible when the file is missing

### 2. Create M13 Drive Events

Use MVP-local dict events for now. They should be appended into the existing
`bus` diagnostics list and the turn log.

Minimum event types:

```text
M13DriveEvaluationEvent
M13DrivePatchProposal
M13DrivePatchCommit
```

Minimum evaluation event fields:

```text
event_id
turn_id
turn_index
source = "m13_drive_evaluator"
candidate_actions
selected_action
top_behavioral_pull_action
pull_margin
evidence_refs
prompt_safe_summary
```

Minimum patch proposal fields:

```text
patch_id
target = "m13_drive_state"
operation
field_path
previous_summary
new_summary
source_event_id
reason
confidence
ttl
```

Minimum commit fields:

```text
commit_id
patch_id
accepted
owner = "MVPDialogueRuntime"
reason
committed_summary
```

### 3. Implement Behavioral Pull Candidate Evaluation

Build a deterministic evaluator from:

- current `user_text`
- `conscious_plan`
- `memory_dynamics`
- retrieved memories
- `response_style_prior`
- learned conversation habits
- relationship value context
- prior M13 drive state

Candidate actions should come from existing surfaces when available:

```text
answer
ask_question
empathize
clarify
disagree
deflect
self_disclose
abstract_share
truthful_refusal
```

For each candidate, compute bounded components:

```text
habit_precision
cue_precision
relation_precision
memory_support
recent_success_proxy
repetition_penalty
control_cost_discount
behavioral_pull
```

Suggested formula shape:

```text
behavioral_pull =
  habit_precision * w_habit
+ cue_precision * w_cue
+ relation_precision * w_relation
+ memory_support * w_memory
+ recent_success_proxy * w_success
- repetition_penalty * w_repetition
```

All weights must live in a small constants object with rationale comments.
No magic numbers should be scattered through `mvp_loop.py`.

### 4. Patch Prompt-Safe Control Guidance

M13.0 may patch `memory_dynamics.control_guidance` before
`build_thinking_prompt`.

Allowed prompt-facing guidance:

```text
drive_guidance:
  preferred_reply_actions
  discouraged_reply_actions
  action_tendency_reason
  caution
  ordinary_language_policy
```

The guidance must be framed as response tendencies, not as visible inner
states. The thinking prompt may receive:

```text
"A recently useful response pattern is available, but do not repeat it
mechanically if the current user need differs."
```

It must not receive:

```text
"behavioral_pull=0.82"
"habit_precision increased"
"I am compelled to do this"
```

### 5. Post-Turn State Update

After reply generation and validation, update M13 drive state using proxy
outcomes:

Positive proxies:

- reply validation unchanged
- post-reply observer did not require repair
- expectation result confirmed
- user-facing reply action matched a high-pull candidate and no safety repair
  was triggered
- memory write candidate was accepted with evidence

Negative proxies:

- reply validation had to rewrite the reply
- post-reply observer required repair or self-correction
- safety layer blocked or altered high-risk content
- expectation result violated
- explicit user correction in the next turn, when available

M13.0 only uses current-turn proxies. Stronger next-turn outcome settlement is
deferred to M13.2.

## Acceptance Criteria

M13.0 is complete when the MVP chat path can compute and log behavioral pull
from bounded memory and habit evidence, use it to influence reply-action
guidance, and persist a compact audited drive-state patch without exposing
internal scores or claiming subjective compulsion.
