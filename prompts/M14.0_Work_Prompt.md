# OpenFEP M14.0 Work Prompt

## Milestone

M14.0: Conscious Idle Reflector And Self-Cognition Patch

## Goal

Give Hu Tao a real "high-order self-observer" on Path B: when the UI is idle
(see M13.4 plumbing), run a special conscious-loop mode that may

- pick a `reflection_focus` from memory dynamics, open items, and M13 drive
  signals,
- propose an auditable `self_cognition_patch`,
- propose memory consolidation intents and open-item updates,
- optionally hand off to the existing M13.3 proactive outreach pipeline so Hu
  Tao occasionally says something concrete, **not** because the system is bored
  or "lonely", but because there is a real, traceable reason.

M14.0 is the milestone where reflection happens. M13.4 only provided the idle
tick. M13.3 still owns proactive outreach delivery. M14.0 connects the three.

## Architecture Position

M14.0 is still Path B and still MVP-local. It must obey `CLAUDE.md`:

```text
UI idle tick (M13.4)
-> MVPDialogueRuntime.run_idle_introspection_turn(...)
-> bus messages: IdleIntrospectionTickEvent, TemporalContextEvent,
                 M13DriveSummaryEvent, MemoryDynamicsIdleSummaryEvent
-> conscious loop (mode = "idle_introspection")
   ↳ conscious_idle_plan JSON only, no visible reply
-> self_cognition / memory / open_item patches and intents
-> optional: turn conscious_idle_plan.outreach_recommendation into a
   ProactiveTurnProposal and call the existing M13.3 outreach path
-> audit events on conversation_log.jsonl
```

The conscious loop remains the orchestrator. M13 modules only contribute
"how strong is the drive to act, and is now a good time". The reflector never
edits memory or self_cognition directly: it produces proposals that are
validated and committed by named owners.

## Dependencies

Hard:

- M8.9 state-ownership contract and `MemoryWriteIntent`
- M9.0 `build_memory_dynamics_guidance`
- M13.0 behavioral pull (`path_patterns_by_action`, `traction_by_action`)
- M13.1 boredom (`boredom_level`, `boredom_band`, `exploration_target`)
- M13.2 reward proxy (`path_feels_stale`, `pending_settlements`,
  `reward_baseline`)
- M13.3 outreach pipeline (`maybe_propose_proactive_turn`,
  `run_proactive_turn`, delivery assessor)
- M13.4 idle tick plumbing (`run_idle_introspection_turn` stub, audit events)

Soft (read-only):

- M11.0 user models
- M12.0 / M12.1 / M12.2 outputs

M14.0 must not cross-write into M11, M12.0, M12.1, or M12.2 ledgers.

## Non-Goals

- Do not start any background loop, daemon, or worker.
- Do not let the reflector become a second LLM persona. It is a JSON-only
  planner inside the existing conscious loop.
- Do not produce `reply` text in the reflector. Only `outreach_recommendation`
  may trigger the existing M13.3 outreach to generate a real reply.
- Do not claim subjective "loneliness", "addiction", "boredom feeling", or
  "I needed to reach out". Use plain operational language.
- Do not extend Path A or M10. The reflector lives only in `mvp_loop.py` and
  related Path B modules.
- Do not write to `self_basic_facts.json`.
- Do not change M13.3 safety boundaries; outreach must still pass the existing
  delivery assessor.

## Required Work

### 1. Conscious Idle Mode

Extend `build_conscious_loop_prompt` (or add a sibling
`build_conscious_idle_prompt`) so that when called with
`mode="idle_introspection"` it:

- knows there is no user input this tick,
- receives a compact idle context (see §2),
- outputs the JSON schema in §3 only,
- is forbidden to generate prose, monologue, or a user-facing reply.

Prompt rules to add (verbatim intent, not exact wording):

- "no user input this tick; do not produce reply text"
- "use neutral operational language; do not claim subjective need"
- "every patch proposal must cite at least one evidence id retrieved this tick"
- "if no concrete focus can be justified, return an empty plan"

### 2. Idle Context Builder

Add a new helper, e.g. `build_idle_context(state, m13_drive_state, ...)`. It
must return a small, prompt-safe summary including:

- top 5 `open_items` with `status="open"` and concrete `next_check`
- short-term memory entries with `salience >= 0.5` not recalled in the last
  N user turns
- `pending_expectations` whose status is `uncertain` for >= K turns
- `m13_drive_state.boredom`: level, band, exploration_target
- `m13_drive_state.affective_reward_proxy`: `path_feels_stale`,
  `caution_about_repeating_easy_path`, `reward_baseline`,
  `pending_settlements` count
- `m13_drive_state.path_patterns_by_action.traction_by_action` top 3
- `temporal_state`: time since last user turn, last reply summary
- session counters: `reflection_count_this_session`,
  `outreach_count_this_session`

Raw state must not be passed in. Numbers may appear but only as small scalars
in a bounded summary. No full JSON dumps in the prompt.

### 3. Output Schema (conscious_idle_plan)

Lock the schema in code. The reflector outputs exactly:

```text
{
  "mode": "idle_introspection",
  "reflection_focus": {
    "topic": "",
    "evidence_refs": [],
    "reflection_kind": "open_item|user_model|self_consistency|habit_calibration|none"
  } | null,
  "self_cognition_patch_proposal": {
    "apply": false,
    "summary_delta": "",
    "new_identity_tensions": [],
    "new_known_limits": [],
    "evidence_refs": [],
    "confidence": 0.0,
    "reason": ""
  },
  "memory_consolidation_proposals": [
    {
      "target": "short_term|long_term",
      "kind": "episode|habit|preference|fact",
      "content": "",
      "confidence": 0.0,
      "evidence_refs": []
    }
  ],
  "open_item_proposals": [
    {
      "op": "update|close|defer",
      "id": "",
      "rationale": ""
    }
  ],
  "outreach_recommendation": {
    "should_outreach": false,
    "reason": "user_active|reflection_only|low_value|already_recent|none",
    "suggested_intent": "",
    "trigger": "reflection_outreach"
  },
  "thought_intensity_hint": "short"
}
```

Normalization helpers must clamp lengths, drop unknown enum values, and reject
any field that looks like a user-facing reply.

### 4. State Ownership And Validation

Add named-owner validators in `mvp_loop.py` or a new `m14_idle_owners.py`:

```text
SelfCognitionPatchOwner.validate_and_commit(proposal, retrieved_ids, ...)
OpenItemPatchOwner.validate_and_commit(proposal, ...)
MemoryConsolidationOwner.translate_to_intents(proposals, ...) -> [MemoryWriteIntent]
```

Validators must enforce:

- `evidence_refs` must intersect IDs actually retrieved this tick.
- `confidence >= MIN_PATCH_CONFIDENCE` (suggest `0.6`); rejects otherwise.
- `summary_delta` length and character set bounded.
- `self_cognition.patch_history` (new array, capped, FIFO) records committed
  patches with timestamp, source, evidence_refs.
- Rejected proposals produce
  `SelfCognitionPatchRejectedEvent` / `OpenItemPatchRejectedEvent` audit rows,
  not silent drops.

Per-session and per-day limits:

```text
MAX_SELF_COGNITION_PATCHES_PER_SESSION = 2
MAX_OPEN_ITEM_PATCHES_PER_SESSION      = 5
MAX_MEMORY_CONSOLIDATIONS_PER_SESSION  = 5
```

Cross writes into M11 / M12.* are forbidden. Validators must refuse any
proposal whose `target` looks like M11 or M12.

### 5. Drive Integration

The reflector consumes drive state read-only. The mapping rules are:

- **Boredom**
  - `band="high"` → prefer `reflection_kind in {open_item, habit_calibration}`;
    `exploration_target` may seed `reflection_focus.topic`.
  - `band="low"` → outreach is discouraged unless an open item has an explicit
    `next_check`.
- **Reward proxy / path staleness**
  - `path_feels_stale=true` → `outreach_recommendation.should_outreach` should
    be biased toward `false` to avoid repeated easy chatter.
  - `pending_settlements > 0` → reflection is preferred over outreach.
- **Behavioral pull**
  - High traction for an action does **not** force outreach. It only seeds
    `suggested_intent` if outreach is otherwise justified.
  - Low traction across the board → smaller `thought_intensity_hint`.

These are deterministic rules implemented around the reflector call, not LLM
prompt hints.

### 6. Outreach Bridge

When `outreach_recommendation.should_outreach` is true:

- The runtime translates the idle plan into a `ProactiveTurnProposal` with
  `trigger="reflection_outreach"` and `ordinary_language_intent` derived from
  `suggested_intent` (clamped, no diagnostics).
- It then calls the **existing** M13.3 path:
  `maybe_propose_proactive_turn` (for safety / cooldown checks) →
  `run_proactive_turn` (for generation + delivery assessor).
- If M13.3 suppresses delivery (e.g. `safety_risk`, `session_limit_reached`),
  the idle plan is **still committed** (reflection + patches). Outreach failure
  must not cause patch rollback.
- Outreach generated this way increments
  `m13_drive_state.initiative.outreach_count_this_session` (shared with M13.3)
  and `idle_introspection.outreach_via_introspection_count_this_session`.

There must be no second generation path. Anything that would produce a second
generation path is a hard fail in code review.

### 7. Audit Events

New event types layered on top of M13.4:

```text
IdleIntrospectionPlanEvent
SelfCognitionPatchProposalEvent
SelfCognitionPatchCommitEvent
SelfCognitionPatchRejectedEvent
OpenItemPatchProposalEvent
OpenItemPatchCommitEvent
OpenItemPatchRejectedEvent
MemoryConsolidationIntentEvent
IdleOutreachProposalEvent
IdleIntrospectionAbortEvent
```

Each event must include `turn_index`, `at`, `engineering_proxy_label`, the
relevant `evidence_refs`, and a short `reason`. Validator rejections must
include `violation_codes` from a closed enum.

### 8. UI Surface (Diagnostics Only)

In the inner-world / diagnostics panel:

- Show the last `conscious_idle_plan` (JSON, expandable).
- Show recent `self_cognition.patch_history` (last 5).
- Show whether the most recent tick led to an outreach attempt and its outcome.

User-visible chat must show **only** real outreach replies, exactly as M13.3
already produces them. Nothing else from the idle plan may appear in the chat
log.

## Acceptance Criteria

M14.0 is complete when:

1. Idle ticks (from M13.4) can drive a single conscious-loop call in
   `mode="idle_introspection"` and produce a bounded JSON plan.
2. Self-cognition, open-item, and memory updates from that plan only happen
   through named owners with `evidence_refs`, confidence thresholds, and
   audit events; rejections are logged, not silent.
3. Reflection and outreach are decoupled: most idle ticks reflect without
   sending a message; outreach goes through the existing M13.3 path or not at
   all.
4. M13.1 boredom, M13.2 reward proxy, and M13.0 behavioral pull demonstrably
   bias plan content and outreach probability in deterministic tests.
5. M11 and M12.* ledgers remain untouched by the reflector.
6. No subjective-need wording appears in any reflector output, prompt, or
   outreach intent.
7. Path A and M10 are not used or extended.
8. All listed suggested tests pass.
