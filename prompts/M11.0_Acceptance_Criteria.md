# M11.0 Acceptance Criteria - User Generative Model And Value Memory Dynamics

This document defines pass/fail gates for M11.0. Each gate is concrete and
testable. M11.0 acceptance is `NOT_ISSUED` until every gate passes. Partial
passes must be reported per-gate, never aggregated into an `ACCEPT` stamp.

The work prompt is `prompts/M11.0_Work_Prompt.md`. This file is the binding
acceptance contract for that prompt.

## Gate 1: Stand-Alone Architecture Boundary

- All new M11 modules live under `segmentum/user_model/`.
- A static check (grep or AST walk) shows that **no file under
  `segmentum/user_model/` imports any symbol from**:
  - `segmentum/memory_dynamics.py`
  - `segmentum/value_memory.py`
  - `segmentum/memory_retrieval.py`
- M11 modules may import `segmentum/memory_evidence.py` (for the
  `MemoryEvidence` output dataclass) and may attach to
  `segmentum/dialogue/runtime/...` hook points read-only.
- A test (`test_m11_does_not_import_memory_dynamics_or_value_memory_or_memory_retrieval`)
  asserts the import boundary at runtime via `sys.modules` inspection.
- Removing any of those three legacy modules from `sys.modules` after
  M11 boot must not raise `ImportError` from any M11 module.

## Gate 2: LLM Use Boundary Enforcement

- The LLM extractor schema in `segmentum/user_model/llm_extractor.py`
  declares only enum-typed and bounded structured fields. No field has
  type `float` or `number`.
- A schema validator rejects any extractor output that contains:
  - any numeric float in any field
  - any field not present in the declared schema
  - any `prediction_id` or `hypothesis_id` not present in the bounded
    snapshot passed in (proposal `id`s for new proposals are exempt;
    their `source_hypothesis_ids` and `source_judgment_ids` must still
    reference snapshot ids)
- `test_extractor_output_with_any_float_is_rejected`,
  `test_extractor_output_with_unknown_field_is_rejected`, and
  `test_extractor_output_referencing_unknown_prediction_id_is_rejected`
  all pass.
- The deterministic layer (`reliability_ledger`, `prediction_ledger`,
  `value_composer`, `evidence_cards`, `m11_runtime`) runs end-to-end
  on a stubbed extractor with no live LLM call.
  `test_dynamics_layer_runs_without_any_llm_call_using_stub_extractor`
  passes.
- A grep audit shows that no float (`value_score`, `source_reliability`,
  `prediction_error`, `future_path_utility`, `expected_free_energy_delta`,
  `retention_pressure`, `retrieval_score`, or any custom name) appears
  in the extractor's prompt template, system message, or output schema.

## Gate 3: UserModel Structural Correctness

- `segmentum/user_model/user_model.py` defines a `UserModel` dataclass
  whose fields cover the minimum set in the work prompt
  (`user_id`, `display_name`, `cognitive_style_hypotheses`,
  `personality_hypotheses`, `preference_hypotheses`,
  `boundaries_and_dislikes`, `relationship_state`,
  `source_reliability_by_domain`, `claim_history_summary`,
  `prediction_history_summary`, `open_uncertainties`,
  `last_updated_turn_id`).
- Each hypothesis carries `hypothesis_id`, `content_summary`, `domain`,
  `confidence_band`, `evidence_refs`, `contradiction_refs`,
  `last_confirmed_turn`, `last_violated_turn`, `permitted_use`.
- `confidence_band` is an enum literal `low | med | high`. A property
  test asserts the type is not `float`.
- `UserModel.to_json()` and `UserModel.from_json(...)` round-trip
  losslessly. The roundtrip test reads no prompt text and no trace dump.
- A single user utterance (one turn, one claim) cannot move any
  hypothesis past the `med` band on its own. Promotion to `high`
  requires multi-turn evidence and a deterministic gate documented in
  `hyperparams.py`. `test_low_evidence_personality_inference_stays_in_low_confidence_band`
  enforces this.

## Gate 4: PredictionLedger And Proposal Lifecycle

- `segmentum/user_model/prediction_ledger.py` is append-only. Existing
  entries are never mutated; status transitions are recorded as new
  rows referencing `prediction_id`.
- `validation_status` ∈ `{pending, confirmed, violated, uncertain}`.
- The ledger answers the query "predictions in the last N turns by
  status" deterministically.
- Every prediction proposal recorded includes `proposal_id`,
  `proposed_prediction_type`, `predicted_value_summary`,
  `confidence_band`, `source_hypothesis_ids`, `source_judgment_ids`,
  `expires_after_turns`, `accepted`, `rejection_reason`.
- A deterministic gate decides `accepted`. The gate's inputs are
  enum/count fields and `hyperparams.py`. The gate's behavior is
  reproducible across runs.
- A per-turn proposal quota lives in `hyperparams.py`. Excess proposals
  are rejected with `rejection_reason = "proposal_quota_exceeded"` and
  do not reach acceptance.
- A proposal whose `expires_after_turns` has elapsed without a matching
  judgment is closed with `validation_status = "uncertain"` and
  `rejection_reason = "expired"`. `test_prediction_proposal_acceptance_and_expiration_are_deterministic`
  passes.
- A proposal whose `source_hypothesis_ids` or `source_judgment_ids`
  reference unknown ids is rejected with
  `rejection_reason = "unknown_source_id"`.

## Gate 5: SourceReliabilityLedger Behavioral Guarantees

`segmentum/user_model/reliability_ledger.py` must satisfy each of the
following property tests on at least 100 random seeds:

- `test_reliability_monotone_under_only_confirmed`: a sequence of only
  `confirmed` judgments in a fixed domain produces a non-decreasing
  `reliability` series.
- `test_reliability_bounded_in_unit_interval`: under any random sequence
  of judgments, `reliability` stays in `[0, 1]`.
- `test_reliability_per_turn_delta_bound`: `|reliability_t - reliability_{t-1}| <= max_delta_per_turn`
  even when many `confirmed` and `violated` events occur in the same turn.
- `test_reliability_single_lie_cannot_destroy_long_history`: after 50
  `confirmed` events in a domain followed by one `violated` event in the
  same domain, `reliability` remains above 0.5 and drops by at most
  `max_delta_per_turn`.
- `test_reliability_domain_isolation`: judgments in domain A do not
  change `reliability` for any other domain.
- `test_reliability_decay_under_silence`: with no events for `>= 5 *
  half_life_turns`, `reliability` converges within 0.10 of the prior
  mean defined by `prior_alpha / (prior_alpha + prior_beta)`.
- `test_reliability_unaffected_by_roleplay_or_joke_modality`: judgments
  whose source claim has `modality ∈ {roleplay, joke, hypothetical}`
  produce zero change to factual-domain reliability.
- `test_reliability_is_independent_of_evidence_text_content`: feeding
  identical structured judgments with different `evidence_text` strings
  produces byte-identical reliability state. **This gate fails the
  milestone if it does not pass.**

## Gate 6: Determinism Reliability Contract

The deterministic layer must satisfy all six clauses of the
"Determinism Reliability Contract" in the work prompt:

- **Pure functions only**: a static check shows that
  `reliability_ledger`, `prediction_ledger`, `value_composer`,
  `evidence_cards` contain no `import re`, no `str.lower()`-on-content
  calls, no global mutable caches, no environment reads, no time reads
  outside of injected `current_turn_id`.
- **All constants in `hyperparams.py`**: a grep audit shows no numeric
  literal other than `0`, `1`, `-1` appears in the bodies of update
  functions in those four modules. Every other constant is sourced from
  a `Hyperparams` dataclass field whose docstring explains operational
  meaning.
- **Update kinds whitelist**: every update function uses only
  count-or-ratio, bounded EMA with documented half-life, Beta/Dirichlet
  posterior over enum events, or bounded clamp with documented
  per-turn delta cap. A reviewer signs off this clause by listing every
  update function and its category in the M11 acceptance report.
- **Property tests**: each dynamics function has a property test for
  boundedness, monotonicity-under-monotone-input, domain isolation,
  per-turn delta bound, no-text-dependency, and decay-under-silence.
  Missing property tests fail the gate.
- **Replay fixtures**: each dynamics function has at least one committed
  fixture pair `(input_judgments.json, expected_state_trace.json)` under
  `fixtures/m11/<function_name>/<case>/`. A change to the function that
  alters a trace requires updated fixtures plus a `rationale.md` entry.
- **Calibration audit**: the audit report defined in Gate 10 ships as
  part of the acceptance artifact.

## Gate 7: Value Composer Reproducibility And Gate Semantics

- `segmentum/user_model/value_composer.py` defines a pure function
  `compose_value(...)` whose output is fully determined by its inputs
  and `hyperparams.py`. Two runs over the same inputs produce
  byte-identical `value_score`, `write_target`, and `gating_reasons`.
- `test_value_score_rises_monotonically_with_reliability_at_fixed_band`:
  holding `memory_value_band`, `recency_weight`,
  `contradiction_unresolved`, and `privacy_or_safety_flag` fixed and
  varying only `source_reliability` from 0 to 1 in 10 steps,
  `value_score` is a non-decreasing series.
- `test_privacy_flag_prevents_long_term_promotion_regardless_of_band`:
  with `privacy_or_safety_flag = True`, `write_target` is never
  `long_term_user_model` regardless of `memory_value_band` or
  `source_reliability`.
- `test_contradiction_unresolved_lowers_value_score`: with
  `contradiction_unresolved = True`, `value_score` is strictly less
  than the same call with `contradiction_unresolved = False`, all
  other inputs equal.
- A reviewer audit checks that `value_score` is computed only from
  ledger-derived values and `hyperparams.py`, and never reads the
  extractor's `surprise_explanation`, raw text, or any other free-form
  string.

## Gate 8: Evidence Cards And Reply Policy Surface

- `segmentum/user_model/evidence_cards.py` produces cards whose schema
  is compatible with `segmentum/memory_evidence.py::MemoryEvidence`.
- Cards expose only the fields listed in the work prompt
  (`memory_id`, `memory_class`, `content_summary`, `confidence_band`,
  `source_reliability` exposed as ordinal label in any prompt-facing
  surface, `permitted_use`, `contradiction_status`, `why_retrieved`).
- `test_prompt_facing_evidence_cards_never_contain_floats_or_extractor_json`:
  the prompt-facing serialization of any card contains no numeric float
  and no extractor JSON fragment. Floats may exist on the in-memory
  card object but must be transformed to ordinal labels before reaching
  any prompt template.
- Cards never expose the LLM's `surprise_explanation` verbatim. A test
  scans the prompt-facing surface for the literal explanation string
  from the extractor and asserts absence.
- `test_reply_shortens_when_brevity_preference_reliable_and_active`,
  `test_reply_asks_clarification_when_intent_prediction_just_violated`,
  and `test_reply_softens_when_domain_reliability_below_threshold` all
  pass on a stubbed extractor scenario.
- A diagnostic trace records, for each reply policy adjustment, which
  `hypothesis_id` or `judgment_id` caused it. The trace is not exposed
  to the prompt; it is artifact-only.

## Gate 9: Non-Interference Guarantee

- A configuration flag (`m11_user_model_enabled`, default `False` for
  legacy personas, default `True` for new personas) gates all M11
  reads, writes, and evidence-card emission.
- `test_existing_persona_with_m11_disabled_byte_identical_to_baseline`:
  with the flag off, a fixed transcript replayed through a legacy
  persona produces byte-identical chat outputs to its pre-M11 baseline
  under each of: rule-mode generator, stub generator, recorded LLM
  fixture.
- `test_m11_enabled_does_not_modify_any_legacy_memory_row`: with the
  flag on, no row in `MemoryStore`, `LongTermMemory`, or the existing
  anchored memory layer is created, edited, deleted, or re-scored by
  M11 code paths. Asserted by snapshot diff before/after a turn.
- `test_existing_chat_retrieval_path_unchanged_with_m11_enabled`:
  the existing retrieval path returns the same candidate list and the
  same ranking with the flag on as with the flag off, holding chat
  state fixed.
- A fresh persona with the flag on initializes a clean UserModel and
  proceeds without altering any pre-existing chat memory rows.

## Gate 10: Calibration Audit Report

The acceptance artifact must include a calibration audit report covering
five hand-authored multi-turn scenarios. Each scenario has a documented
threshold; the gate fails if any scenario fails its threshold.

- `scenario_stable_preference`: across 5 turns of consistent preference
  statements in domain `self_reported_preferences`, the preference
  hypothesis reaches `confidence_band = "high"` and
  `source_reliability_for("self_reported_preferences") >= 0.7`.
- `scenario_detected_lie`: a `violated` judgment in domain
  `self_reported_history` reduces that domain's reliability by
  `>= 0.05` within one turn while raising the value score of the
  contradiction memory.
- `scenario_roleplay_isolation`: a multi-turn roleplay segment with
  modality `roleplay` produces zero change to factual-domain
  reliability and zero promotion of roleplay claims to long-term
  user-model facts.
- `scenario_contradiction`: two contradictory user claims about the
  same domain are retained as a contradiction with `permitted_use`
  downgraded to `cautious_hypothesis`. Neither claim becomes a
  `permitted_use = "explicit_fact"` row.
- `scenario_long_silence`: after 100 turns with no judgments in a
  previously high-reliability domain, that domain's reliability
  converges within 0.15 of the prior mean.

Each scenario includes:

- the input transcript (turn-by-turn user/agent text)
- the recorded extractor output per turn (used as fixture)
- the expected state trace per turn (committed)
- the per-scenario pass/fail check

## Gate 11: Acceptance Artifact Reproducibility

- The acceptance artifact lives under `artifacts/m11_acceptance/` and
  contains the fields listed in the work prompt:
  `turns`, `extractor_outputs`, `user_model_before_after`,
  `prediction_ledger`, `prediction_proposals`,
  `reliability_ledger_updates`, `memory_value_compositions`,
  `evidence_cards`, `reply_policy_effects`, `quarantined_hypotheses`,
  `calibration_audit_report`.
- Replaying the artifact's `extractor_outputs` as fixtures through the
  M11 pipeline produces byte-identical `reliability_ledger_updates`,
  `memory_value_compositions`, `evidence_cards`, and
  `reply_policy_effects`.
- `test_value_scores_are_byte_identical_across_runs_with_same_extractor_fixture`
  passes.
- The artifact demonstrates each of the following at least once:
  - one stable preference learned across turns: rising reliability,
    rising `value_score`, eventual `write_target = "long_term_user_model"`
  - one contradicted claim that raises memory value but lowers domain
    reliability
  - one roleplay or joke that does not damage factual source reliability
  - one uncertain personality hypothesis that stays in
    `confidence_band = "low"` and is not promoted
  - one later turn where reply policy is changed by a UserModel
    evidence card without any change to the existing chat memory rows

## Gate 12: Three-Layer Acceptance Taxonomy

Following the M4.11 convention, the M11 acceptance report emits three
explicit fields and an `ACCEPT` stamp requires all three to be true:

- `structural_pass`: Gates 1, 2, 3, 4 pass. The architecture boundary,
  LLM use boundary, UserModel shape, and prediction ledger lifecycle
  are correct.
- `behavioral_pass`: Gates 5, 6, 7, 8, 9 pass. Reliability dynamics,
  determinism contract, value composition, evidence cards, reply
  policy, and non-interference behave as specified under property
  tests, replay fixtures, and stub extractor runs.
- `phenomenological_pass`: Gates 10, 11 pass. The five calibration
  scenarios meet their thresholds and the acceptance artifact is
  reproducible.

A milestone marked `ACCEPT` without all three being true is reported
as a contract violation.

## Not in scope

- No acceptance based on the M11 module wrapping or composing with
  `compute_cue_match`, `_mood_match`, `ValueMemoryEvaluator._*_gain`,
  or any other token-bag scoring function from the legacy memory
  dynamics layer, however cleverly renamed.
- No acceptance based on LLM-produced floats, however calibrated.
- No acceptance from a "deterministic" function whose output depends
  on raw text of `evidence_text`, `surprise_explanation`, or claim
  content.
- No acceptance from a calibration scenario whose extractor output
  was hand-authored to match the expected state trace; the extractor
  output must be a captured run, not a synthesized fixture matching
  the desired result.
- No acceptance from in-memory tests alone if the legacy chat memory
  baseline cannot be replayed byte-identical with the M11 flag off.
- No acceptance based on prompt-text claims of M11 behavior; only on
  code, tests, and the deterministic acceptance artifact.
