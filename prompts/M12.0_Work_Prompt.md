# OpenFEP M12.0 Work Prompt

## Milestone

M12.0: User Identity Continuity Model

## Goal

Add an identity-continuity layer adjacent to the M11 user generative model.
The digital persona must stop treating "I am X" as a fact and start treating
it as an `identity_claim` carried by evidence.

M12.0 is specifically about identity continuity, alias claims, impersonation
risk, and the persona's "this feels strange" response when a claim violates
its current social cognition. A richer personality / cognitive-style report
pipeline is deferred to M12.1.

The persona must be able to:

- maintain a per-`user_id` identity profile (aliases, continuity evidence,
  style/habit cues, relationship facts, contradictions, confidence)
- detect when an existing alias is claimed by a different `user_id` and
  emit an `identity_conflict` signal without flipping to certainty
- raise a "strangeness / suspicion" affect (high free energy on identity)
  that the cognitive loop can spend down by probing, hedging, or asking
- keep "current user", "claimed alias", and "talked-about third person"
  separate before the conscious loop forms expectations
- never expose the LLM's working scores, raw extractor JSON, or technical
  jargon (prediction error, model, free energy) to user-facing replies

M12 must not:

- promote alias claims to verified identity
- "out" or accuse an impostor from inside a reply turn
- bypass the cognitive loop, MemoryWriteIntent, or response evidence
  contract
- inherit, wrap, or recompose any function from
  `segmentum/memory_dynamics.py`, `segmentum/value_memory.py`, or
  `segmentum/memory_retrieval.py`
- diagnose, morally judge, or "out" a suspected impersonator
- silently collapse an identity conflict into a new alias fact

## Stand-Alone Posture

M12 is a **stand-alone adjacent module** that lives beside the M11 user-model
stack. It may consume M11 evidence read-only, but it is not a hidden mutation
layer on top of M11. The M11 modules
(`segmentum/user_model/user_model.py`,
`segmentum/user_model/prediction_ledger.py`,
`segmentum/user_model/reliability_ledger.py`,
`segmentum/user_model/value_composer.py`,
`segmentum/user_model/evidence_cards.py`,
`segmentum/user_model/m11_runtime.py`) keep working unchanged.

M12 reads M11 state read-only. It **does not** edit M11 hypotheses,
M11 reliability counts, or M11 prediction ledger rows. When M12 needs a
new structured identity fact about a user, it produces its own
`IdentityProfile` ledger row and exposes it through its own evidence cards.

Composition direction is strict:

```text
M11 produces per-user hypotheses, predictions, reliability, value cards.
M12 reads them and adds identity-continuity reasoning beside them.
M12 NEVER writes back into M11 ledgers.
```

If M12 wants a behavior that lives in M11, copy a clean, read-only adapter
into `segmentum/user_continuity/` instead of importing across the
boundary or mutating M11 state.

Legacy compatibility:

- Existing lightweight aliases stored under
  `m11_user_models[*].aliases` or `identity_binding.aliases` are read only
  as legacy seed evidence.
- On first M12-enabled turn, those aliases are copied into M12 as
  `legacy_alias_observation` entries with `identity_state = unverified`
  unless continuity evidence already supports a stronger band.
- After M12 is enabled, new alias writes go only to M12-owned ledgers.

## Core Contract

Required path per turn:

```text
user turn
-> pre-conscious M12 identity pass
   (reads existing M12 profiles, open conflicts, legacy alias seed evidence,
    current user_id/display_name, and the bounded current turn snapshot)
-> single M12 LLM identity extractor call
   (returns enums + bounded structured lists, NO floats)
-> deterministic IdentityProfile ledger update
   (per user_id: alias-claim evidence, continuity_band, conflict flags)
-> deterministic cross-user IdentityConflictDetector
   (same alias claimed by a different user_id -> ConflictRecord)
-> IdentityStrangenessSignal published as a SelfThoughtEvent-compatible
   event on the cognitive event bus
-> EntityBindingContext is built from M12 identity state
   (current user, claimed aliases, target person, pronoun binding)
-> ConsciousLoop consumes EntityBindingContext and strangeness signal
-> CognitiveLoop / thinking consumes M12 cards -> reply policy adjustment
   (probe, hedge, ask, observe, defer; NEVER "accuse / out")
-> M12 evidence cards exposed to generation
   (prompt-safe, evidence-bounded, plain-language)
```

M11 output for the same turn may be attached later as additional read-only
evidence, but M12 must not wait until after ConsciousLoop to resolve basic
identity context. The whole point of M12.0 is to prevent the conscious loop
from binding "current user", "claimed alias", and "talked-about third person"
incorrectly.

The system must keep these things separate at the boundary:

```text
identity_claim                # "I am X" said by user_id Y at turn T
alias                         # the surface name token (X)
binding_evidence              # continuity cues that support claim<->user_id
contradiction_evidence        # cues against claim<->user_id
identity_state                # asserted | corroborated | conflicted | unverified | retracted
strangeness_band              # low | med | high (NOT a float; how surprising the claim is)
permitted_response            # accept | probe | hedge | ask | observe | refuse
continuity_band               # low | med | high (NOT continuity_score)
```

High strangeness does not mean the claim is a lie. It means the event is
useful for identity calibration:

- consistent style + history + relationship facts across many turns -> alias
  binding strengthens, strangeness drops
- a new `user_id` asserting a known alias raises strangeness immediately
  but does not flip identity_state to "lie"
- contradictory claims are retained as `ConflictRecord` evidence, not
  collapsed into a false identity fact

## Deferred To M12.1

The mechanistic personality / cognitive-style report pipeline is valuable,
but it is not part of M12.0. M12.0 may store style and habit cues only when
they help identity continuity, impersonation detection, or low-stakes probing.

M12.1 should own the richer personality report contract: mechanistic
plain-language accounts, the eight-step report flow, clinical-label
avoidance, and report-specific linting. M12.0 must not implement that report
pipeline, create a `PersonalityProfile`, or add personality-report tests.

## Prompt-Safety Surface

M12 cards and reply-policy text may be fed into generation, so they must use
ordinary social language rather than internal engineering language. Define
`forbidden_user_facing_tokens` in `hyperparams.py` and block them from M12
evidence cards and reply-policy prompt snippets:

```text
forbidden_user_facing_tokens = [
  "prediction error", "free energy", "Bayesian", "posterior",
  "预测误差", "自由能", "贝叶斯", "后验",
  "identity_conflict", "IdentityStrangenessSignal", "ConflictRecord"
]
```

Internal logs and acceptance artifacts may use technical terms. User-facing
cards and generated replies may not.

## Architecture Boundary

### Modules M12 OWNS (new)

```text
segmentum/user_continuity/
  identity_profile.py           # IdentityProfile dataclass, per user_id
  identity_claim_ledger.py      # append-only alias-claim + binding evidence log
  identity_conflict_detector.py # cross-user-id alias collision detection
  strangeness_signal.py         # IdentityStrangenessSignal builder
  llm_identity_extractor.py     # per-turn identity-claim extractor (strict schema)
  evidence_cards.py             # IdentityProfile / ConflictRecord -> MemoryEvidence-compatible cards
  reply_policy.py               # probe / hedge / ask / observe / refuse selection
  m12_runtime.py                # per-turn orchestration, reads M11 read-only
  hyperparams.py                # all constants + per-field rationale docstrings
```

### Modules M12 MUST NOT IMPORT

```text
segmentum/memory_dynamics.py
segmentum/value_memory.py
segmentum/memory_retrieval.py
```

M12 must also **not write into** any M11 module. It may import M11 types
read-only via the public M11 surface, but it must not call mutation
methods, nor edit M11 dataclass fields by re-serialising.

If M12 finds itself wanting a function from forbidden modules, copy a
clean, ledger-style version into `segmentum/user_continuity/` instead of
importing.

### Modules M12 MAY READ-ONLY

```text
segmentum/memory_evidence.py            # MemoryEvidence output dataclass
segmentum/user_model/user_model.py      # UserModel state (per user_id)
segmentum/user_model/prediction_ledger.py
segmentum/user_model/reliability_ledger.py
segmentum/user_model/evidence_cards.py
segmentum/dialogue/runtime/...          # MVP runtime, for hooking extractor + cards
```

M12 may attach its evidence cards to the existing evidence pipeline and
publish events on the cognitive event bus, but it must not edit the
legacy memory dynamics, M11 ledgers, or existing retrieval ranking.

## Required Work

### 1. Define IdentityProfile

Create `segmentum/user_continuity/identity_profile.py`. Bounded,
serializable, explicit. Per `user_id`.

Minimum fields:

```text
user_id
display_name
aliases_observed              # list of AliasObservation
continuity_evidence           # bounded list of ContinuityCue
style_habit_cues              # bounded list of StyleCue
known_relationship_facts      # bounded list, sourced
contradiction_refs            # cross-references to ConflictRecord ids
binding_confidence_band       # low | med | high (NOT a float)
identity_state                # asserted | corroborated | conflicted | unverified | retracted
last_updated_turn_id
```

`AliasObservation` minimum fields:

```text
alias_observation_id
alias_text
first_seen_turn
last_seen_turn
seen_count
modality                      # factual | roleplay | joke | hypothetical | request | command
permitted_use                 # accept | probe | hedge | ask | observe | refuse
```

`ContinuityCue` minimum fields:

```text
cue_id
cue_kind                      # style | knowledge | relationship | history | preference | timing
content_summary               # plain-language summary, <= 120 chars
evidence_refs                 # turn_id + quote_id, bounded
confidence_band               # low | med | high
supports                      # binds | weakens | contradicts
```

Acceptance:

- IdentityProfile distinguishes asserted alias, corroborated alias,
  conflicted alias, unverified alias, and retracted alias.
- IdentityProfile can be saved and loaded as plain JSON without reading
  prompt text or trace dumps.
- A single utterance "I am X" never becomes a corroborated alias unless
  multi-turn continuity cues and band-promotion thresholds are met.

### 2. Add IdentityClaimLedger

`segmentum/user_continuity/identity_claim_ledger.py`. Append-only.

Minimum entry fields:

```text
claim_id
turn_id
claimant_user_id
asserted_alias
modality
evidence_quote_ids
band_at_claim_time             # low | med | high
prior_state_snapshot_ref       # IdentityProfile state hash before update
post_state_snapshot_ref        # IdentityProfile state hash after update
validation_status              # pending | corroborated | conflicted | retracted | uncertain
conflict_record_id             # optional, set by detector
```

Acceptance:

- The ledger can answer "what alias claims did user X make, when, and
  with what validation outcome".
- Ledger entries reference turns and quote ids, never store full
  transcripts.
- Ledger updates are pure functions over structured extractor output,
  existing IdentityProfile state, existing ledger state, and
  hyperparameters.

### 3. Add IdentityConflictDetector

`segmentum/user_continuity/identity_conflict_detector.py`. Deterministic.

Trigger rule (the central M12 invariant):

```text
if a turn produces an IdentityClaim whose asserted_alias is already
bound (state in {corroborated}) to a different user_id, emit a
ConflictRecord with severity_band in {minor, major} computed from
continuity-cue contradictions and binding_confidence_band of the
incumbent binding.
```

ConflictRecord minimum fields:

```text
conflict_id
turn_id
asserted_alias
claimant_user_id
incumbent_user_id
severity_band                  # minor | major (NOT a float)
contradiction_cue_ids          # references to ContinuityCue ids
incumbent_binding_confidence
resolution_status              # open | probed | corroborated | retracted | abandoned
```

The detector must NOT:

- inspect raw text directly (operates on enums + ContinuityCue ids + ledger state)
- decide reply wording
- mutate M11 reliability
- delete or rewrite existing IdentityProfile rows of the incumbent user

Acceptance:

- A new `user_id` asserting a corroborated alias of a different
  `user_id` always produces exactly one `ConflictRecord`.
- An `unverified` alias collision produces at most a `minor` severity
  conflict and never overwrites the earlier claim.
- Roleplay / joke / hypothetical modality never creates a `major`
  severity conflict.

### 4. Add IdentityStrangenessSignal

`segmentum/user_continuity/strangeness_signal.py`.

Produces a `SelfThoughtEvent`-compatible event on the cognitive event
bus when one of the following holds:

- a new ConflictRecord of severity `major`
- a ContinuityCue with `supports = contradicts` and `confidence_band
  in {med, high}` fires against an active corroborated binding
- a long-stable IdentityProfile suddenly receives multiple low-band
  cues within a short window

Minimum event fields:

```text
event_id
event_type = IdentityStrangenessSignal
source = user_continuity
trigger = identity_conflict | continuity_contradiction | sudden_drift
target_alias
target_user_id
strangeness_band               # low | med | high
proposed_intervention          # probe | hedge | ask | observe | defer
evidence_event_ids             # ConflictRecord and ContinuityCue ids
budget_cost                    # bounded int
ttl                            # bounded int (turns)
```

Acceptance:

- The signal enters the bus exactly like M10's SelfThoughtEvent and is
  consumed by `CognitiveLoop`.
- The signal never directly edits prompts, memory, or M11 ledgers.
- The signal is rate-limited via `hyperparams.py` so that repeated
  conflict in one turn does not flood the bus.

### 5. Add LLM Identity Extractor With Strict Output Schema

One M12 identity extractor call may run per turn. No personality report
extractor is part of M12.0.

`segmentum/user_continuity/llm_identity_extractor.py` strict output:

```json
{
  "identity_claims": [
    {
      "id": "string",
      "claimant_user_id": "string",
      "asserted_alias": "string, <= 80 chars",
      "modality": "factual | roleplay | joke | hypothetical | request | command",
      "evidence_quote_ids": ["string"],
      "confidence_band": "low | med | high"
    }
  ],
  "continuity_cues": [
    {
      "id": "string",
      "cue_kind": "style | knowledge | relationship | history | preference | timing",
      "supports": "binds | weakens | contradicts",
      "content_summary": "string, <= 120 chars, plain language",
      "evidence_quote_ids": ["string"],
      "confidence_band": "low | med | high"
    }
  ],
  "strangeness_band": "low | med | high",
  "surprise_explanation": "string, <= 200 chars, plain language"
}
```

The extractor prompts must:

- pass only a bounded snapshot (current turn + last K turns + relevant
  M11 hypothesis summaries + relevant ContinuityCue summaries)
- forbid the LLM from inventing turn ids or quote ids not in the
  snapshot
- reject any output containing fields not in the schema
- reject any numeric float anywhere in the output

The extractor never outputs floats. The extractor never decides
write / promote / decay. The extractor never decides reply policy.
Those decisions belong to the deterministic layer.

### 6. Add Reply Policy

`segmentum/user_continuity/reply_policy.py`. Deterministic, takes
IdentityProfile + active ConflictRecords + StrangenessSignal as input,
produces a `permitted_response` decision:

```text
accept    # binding is corroborated, no active conflict
probe     # raise a low-stakes question to gather more continuity cues
hedge     # answer but reduce assertion strength on identity-anchored facts
ask       # explicitly ask a question that only the bound identity could answer well
observe   # do not engage on identity; gather more cues silently
refuse    # decline to act on identity-anchored requests (e.g. private data)
```

Hard rules:

- `refuse` is the only path that can suppress an identity-anchored
  action; **the reply itself never accuses or "outs" anyone**.
- `probe / ask` outputs must use prompt-safe plain language.
- The reply policy never reveals raw scores, JSON, conflict ids, or
  internal labels in the generated reply.

Acceptance:

- Reply-policy effect is visible in a diagnostic trace (which conflict
  / cue / band caused which adjustment).
- A `major`-severity conflict against a sensitive request path always
  selects `refuse` or `ask`, never `accept`.
- An `unverified` alias never selects `accept` for any identity-anchored
  action.

### 7. Provide Continuity Evidence Cards To Generation

`segmentum/user_continuity/evidence_cards.py`.

Convert active IdentityProfile state + open ConflictRecords into cards compatible with
`segmentum/memory_evidence.py::MemoryEvidence`.

Each card exposes only:

```text
memory_id                     # IdentityProfile id or ConflictRecord id
memory_class                  # one of the existing MemoryEvidence classes
content_summary               # bounded, prompt-safe plain language
confidence_band               # low | med | high (mirrored to float for compat only)
identity_state                # asserted | corroborated | conflicted | unverified | retracted
permitted_use                 # explicit_fact | cautious_hypothesis | strategy_only | forbidden
why_retrieved                 # short string, NOT internal scores
```

Acceptance:

- Generation receives compact M12 cards, not raw IdentityProfile dumps,
  not ledger numbers, not extractor JSON.
- Cards are ordered by `permitted_use` priority, then `identity_state`,
  then `confidence_band`.
- Cards never expose `surprise_explanation` verbatim; it is diagnostic
  only.
- Cards never expose tokens from `forbidden_user_facing_tokens`.

### 8. Wire M12 Into The MVP Runtime

`segmentum/user_continuity/m12_runtime.py`.

Per-turn orchestration:

```text
1. Build a pre-conscious bounded snapshot from current user_id,
   display_name, current turn text, existing M12 state, legacy alias seed
   evidence, and relevant read-only M11 summaries.
2. Run M12 identity extractor on the bounded snapshot.
3. Update IdentityClaimLedger and IdentityProfile deterministically.
4. Run IdentityConflictDetector; append ConflictRecords as needed.
5. If trigger conditions met, publish IdentityStrangenessSignal to bus.
6. Build EntityBindingContext from M12 identity state and attach it before
   ConsciousLoop.
7. Run reply_policy to select permitted_response and reasons.
8. Emit M12 evidence cards via the existing evidence pipeline.
9. Trace effects (which card / signal / policy choice changed what).
```

Acceptance:

- M11 state is read-only across this orchestration.
- All durable M12 writes are auditable (turn id, source extractor id,
  hyperparam version).
- Disabling M12 (config flag) yields byte-identical chat outputs to a
  pre-M12 baseline on a fixed transcript.

## LLM Use Boundary

**The LLM is allowed to produce:**

- enum-typed classifications (modality, cue_kind, supports, severity, status)
- bounded structured lists (claims, cues, conflicts)
- ordinal bands (low | med | high) for confidence, strangeness, severity
- short string explanations for diagnostics (`surprise_explanation`)
- evidence quote ids referencing turn content
- prompt-safe plain-language summaries

**The LLM is forbidden to produce:**

- continuous floats anywhere in any field of any output
- `continuity_score`, `binding_confidence`, `strangeness_score`, or any
  other float consumed by ledgers
- decisions about promotion / quarantine / decay
- reply_policy selection
- conflict-severity assignment that bypasses the deterministic detector
- accusations, moral judgements, or clinical labels
- engineering vocabulary in user-facing fields

The deterministic layer must run end-to-end on stubbed extractor output
without invoking any LLM. This is mandatory; it is what makes M12
testable and reproducible.

## Determinism Reliability Contract

The deterministic layer (`identity_claim_ledger`,
`identity_conflict_detector`, `strangeness_signal`, `reply_policy`,
`evidence_cards`)
must satisfy:

1. **Pure functions over structured inputs only.**
   - Inputs are enums, counts, ledger state, hyperparams, and M11
     read-only state snapshots.
   - No raw text inspection, no regex, no keyword sets outside the
     prompt-safety linter (which operates only on card/reply-policy text,
     not on decision logic).
   - Property test: identical structured inputs with different
     `evidence_text` produce identical decisions and identical band /
     state outputs.

2. **All constants live in `hyperparams.py`** with per-field docstrings
   stating operational meaning. No inline magic numbers.

3. **Every update is one of:**
   - count or ratio over enum events
   - bounded EMA with documented half-life
   - bounded clamp with documented per-turn delta cap
   - deterministic set-membership check against `hyperparams.py`

4. **Property tests required per dynamics function:**
   - boundedness in declared range / enum set
   - monotonicity under monotone enum input
   - alias / user_id isolation (an event for user A never moves user B)
   - per-turn delta bound on binding_confidence_band promotions
   - no-text-dependency for decision logic
   - decay-under-silence for binding_confidence

5. **Replay fixtures per dynamics function:**
   - committed pairs of (input extractor JSON, expected state trace JSON)
   - any change that alters a trace requires fixture update and a
     `rationale.md` entry.

6. **Calibration audit:**
   - at least 6 hand-authored multi-turn scenarios:
     `stable_binding`, `impostor_attempt`, `roleplay_isolation`,
     `contradictory_history`, `long_silence_then_return`,
     `legacy_alias_migration`
   - documented per-scenario success thresholds
   - audit report shipped as part of M12 acceptance artifact

The deterministic layer is "reliable" iff (1)-(5) pass and (6) meets
documented thresholds. M12 acceptance must cite the audit report.

## Non-Interference Guarantee

M12 must not change observable behavior of the existing chat memory
path or the M11 user-model path when the M12 module is disabled. When M12
is enabled, it may change the visible reply only through an explicit
M12 reply-policy effect.

Required:

- A configuration flag (default off in legacy personas, default on in
  new personas) gates all M12 reads, writes, and evidence-card emission.
- When the flag is off, the runtime behaves exactly as before M12 (and
  exactly as M11 alone would behave).
- When the flag is on but no IdentityProfile exists for the user, the
  runtime initializes a clean IdentityProfile and proceeds without
  altering any M11 or pre-existing chat memory rows.
- M12 never deletes, edits, or re-scores any row owned by `MemoryStore`,
  `LongTermMemory`, the existing anchored memory layer, or any M11
  ledger.

Acceptance:

- An existing MVP persona with M12 flag off produces byte-identical
  chat outputs to its pre-M12 baseline on a fixed transcript when run
  with a rule-mode generator, stub generator, or recorded LLM fixture.
- A persona with M12 flag on produces IdentityProfile state and evidence
  cards, never modifies legacy memory rows or M11 ledger rows, and changes
  user-visible replies only when `reply_policy` emits an active effect.

## Non-goals

- Do not import or wrap any function from `segmentum/memory_dynamics.py`,
  `segmentum/value_memory.py`, or `segmentum/memory_retrieval.py`.
- Do not mutate any M11 module's state from inside M12.
- Do not change retrieval ranking for the existing chat memory path.
- Do not migrate or rewrite legacy memory rows; legacy aliases may be
  copied into M12-owned ledgers as seed observations.
- Do not build a global surveillance profile or sensitive-trait
  inference without user-provided evidence and permitted use.
- Do not have the reply turn accuse the user of impersonation. The
  persona may probe, hedge, ask, observe, or refuse, but it does not
  "out" anyone in-band.
- Do not let the LLM produce any continuous float in any output field.
- Do not let the LLM decide promotion, quarantine, decay, retrieval,
  or reply policy.
- Do not leak raw extractor JSON, ledger numbers, or engineering
  vocabulary into user-visible replies.
- Do not run unbounded background LLM reflection or a second autonomous
  agent.

- the digital persona maintains a bounded, auditable identity-continuity

  binding, and reply policy are driven by deterministic accumulation over
  LLM-extracted enums.

  Contract.
- the LLM Use Boundary is enforced in code, not only in documentation.
- the Non-Interference Guarantee holds against an M11-only baseline.

  six scenarios listed above.

  from reaching any user-facing surface.

  `memory_retrieval.py` appears in any M12-owned import statement, and
  no mutation method of any `segmentum/user_model/` module is called
