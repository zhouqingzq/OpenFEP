# OpenFEP M13.3 Work Prompt

## Milestone

M13.3: UI-Level Bounded Initiative

## Goal

Add real UI-facing initiative on top of the MVP-local M13 drive layer. Current
`followup_replies` are same-turn extra bubbles, not true initiative. M13.3
should let Hu Tao generate a bounded proactive turn after idle time or a clear
internal agenda signal, while preserving user control, auditability, and
safety.

The target behavior:

```text
M13 drive state / boredom / unresolved agenda
+ idle time
+ consent / UI setting
+ cooldown / budget
-> ProactiveTurnProposal
-> UI scheduler / idle tick
-> safety and relevance gate
-> optional proactive message
-> logged as system-initiated turn
```

This milestone is about product architecture as much as cognition. Without a
UI scheduler or heartbeat, the runtime cannot truly initiate; it can only add
follow-up text inside a user-triggered turn.

## Dependencies

M13.3 builds on:

- M13.0 behavioral pull
- M13.1 boredom / exploration bias
- M13.2 reward proxy and pending settlement
- existing Streamlit UI chat path

It may ship behind a disabled-by-default setting.

## Non-Goals

- Do not create an always-running autonomous agent.
- Do not send proactive messages without an explicit UI setting or local
  development flag.
- Do not use proactive turns for persuasion, dependency building, or
  attention capture.
- Do not bypass safety, evidence, memory, or response validation.
- Do not claim the persona "woke up" or "felt lonely" unless the user has
  explicitly opted into fictional roleplay wording and the message remains
  clearly within persona style.
- Do not run unbounded background LLM calls.

## Required Work

### 1. Add Proactive Initiative State

Extend M13 state:

```text
m13_drive_state:
  initiative:
    enabled
    user_opt_in
    idle_threshold_seconds
    cooldown_turns
    cooldown_until_timestamp
    max_proactive_per_session
    proactive_count_this_session
    last_proactive_turn_at
    pending_proactive_proposal
    last_suppression_reason
```

Defaults must be conservative:

```text
enabled = false
user_opt_in = false
max_proactive_per_session = 1
```

### 2. Define ProactiveTurnProposal

Minimum fields:

```text
proposal_id
created_at
source = "m13_initiative_policy"
trigger
trigger_evidence_refs
urgency_band
expected_user_value_band
risk_band
proposed_action
proposed_topic
ordinary_language_intent
expires_at
cooldown_cost
```

Allowed triggers:

- high unresolved exploration target after idle time
- repeated high boredom with a safe small next target
- user asked to continue later
- pending open item has a clear next check
- prior correction needs a concise repair or clarification
- user explicitly asked the persona to remind, follow up, or keep thinking

Blocked triggers:

- loneliness simulation
- jealousy / dependency pressure
- punishment, guilt, or emotional coercion
- curiosity about sensitive personal data
- low-value chatter just because the system is idle

### 3. Add Initiative Policy

The initiative policy should be deterministic and cheap. It reads:

- `m13_drive_state`
- `temporal_state`
- `open_items`
- recent conversation log metadata
- latest diagnostics
- UI opt-in setting

It produces either:

```text
ProactiveTurnProposal
```

or a suppression reason:

```text
disabled
not_opted_in
cooldown_active
idle_time_too_short
no_high_value_target
safety_risk
insufficient_evidence
session_limit_reached
```

### 4. Add UI Scheduler / Idle Tick Hook

Implement one bounded UI mechanism. Acceptable options:

- Streamlit-side polling when the chat page renders
- a local app heartbeat that checks elapsed idle time
- a manual "let Hu Tao continue" button for first acceptance

The scheduler must call a new runtime method, not fake a user message:

```text
MVPDialogueRuntime.maybe_propose_proactive_turn(...)
MVPDialogueRuntime.run_proactive_turn(proposal_id, ...)
```

The proactive turn must be logged with role/source metadata:

```text
role = "assistant"
source = "m13_proactive_turn"
trigger = proposal.trigger
not_user_requested_current_turn = true
```

### 5. Generate Proactive Text Through Existing Safety Boundaries

Proactive generation may reuse the thinking prompt path, but it must receive a
special user-text surrogate such as:

```text
"[system idle tick: generate only if proposal remains useful and safe]"
```

Generation constraints:

- one short message by default
- no demand for response
- no sensitive inference
- no raw diagnostics
- no claim of subjective need
- mention concrete useful context if possible

Good shape:

```text
"I thought of a cleaner way to frame the M13 split: start with path pull,
then add boredom as the switch signal. Want me to sketch the test gates?"
```

Bad shape:

```text
"I got bored waiting and needed to talk to you."
```

### 6. Consent And UI Control

Add a visible local setting or development flag:

```text
Enable bounded proactive messages
```

Required behavior:

- off by default
- easy to disable
- session-scoped count and cooldown
- no proactive messages during active typing / pending user message
- no proactive messages after safety-risk suppression

### 7. Audit Trail

Every proactive attempt must be logged:

```text
M13ProactiveCheckEvent
M13ProactiveProposalEvent
M13ProactiveSuppressionEvent
M13ProactiveGenerationEvent
```

The log must show:

- why it did or did not fire
- which state fields were read
- whether user opt-in was active
- whether cooldown/session cap suppressed it
- what visible text was sent, if any

## Acceptance Criteria

M13.3 is complete when the UI can, behind explicit opt-in, run a bounded idle
initiative check, produce or suppress a proactive turn with an auditable reason,
and send at most safe, useful, short proactive messages through the existing
MVP runtime and safety boundaries.
