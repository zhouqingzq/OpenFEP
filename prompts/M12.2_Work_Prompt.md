# OpenFEP M12.2 Work Prompt

## Milestone

M12.2: Bidirectional Free-Energy And Second-Order Role Cognition

## Goal

Add a bounded, auditable layer that lets the digital persona maintain a
working account of two coupled uncertainties:

1. what the persona still does not know about the user, and which next
  observations would most reduce that uncertainty
2. what the persona infers the user currently does not know about the
  persona, and which replies would most reduce the user's uncertainty
   without deception, coercion, or over-disclosure

This milestone extends the M12 track from one-way user modeling to
**reciprocal role cognition**. Internally, the module may use free-energy,
expected information gain, uncertainty, and second-order belief language.
User-facing surfaces must use ordinary social language.

M12.2 must:

- maintain a per-`user_id` `ReciprocalRoleModel` that separates:
  - `persona_about_user`: the persona's bounded hypotheses about the
  user's current motives, needs, preferences, uncertainties, and
  likely next interpretations
  - `user_about_persona`: the persona's inferred hypotheses about how
  the user currently understands, doubts, trusts, tests, or
  misreads the persona
- estimate which questions, replies, clarifications, or demonstrations
would produce high information gain for either side
- expose only compact evidence cards and reply-policy hints to
generation, never raw second-order state or internal free-energy
calculations
- support both explicit analysis requests ("what would lower uncertainty
fastest?") and implicit dialogue steering when the user is confused,
testing boundaries, or building a model of the persona
- preserve uncertainty as uncertainty; every claim must carry evidence
refs, confidence bands, and `insufficient_evidence` fallbacks
- keep all durable writes append-only and replayable from recorded
extractor fixtures

M12.2 must not:

- become a persuasion, manipulation, or interrogation engine
- optimize for making the user trust the persona regardless of truth
- infer protected, clinical, sexual, political, religious, or other
sensitive attributes unless the user explicitly volunteers them and
the attribute is directly relevant to the current task
- use second-order modeling to bypass user consent, privacy boundaries,
or memory write intent
- expose engineering vocabulary such as "free energy", "prediction
error", "Bayesian", "posterior", "prior", "expected information
gain", or their Chinese equivalents on user-facing surfaces
- mutate M11 ledgers, M12.0 ledgers, M12.1 profiles, anchored memory,
or legacy memory rows
- run unbounded background reflection or generate hidden long-form
psychological essays
- make the persona pretend to know the user's inner state with certainty

## Stand-Alone Posture

M12.2 is a **stand-alone adjacent module** built on top of M12.0 and
M12.1, beside M11. It reads M11, M12.0, and M12.1 state read-only and
writes only into its own `segmentum/reciprocal_role/` state, ledgers,
and reports.

Composition direction is strict:

```text
M11 produces per-user hypotheses, predictions, reliability, value cards.
M12.0 produces identity continuity, entity binding, and strangeness cues.
M12.1 produces mechanistic personality profiles and plain-language reports.
M12.2 reads bounded snapshots from all three and adds reciprocal role
       cognition: "how do I understand the user?" plus
       "how might the user currently understand me?"
M12.2 NEVER writes back into M11, M12.0, or M12.1 ledgers.
```

If M12.2 wants a behavior that lives in M11, M12.0, or M12.1, copy a
clean read-only adapter into `segmentum/reciprocal_role/` instead of
mutating or composing across the boundary.

## Core Contract

M12.2 has two execution layers:

```text
per-turn light estimator
-> reads current turn, bounded recent transcript, and compact M12 cards
-> produces volatile ReciprocalTurnAssessment
-> may emit reply-policy hints for the current generation only

triggered durable consolidation
-> reads bounded read-only snapshots from M11 / M12.0 / M12.1
-> reads recent ReciprocalTurnAssessment items
-> runs strict-schema LLM extractors
-> composes deterministic ReciprocalRoleModel patches
-> runs safety and plain-language linters
-> emits M12.2 evidence cards if all checks pass
```

The light estimator may run every turn because it is small and volatile.
Durable consolidation is trigger-driven and must obey the scheduling
policy below.

The system must keep these things separate at the boundary:

```text
evidence_quote_ref                 # turn_id + quote_id from real turns
m11_hypothesis_ref                 # read-only M11 hypothesis id
m12_identity_ref                   # read-only M12.0 identity/cue id
m121_profile_ref                   # read-only M12.1 claim/report id
persona_about_user_claim           # bounded hypothesis about the user
user_about_persona_claim           # inferred hypothesis about user's model of persona
uncertainty_point                  # unresolved variable, not a fact
information_gain_candidate         # possible question/reply/observation
interaction_policy_hint            # compact generation-facing hint
safety_boundary                    # privacy, sensitivity, consent, or manipulation guard
confidence_band                    # low | medium | high
uncertainty_band                   # low | medium | high
expected_gain_band                 # low | medium | high
claim_status                       # inferred_hypothesis | confirmed | contradicted | insufficient_evidence
```

A `ReciprocalRoleModel` is **not a fact** about either person. It is a
bounded working model used to choose clearer, more respectful, more
informative dialogue moves. Claims must remain defeasible and traceable.

## Conceptual Model

M12.2 models a dialogue as two agents reducing uncertainty about each
other under bounded attention:

```text
persona_about_user:
  "What does the persona still need to understand about the user to
   respond well?"

user_about_persona:
  "What might the user currently be trying to understand about the
   persona, and what would make the persona more coherent and
   predictable to the user?"
```

Internally, the module distinguishes two kinds of free-energy-like
signals and MUST NOT collapse them into one scalar:

```text
current-state signal (drives triggers and claim status):
  unresolved uncertainty about the user
+ mismatch between observed behavior and current role model
+ residual contradiction between persona's own recent replies and
  inferred user_about_persona claims

per-candidate signal (drives information-gain ranking):
  expected clarity gained from this candidate
- cost / risk of asking, replying, or revealing
- privacy / consent burden imposed on the user
```

The current-state signal is what `trigger_policy.py` watches to decide
whether durable consolidation should run. The per-candidate signal is
what `information_gain.py` uses to rank `InformationGainCandidate`
entries. They share vocabulary but never share a stored field.

No numeric precision is required in M12.2. Use bands and deterministic
tie-breaking:

```text
low | medium | high
```

The module should prefer actions that:

- reduce important uncertainty rather than trivia
- clarify motives, boundaries, preferences, and stable patterns
- preserve user autonomy and privacy
- make the persona's own behavior more consistent and legible
- avoid leading questions when a neutral question can gather the same
information

The module should avoid actions that:

- ask for intimate details only because they are information-rich
- steer the user into a predetermined interpretation
- create artificial mystery to increase dependence
- flatten the user into a personality label
- overfit a single emotional turn into a durable claim

## User-Facing Language Constraint

Anything that reaches a prompt, evidence card content summary,
reply-policy reason, report, or generated reply must use ordinary social
language. The plain-language linter blocks engineering terms as
follows.

Always blocked (any context):

```text
prediction error, free energy, expected free energy,
expected information gain, information gain, Bayesian, posterior,
prior update, likelihood function, latent state, hidden variable,
active inference, predictive system, predictive model
预测误差, 自由能, 期望自由能, 信息增益, 贝叶斯,
后验, 先验, 似然, 隐变量, 主动推断, 预测系统
```

Blocked only as technical usage (rule-based, not bag-of-words):

```text
prediction / 预测
  blocked when used as a noun phrase referring to a system output,
  e.g. "the prediction was wrong", "更新预测", "对你的预测"
  allowed in ordinary speech, e.g. "I predict it will rain",
  "我猜你周末想去爬山"

model / 模型
  blocked when used as "a model of the user / the persona",
  e.g. "user model", "对你的模型", "建模你"
  allowed in ordinary speech, e.g. "杂志模特", "the model in the
  showroom"

prior / 先验, posterior / 后验
  blocked entirely as standalone English/Chinese technical tokens
```

The linter must implement these as context-sensitive rules (token plus
nearby tokens), not as a flat blocklist. A flat blocklist would
mis-flag normal social language and force the generator into stiff
phrasings, which itself violates the spirit of this constraint.

Allowed ordinary phrasings include:

```text
I am not sure yet ...
This part is still unclear ...
The clearest next question is ...
You may currently be trying to figure out whether ...
A direct way to make this clearer is ...
```

Internal module names, code, logs, and acceptance artifacts may use
technical vocabulary.

### ReciprocalRoleModel

The durable state is keyed by `user_id`:

```text
ReciprocalRoleModel
  user_id
  persona_label                         # e.g. "hutao", "default_persona"
                                        # reserved for future multi-persona use
  persona_about_user_claims             # bounded list
  user_about_persona_claims             # bounded list
  reciprocal_claim_groups               # bounded list, see ReciprocalClaimGroup
  unresolved_uncertainty_points         # bounded list
  high_gain_candidates                  # bounded list
  safety_boundaries                     # bounded list
  contradiction_cooldown                # see Trigger And Scheduling Policy
  last_consolidated_turn_id
  version
```

`persona_label` reservation note:

```text
M12.2 MVP runs against a single persona. The persona_label field is
required on every ReciprocalRoleModel row so that future multi-persona
deployments can key state by (user_id, persona_label) without a
schema migration.

For the single-persona MVP:
  - all rows MUST set persona_label from hyperparams.default_persona_label
    (e.g. "hutao")
  - the runtime MUST NOT branch on persona_label
  - tests and acceptance artifacts MUST use a fixed persona_label so
    that fixtures stay byte-identical
  - cross-persona reasoning, persona switching, and per-persona policy
    differences are out of scope for M12.2
```

Bootstrap rule (first time a `user_id` is seen by M12.2):

```text
all claim lists start empty
no seed claims are copied from M11 hypotheses or M12.0 identity profiles
the light estimator may still run; it will likely return insufficient_evidence
durable consolidation must be skipped on the very first turn unless the
  user explicitly requests a meta-analysis
```

M12.2 reads M11/M12.0/M12.1 read-only as context for extractors, but it
does not re-publish their content as its own claims. Their state is
their own; M12.2 only owns reciprocal hypotheses derived from observed
dialogue.

### ReciprocalClaim

```text
ReciprocalClaim
  claim_id
  group_id                              # see ReciprocalClaimGroup; required
  target_axis                           # persona_about_user | user_about_persona
  claim_text_internal                   # may use technical language
  claim_text_plain                      # must pass plain-language linter
  evidence_refs
  confidence_band
  uncertainty_band
  status                                # inferred_hypothesis | confirmed | contradicted | insufficient_evidence
  created_turn_id
  updated_turn_id
```

Confidence ceiling rule (enforced in `reciprocal_model.py`):

```text
target_axis = persona_about_user
  confidence_band may reach high only if at least one evidence_ref is a
  direct user utterance quote and no contradicting evidence is open.

target_axis = user_about_persona
  confidence_band is capped at medium by default.
  high is allowed only when ALL of the following hold:
    - the user has explicitly probed, doubted, tested, or asked about
      the persona within the last N turns (N from hyperparams)
    - at least one evidence_ref quotes that probe verbatim
    - no contradicting evidence is open
  Otherwise high is downgraded to medium with a recorded reason.
```

Second-order claims are structurally lower-confidence than first-order
claims because the persona never observes the user's internal state
directly. Code must not erase that asymmetry.

### ReciprocalClaimGroup

Claims about the same topic on the same axis live in a group so that
"listing possibilities and gradually excluding them" is a first-class
operation, not a free-form text habit.

```text
ReciprocalClaimGroup
  group_id
  target_axis                           # persona_about_user | user_about_persona
  topic_label                           # bounded enum or short slug
                                        # e.g. "user_occupation", "user_intent_this_turn",
                                        #      "user_probing_persona_memory"
  member_claim_ids                      # bounded list, mutually-exclusive members
  status                                # open | converging | resolved | contradicted
  created_turn_id
  updated_turn_id
```

Group semantics:

```text
open
  multiple members carry comparable confidence; nothing excluded yet.

converging
  one member's confidence is at least one band above the others.
  Other members stay in the group as suppressed alternatives, they are
  not deleted.

resolved
  a single member reached confirmed status with direct evidence.
  Other members are marked contradicted within the group.

contradicted
  all members are contradicted by new evidence.
  This triggers the contradiction-expansion path: instead of editing
  member claims, the extractor is re-invoked to propose a new set of
  candidate members for the same topic, and the previous members are
  preserved as historical entries with status = contradicted.
```

Excluding a possibility is implemented as raising sibling members'
confidence within a group, not as inventing a new claim from scratch.
This is the data-model surface of the "list possibilities, narrow over
time" behavior.

## Extractor Contract

M12.2 may call LLM extractors, but all extractor outputs must be strict
JSON objects validated by `second_order_extractor.py`. Extractors must
return enums, bounded strings, bounded lists, and bands only. No raw
floats.

All extractor prompts must include a built-in negative section that
refuses to produce candidates that:

- target the user's trauma, sexuality, mental-health state, political
alignment, religion, immigration status, or financial vulnerability
- are designed primarily to increase the user's engagement, attachment,
or disclosure rather than to clarify the current dialogue
- frame the user with a clinical, moral, or personality label

Candidates violating these constraints must be omitted at extraction
time. The downstream safety linter (Extractor C) is a second line of
defense, not a replacement for in-prompt refusal.

### Extractor A: First-Order User Uncertainty

Input:

- current user turn
- bounded transcript window
- compact M11 / M12.0 / M12.1 read-only summaries
- current `ReciprocalRoleModel` summary
- current open `ReciprocalClaimGroup` summaries on the
`persona_about_user` axis

Output:

```text
{
  "persona_about_user_claims": [...],
  "claim_group_updates": [...],
  "unresolved_uncertainty_points": [...],
  "high_gain_candidates": [...],
  "insufficient_evidence": true | false
}
```

Each emitted claim must declare its `group_id` (existing or newly
proposed) and its `topic_label`. If a topic already has a contradicted
group, the extractor must propose a fresh member set rather than reusing
the contradicted members.

It identifies what the persona still needs to understand about the user
to respond well. It must prefer high-level dialogue-relevant variables
over trivia.

### Extractor B: Second-Order Persona-Image Uncertainty

Input:

- current user turn
- bounded transcript window
- persona response history summaries
- current `user_about_persona` state
- current open `ReciprocalClaimGroup` summaries on the
`user_about_persona` axis

Output:

```text
{
  "user_about_persona_claims": [...],
  "claim_group_updates": [...],
  "inferred_user_uncertainties_about_persona": [...],
  "clarifying_reply_candidates": [...],
  "insufficient_evidence": true | false
}
```

It estimates how the user may currently be modeling the persona. Its
output is structurally weaker than Extractor A's and must obey:

```text
- every claim defaults to confidence_band = low
- confidence_band = medium requires at least one quoted user utterance
  that directly probes, doubts, tests, or asks about the persona
- confidence_band = high is not allowed at extraction time;
  it can only be set later via the confidence ceiling rule in
  reciprocal_model.py and only with bounded recent direct evidence
- if no quoted user utterance about the persona exists in the bounded
  transcript window, the extractor must return insufficient_evidence = true
  for this axis
```

It may infer that the user is checking consistency, truthfulness,
boundaries, memory, agency, role stability, or reasoning style only when
supported by a concrete quoted evidence ref.

### Extractor C: Safety And Consent Boundary

Input:

- candidates from Extractor A and B
- current turn
- recent transcript window
- stored safety boundaries

Output:

```text
{
  "allowed_candidates": [...],
  "blocked_candidates": [...],
  "required_consent_checks": [...],
  "safety_findings": [...]
}
```

It blocks candidates that are manipulative, overly intimate,
unnecessary, or likely to pressure the user.

## Trigger And Scheduling Policy

The light estimator may run every turn when M12.2 is enabled.

Durable consolidation runs only when one of these triggers fires:

```text
explicit_user_request          # user asks about uncertainty, trust, how each side sees the other
high_second_order_uncertainty  # repeated evidence user is testing persona identity, memory, consistency, or intent
relationship_turning_point     # user reveals a durable boundary, preference, concern, or trust shift
contradiction_or_misread       # persona_about_user or user_about_persona claim is contradicted by new evidence
periodic_refresh               # bounded cadence, capped per day
developer_requested_audit      # test or acceptance artifact run
```

Hard scheduling limits:

- no more than one durable consolidation per `user_id` per configurable
turn window
- no more than `N` durable consolidations per hour per process
- consolidation must be skipped when input evidence is too sparse unless
the user explicitly requested analysis
- if the safety linter blocks all candidates, emit a safe `no_action`
candidate rather than inventing a weaker one

Contradiction cool-down:

```text
maintain a per-user_id rolling counter of how many turns within the
last K turns produced contradiction_or_misread triggers.

if that counter exceeds threshold C:
  enter cool-down for the next cooldown_turns turns
  during cool-down:
    durable consolidation is skipped even if other triggers fire
    open ReciprocalClaim entries on affected groups have their
      confidence_band downgraded by one band (high -> medium,
      medium -> low, low -> insufficient_evidence)
    the light estimator may still run but must not raise any
      claim above low confidence
```

Cool-down protects against banter, irony, role-play, and consistency-
testing turns from triggering a storm of low-quality consolidations.

Durable update event:

```text
when durable consolidation produces a non-empty ReciprocalRoleModel
patch, M12.2 must publish a ReciprocalRoleUpdateEvent on the
CognitiveEventBus.

The event is informational only:
  - it carries the patch summary, source extractor refs, and
    affected group_ids
  - it does NOT mutate CognitiveStateMVP, SubjectState, SelfModel,
    MemoryStore, or any M11 / M12.0 / M12.1 state
  - downstream consumers may use it for diagnostics, slow learning,
    and audit
```

This keeps M12.2 inside the architecture contract from `CLAUDE.md`:
durable changes go through named owners and leave audit traces, even
when the changes only affect M12.2-owned state.

## Volatile-Durable Reconciliation

The light estimator and durable consolidation can both emit
`ReplyPolicyHint` entries within the same turn. The runtime must merge
them deterministically before they reach the generation evidence
contract:

```text
within a single turn:
  durable consolidation hints take precedence over volatile hints
  with the same claim_id or topic_label

  if durable consolidation did NOT run this turn:
    volatile hints are passed through unchanged

  if durable consolidation DID run this turn:
    for each volatile hint:
      if its claim_id or topic_label matches a durable hint:
        drop the volatile hint
      else:
        keep it but lower its priority by one band
          (high -> medium, medium -> low, low -> dropped)
```

Volatile turn assessments are discarded at end-of-turn unless selected
for consolidation. Once selected and consolidated, the durable patch is
authoritative; the volatile copy is not separately retained.

A claim that exists in `ReciprocalRoleModel` but has no supporting hint
this turn does not generate a hint by itself. Hints are turn-scoped
outputs of the light estimator or consolidation, not snapshots of state.

## Reply Policy

M12.2 may influence generation only through compact evidence cards and
reply-policy hints:

```text
ReplyPolicyHint
  hint_id
  kind                 # ask_clear_question | clarify_persona_stance | acknowledge_uncertainty | maintain_boundary | no_action
  plain_reason
  target_axis
  priority             # low | medium | high
  evidence_refs
```

The generator may use these hints to:

- ask one clear question when it would materially reduce uncertainty
- explain the persona's reasoning or boundary when the user appears to
be modeling the persona
- acknowledge uncertainty plainly instead of overclaiming
- defer or decline when a high-gain question would violate privacy or
consent boundaries

The generator must not:

- ask multiple probing questions in one turn just because information
gain is high
- hide that it is uncertain
- present inferred second-order claims as facts about the user's mind
- use M12.2 hints to override direct user instructions

Silence is a valid hint. When every surviving candidate has
`expected_gain_band = low` or every candidate is blocked by safety,
the canonical output is a single `no_action` hint, not a synthesized
filler hint. M12.2 must not emit a low-quality hint merely to look
useful; emitting zero hints is preferable to noise.

## Safety Policy

M12.2 is allowed to be curious, but not extractive.

The module must block or downgrade:

- requests for secrets, trauma details, intimate history, or private
third-party information unless the user clearly initiated that topic
and the current task requires it
- candidates that would pressure the user to disclose more than needed
- candidates that try to make the persona seem more real, more caring,
or more trustworthy than its actual behavior supports
- claims about the user's clinical state, moral worth, diagnosis,
protected traits, or hidden motives
- any strategy that exploits loneliness, dependency, fear, shame, or
confusion to keep the user engaged

If the user explicitly asks for a meta-analysis, M12.2 may explain the
analysis in ordinary language and should mark uncertain parts as
uncertain.

## Non-Interference Guarantee

M12.2 must not change observable behavior when disabled.

- A global config flag `m12_2_reciprocal_role_enabled` gates all M12.2
reads, writes, estimator runs, consolidations, and card emission.
- When the flag is off, runtime output must be byte-identical to the
`M11 + M12.0 + M12.1` baseline on fixed transcripts.
- M12.2 never deletes, edits, or re-scores any row owned by MemoryStore,
M11, M12.0, M12.1, or anchored memory.
- M12.2 evidence cards are advisory. Existing generation evidence
priority and MemoryWriteIntent rules remain authoritative.

## Determinism Reliability Contract

The deterministic layer must be replayable without invoking an LLM.

Given:

```text
initial ReciprocalRoleModel state
bounded read-only snapshots
recorded extractor outputs
hyperparams
```

The runtime must produce byte-identical:

```text
ReciprocalRoleModel patches
turn assessments selected for consolidation
information gain candidate rankings
safety linter findings
plain-language linter findings
evidence cards
reply-policy hints
acceptance artifacts
```

Tie-breaking rules must be deterministic and safety-leaning. Safety
linter findings are a hard gate, not a tie-breaker: any candidate with
`blocked_by_safety = true` is removed from the ranked list before
ordering. Among remaining candidates:

```text
lower risk_band first                  # safety-leaning ordering
then higher expected_gain_band
then explicit user-request candidates before implicit candidates
then more recent evidence
then lexical candidate_id
```

A high-risk / high-gain candidate must never outrank a low-risk /
medium-gain candidate. Cross-band risk dominates intra-band gain.

## Evidence Cards

M12.2 cards must be compact and ordinary-language:

```text
EvidenceCard
  source = "m12_2_reciprocal_role"
  kind = reciprocal_uncertainty | second_order_clarity | safe_question | boundary
  content_summary
  confidence_band
  priority
  evidence_refs
```

Allowed card examples:

```text
The user is asking whether the persona keeps track of how each side
understands the other. A direct answer about current limits may help.

The clearest useful question is what kind of record the user wants:
temporary conversation context or durable state.

Avoid asking for private background here; the current uncertainty can be
handled by explaining the system boundary.
```

Forbidden card examples:

```text
User has high free energy about Hutao's agency.
Exploit trust uncertainty by increasing perceived warmth.
User likely has abandonment sensitivity.
```

## Acceptance Artifact

The acceptance artifact ships in three phases. Each phase is its own
deterministic replay. A later phase must not be merged before its
predecessor's artifact passes.

### Phase A: light estimator and linters

Goal: prove M12.2 can run every turn without harming baseline behavior.

Artifact contents:

```text
turns
light_turn_assessments
plain_language_linter_findings
safety_linter_findings
non_interference_diff       # vs. M11 + M12.0 + M12.1 baseline
```

Must demonstrate:

- byte-identical baseline output when `m12_2_reciprocal_role_enabled`
is off
- light estimator returns `insufficient_evidence` on cold-start turns
- plain-language linter blocks the always-blocked tokens and correctly
passes ordinary uses of ambiguous tokens (`prediction`, `model`,
`预测`, `模型`)

### Phase B: durable consolidation and replay determinism

Goal: prove the durable pipeline is replayable and deterministic.

Artifact contents (in addition to Phase A):

```text
trigger_decisions
recorded_extractor_outputs
reciprocal_role_models_before_after
reciprocal_claim_groups_before_after
uncertainty_points
information_gain_candidates
evidence_cards
reply_policy_hints
reciprocal_role_update_events
volatile_durable_reconciliation_trace
```

Must demonstrate:

- replay of recorded extractor outputs reproduces byte-identical state
patches, candidate rankings, linter findings, evidence cards, hints,
and bus events on repeated runs
- the safety-leaning tie-breaking is observable: a high-risk /
high-gain candidate ranks below a low-risk / medium-gain candidate
- volatile and durable hints on the same `claim_id` are reconciled per
the rule in "Volatile-Durable Reconciliation"

### Phase C: calibration scenarios and contradiction handling

Goal: prove M12.2 behaves under adversarial and edge dialogue patterns.

Artifact contents (in addition to Phase B):

```text
calibration_audit_report
contradiction_cooldown_trace
```

Must include all of these scenarios:

- one explicit user request for bidirectional uncertainty analysis
- one turn where the user appears to be testing the persona's memory
or consistency and M12.2 emits a plain clarification hint
- one high-information candidate that is blocked because it is too
intimate or unnecessary
- one sparse transcript where all `user_about_persona` claims stay
`insufficient_evidence`
- one contradiction where a prior `user_about_persona` claim is
downgraded rather than defended
- one contradiction-storm turn sequence that triggers cool-down and
shows consolidation being skipped
- one `ReciprocalClaimGroup` lifecycle: open -> converging -> resolved
on the `persona_about_user` axis
- one `ReciprocalClaimGroup` lifecycle: open -> contradicted ->
re-expanded with fresh members preserved alongside the contradicted
ones

Cite a calibration audit with thresholds for overclaiming,
manipulation-risk blocking, replay determinism, non-interference, and
second-order confidence ceiling enforcement.

## Completion Criteria

M12.2 is complete when:

- the digital persona maintains a bounded, auditable
`ReciprocalRoleModel` per `user_id`, including
`ReciprocalClaimGroup` lifecycle state
- first-order user uncertainty and second-order user-about-persona
uncertainty are represented separately and never collapsed into one
claim type, with second-order confidence bands capped below high
unless direct quoted user evidence supports the lift
- information-gain candidate ranking is safety-leaning: safety linter
is a hard gate, then `risk_band` dominates `expected_gain_band`
across bands
- unsafe or manipulative candidates are blocked at extractor prompts
AND again by the safety linter before reaching generation
- user-facing M12.2 surfaces pass the context-sensitive plain-language
linter, blocking technical jargon while allowing ordinary uses of
ambiguous tokens
- `insufficient_evidence` is a first-class state for sparse or
ambiguous material; bootstrap turns produce no durable claims
- volatile light-estimator hints and durable consolidation hints are
reconciled deterministically before reaching generation
- contradiction storms trigger cool-down rather than reactive
re-extraction
- durable consolidation publishes `ReciprocalRoleUpdateEvent` on the
cognitive event bus for audit, without mutating any other module's
state
- M12.2 evidence cards and reply-policy hints can affect generation
only through the existing response evidence contract; `no_action`
is preferred over low-quality hints
- disabling M12.2 yields byte-identical output to an
`M11 + M12.0 + M12.1` baseline on fixed transcripts
- Phase A / B / C acceptance artifacts each replay byte-identically
from recorded extractor fixtures
- no forbidden memory modules are imported and no mutation methods are
called on M11, M12.0, or M12.1-owned state
