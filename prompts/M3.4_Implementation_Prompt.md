# M3.4 Process-Valence Motivation Loop Prompt

## Scope

Introduce a process-valence motivation layer that distinguishes unresolved wanting, closure, post-closure decay, and boredom-driven reorientation. The mechanism must change inquiry persistence and action bias, not merely raise a generic exploration reward.

## Non-Goals

- No fixed `lazy_drive` or hard-coded curiosity drive.
- No fake motivation state that appears only in logs.
- No replacement of the existing goal stack, inquiry scheduler, or slow-learning pipeline.

## Canonical Seeds

- `34`
- `134`

## Acceptance Gates

- `schema`: process-valence state survives round-trip in runtime payloads and reports.
- `determinism`: fixed focus transitions produce the same phase progression.
- `causality`: unresolved targets persist longer and closed targets cool off after confirmation.
- `ablation`: removing process-valence weakens persistence, closure drop-off, and boredom reorientation.
- `stress`: low-tension and closure phases stay bounded and do not collapse into permanent exploration.
- `regression`: M3.1-M3.3 acceptance behavior remains intact.

## Audit Risks

- Exploration reward inflation is presented as process motivation.
- Closure never reduces attention to the resolved object.
- Boredom is indistinguishable from random exploration noise.

## Repair Loop

1. Run M3.4 motivation, ablation, and acceptance tests.
2. Regenerate artifacts and inspect the machine-readable report.
3. Fix causal failures before cosmetic report issues.
4. Re-run M3.4 tests plus listed regressions after every repair.
5. Stop only when the acceptance report is `PASS`.
