# OpenFEP M6.1 Implementation Prompt

## Milestone

M6.1: Cognitive Event Layer

## Goal

Add a lightweight cognitive event layer on top of the existing runtime so key internal dialogue steps become observable, filterable, serializable, and traceable.

This milestone must not alter decision behavior. It creates infrastructure for M6.2-M6.7.

## Work Principles

- Reuse the spirit and serialization style of `segmentum/io_bus.py`.
- Events are internal observability infrastructure, not consciousness or policy.
- Every event type must have at least one consumer: trace, state update, prompt assembly audit, or evaluation.
- No event publication may be required for normal behavior to work.
- No consumer means no side effects.

## Required Code Changes

### 1. Add cognitive event module

Add:
- `segmentum/cognitive_events.py`

Implement:
```python
@dataclass(frozen=True)
class CognitiveEvent:
    event_id: str
    event_type: str
    turn_id: str
    cycle: int
    session_id: str
    persona_id: str
    source: str
    timestamp: str
    salience: float
    priority: float
    ttl: int
    payload: dict[str, object]

    def to_dict(self) -> dict[str, object]: ...
    @classmethod
    def from_dict(cls, payload: Mapping[str, object]) -> "CognitiveEvent": ...

class CognitiveEventBus:
    def publish(self, event: CognitiveEvent) -> CognitiveEvent: ...
    def events(self) -> tuple[CognitiveEvent, ...]: ...
    def filter(
        self,
        *,
        event_type: str | None = None,
        source: str | None = None,
        min_salience: float | None = None,
    ) -> tuple[CognitiveEvent, ...]: ...
    def clear_expired(self) -> None: ...
    def to_dict(self) -> dict[str, object]: ...
    @classmethod
    def from_dict(cls, payload: Mapping[str, object] | None) -> "CognitiveEventBus": ...
```

MVP event types:
- `ObservationEvent`
- `MemoryActivationEvent`
- `DecisionEvent`
- `CandidatePathEvent`
- `PathSelectionEvent`
- `PromptAssemblyEvent`
- `GenerationEvent`
- `OutcomeEvent`

### 2. Add event factory helpers

Helpers must create deterministic IDs from session, turn, cycle, event type, source, and sequence index. Do not use wall-clock time for identity.

Timestamp may be ISO text for trace readability, but deterministic tests must not depend on it.

Event identity must include `persona_id` in payload and filtering metadata. `persona_id` is required for multi-persona isolation and must not be inferred only from display name.

### 3. Integrate without behavior change

Integration points:
- `DialogueObserver.observe` or `run_conversation` publishes `ObservationEvent`
- `SegmentAgent.decision_cycle_from_dict` result publishes `DecisionEvent`, `CandidatePathEvent`, and `PathSelectionEvent`
- `build_fep_prompt_capsule` caller publishes `PromptAssemblyEvent`
- generator wrapper or conversation loop publishes `GenerationEvent`
- outcome classifier publishes `OutcomeEvent`

Events may be assembled in `run_conversation` first if that is the least invasive path.

### 4. Preserve compatibility

- Existing `PerceptionBus` and `ActionBus` behavior must not change.
- Existing tests expecting old diagnostics and generation output must continue to pass.
- Event bus must be optional and safe to ignore.

## Boundary Constraints

- Do not replace `PerceptionBus` or `ActionBus`.
- Do not put raw prompt text into events unless an explicit debug flag exists.
- Do not store full raw conversation history in event payloads.
- Do not use the event bus as a second source of policy truth.
- Do not require event consumers for `run_conversation` to return a reply.
- Do not publish persona-scoped events without `persona_id`.
- Do not let events from one persona be consumed by another persona's conscious artifacts.

## Acceptance Criteria

- Events serialize to JSON-safe dictionaries.
- Events include `turn_id`, `cycle`, `session_id`, `persona_id`, `source`, `salience`, `priority`, and `ttl`.
- Event order is deterministic for the same scripted dialogue and seed.
- The event layer can be enabled in a dialogue turn without changing chosen action or generated text under the rule generator.
- There is at least one planned or implemented consumer for each MVP event type.
- Existing M5.3 and M5.6 dialogue tests remain stable.
