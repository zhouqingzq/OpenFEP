# OpenFEP M8.9 Work Prompt

## Milestone

M8.9: MVP Architecture Contract Hardening

## Goal

Insert a small bridge milestone between M8 Anchored Memory Contract and M9 Memory Dynamics Integration. The goal is not to add a new cognition runtime. The goal is to align the current M6-M8 implementation with the MVP architecture contract:

```text
Chat / self-thought / retrieval evidence
-> CognitiveEventBus
-> CognitiveLoop
-> attention and budget
-> cognitive state update
-> memory write intent / retrieval evidence
-> response evidence contract
-> prompt-safe generation
-> outcome feedback
```

M8.9 should make the architecture harder to accidentally bypass.

## Architecture Principle

Use this ownership statement throughout the implementation:

```text
The cognitive loop is the only orchestrator of current self-cognition.
FEP, memory, self-thought, and meta-control may calculate or propose updates.
Durable self-cognition changes must pass through a named owner with source,
reason, confidence, and traceability.
```

## Required Work

### 1. Freeze StateOwnership Contract

Add or update a small architecture document and, where practical, tests that map:

| State surface | Owner | Update path | Prompt eligibility |
| --- | --- | --- | --- |
| `CognitiveStateMVP` | cognitive state reducer | `CognitiveLoop.consume_and_update` | compressed only |
| `SubjectState` | subject state owner | explicit derivation or later patch commit | summary only |
| `SelfModel` | slow persona model owner | slow consolidation / audited update | compact self-prior only |
| `MemoryStore` / anchored memory | memory owner | write intent or explicit commit | evidence-filtered only |
| `MetaControlSignal` | meta-control owner | deterministic derivation from state | bounded guidance only |
| `FEPPromptCapsule` | prompt adapter | compressed summary builder | prompt-safe fields only |

Acceptance:

- A reviewer can answer who owns every self-related state surface.
- Prompt-facing code cannot treat raw state objects as prompt text.

### 2. Introduce MemoryWriteIntent For Anchored Dialogue Facts

Current issue: dialogue fact extraction can directly write anchored memory in `run_conversation`.

Required behavior:

```text
DialogueFactExtractionEvent
-> MemoryWriteIntent
-> MemoryStore commit
-> MemoryWriteResultEvent
```

Keep the existing anchored memory semantics:

- asserted means "speaker said this", not objective truth
- hypothesis cannot become explicit fact
- retracted beats asserted/corroborated
- forbidden and private items cannot become explicit memory context
- agent self-utterances must not be treated as user facts

Acceptance:

- Each anchored item has source turn / utterance and write-intent trace.
- Tests still pass for M8 anchored memory and citation audit.
- Direct helper storage is either removed or wrapped as an explicit commit path.

### 3. Build ResponseEvidenceContract MVP

Add a generation-facing evidence boundary with these buckets:

```text
current_input_facts
anchored_facts
retrieved_memories
external_evidence
unverified_claims
unknowns
forbidden_assumptions
style_constraints
```

Contract:

- Specific factual details must come from current input, anchored memory, retrieved memory, or external evidence.
- Unverified details must be rendered as uncertain.
- Missing long-term memory means unknown stance.
- Style constraints may warm or soften the reply; they may not promote unknown details into facts.

Acceptance:

- PromptBuilder / ResponseGenerator can consume the contract or a compact rendering of it.
- Memory citation audit can point to the relevant evidence bucket.
- A reply that makes an unsupported specific memory claim is flagged or retried.

## Non-goals

- Do not replace `SegmentAgent`.
- Do not replace `CognitiveEventBus`, `CognitiveLoop`, or `FEPPromptCapsule`.
- Do not implement full M9 memory dynamics here.
- Do not implement self-initiated exploration agenda here.
- Do not build `Conscious Projection Runtime` here.
