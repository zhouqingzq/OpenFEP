# M3.5 Emergent Cognitive Styles Prompt

## Scope

Make cognitive styles emerge from parameter surfaces, memory pressure, effort allocation history, and uncertainty handling. The system must exhibit stable style differences without introducing an explicit `lazy_drive`.

## Non-Goals

- No hard-coded fixed style labels that bypass history.
- No explicit laziness drive or shortcut reward.
- No standalone benchmark disconnected from runtime state and persistence.

## Canonical Seeds

- `35`
- `135`

## Acceptance Gates

- `schema`: style surfaces, effort allocation history, and summaries survive round-trip.
- `determinism`: the same subject history yields the same style snapshot.
- `causality`: style-sensitive surfaces change action or compute-allocation bias.
- `ablation`: removing style adaptation weakens stable cross-subject differences.
- `stress`: restart and continued execution do not produce unexplained style drift.
- `regression`: M3.1-M3.4 acceptance behavior remains intact.

## Audit Risks

- Style is implemented as a fixed label rather than a learned tendency.
- “Smart lazy” behavior only appears because a hidden reward is hard-coded.
- Style claims are not supported by parameter, memory, and history evidence together.

- `artifacts/m35_cognitive_style_stress.json`

## Repair Loop

1. Run M3.5 style, ablation, and acceptance tests.
2. Regenerate artifacts and inspect the acceptance report.
3. Fix causal or persistence failures before polish issues.
4. Re-run M3.5 tests and listed regressions after every repair.
5. Stop only when the acceptance report is `PASS`.
