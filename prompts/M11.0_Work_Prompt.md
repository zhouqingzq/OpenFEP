# OpenFEP M11.0 Work Prompt

## Milestone

M11.0: User Generative Model And Value Memory Dynamics

## Goal

Replace the previous M11 conscious-projection milestone with a user-modeling
milestone for the chat-facing digital persona.

The digital persona must maintain a bounded generative model of the current
interlocutor. The value of any memory written about the user is the degree to
which it reduces future free energy around:

- user intent
- user preferences and boundaries
- user cognitive/personality patterns
- source reliability and claim trustworthiness
- relationship state and continuity
- expected user reaction to the persona's replies

M11 must make this user model value-sensitive without:

- turning user claims into unverified facts
- leaking raw memory or scores into prompts
- creating a second hidden cognition layer
- breaking the existing chat memory path

## Stand-Alone Posture

M11 ships as a **stand-alone module** that lives beside the existing chat
memory stack, not on top of it. The current memory-dynamics layer
(`segmentum/memory_dynamics.py`, `segmentum/value_memory.py`,
`segmentum/memory_retrieval.py`) is **not** the substrate for M11. Specifically:

- `compute_cue_match` is substring + whitespace-Jaccard surface matching.
- `_mood_match` uses an English-only lexicon and silently returns 0 for
  Chinese conversations.
- `ValueMemoryEvaluator._future_reuse_gain / _cost_reduction /
  _error_avoidance_gain / _transferability / _recurrence_potential` are
  token-bag heuristics, not free-energy quantities.

M11 must not inherit, wrap, or compose with these scoring functions. Doing
so would dress legacy keyword heuristics in user-model vocabulary and
produce a "deterministic float" that is structurally a vibes signal.

M11 introduces its own minimal user-model state, its own LLM extraction
boundary, and its own deterministic accumulation ledgers. The existing
chat memory path (short_term, long_term, anchored items, retrieval cards)
keeps working unchanged.

## Core Contract

Required path per turn:

```text
user turn
-> single LLM extractor call
   (returns enums + bounded structured lists, NO floats)
-> deterministic ledger updates
   (UserModel, PredictionLedger, SourceReliabilityLedger)
-> deterministic memory value composition
   (ordinal band + reliability + recency -> bounded float)
-> UserModel evidence cards exposed to generation
   (prompt-safe, evidence-bounded)
-> reply policy adjustment
   (clarification, hedging, length, repair priority)
```

The system must keep these things separate at the boundary:

```text
claim content
speaker / source
modality (factual | roleplay | joke | hypothetical | request | command)
evidence quote ids
ordinal confidence band (low | med | high)
prediction status (confirmed | violated | uncertain | pending)
hypothesis activation relation (activates | contradicts | irrelevant)
```

High free energy does not mean a claim is true. It means the event is useful
for model calibration:

- repeated honesty lowers uncertainty and increases source reliability
- a detected lie raises immediate free energy and memory value, but lowers
  source reliability for the relevant domain
- contradictory claims should be retained as evidence of uncertainty, not
  collapsed into a false "fact"

## Architecture Boundary

### Modules M11 OWNS (new)

```text

  user_model.py            # UserModel dataclass, serialize / load
  prediction_ledger.py     # append-only prediction + judgment log
  reliability_ledger.py    # SourceReliability, Bayesian update

  value_composer.py        # ordinal band + reliability -> bounded float
  evidence_cards.py        # UserModel -> MemoryEvidence-compatible cards

```

### Modules M11 MUST NOT IMPORT

```text
segmentum/memory_dynamics.py
segmentum/value_memory.py
segmentum/memory_retrieval.py
```

If M11 finds itself wanting a function from these modules, copy a clean
ledger-style version into `segmentum/user_model/` instead of importing.

### Modules M11 MAY READ-ONLY

```text
segmentum/memory_evidence.py        # MemoryEvidence output dataclass
segmentum/dialogue/runtime/...      # the MVP runtime, for hooking the
                                    # extractor and exposing cards
```

M11 may attach its evidence cards to the existing evidence pipeline, but
must not edit the legacy memory dynamics or replace existing retrieval.

## Required Work

### 1. Define UserModel MVP

Create `segmentum/user_model/user_model.py`. Bounded, serializable, explicit.

Minimum fields:

```text
user_id
display_name
cognitive_style_hypotheses
personality_hypotheses
preference_hypotheses
boundaries_and_dislikes
relationship_state
source_reliability_by_domain
claim_history_summary
prediction_history_summary
open_uncertainties
last_updated_turn_id
```

Each hypothesis must carry:

```text
hypothesis_id
content_summary
domain
confidence_band            # low | med | high (NOT a float)
evidence_refs              # turn_id + quote_id list, bounded
contradiction_refs
last_confirmed_turn
last_violated_turn
permitted_use              # explicit_fact | cautious_hypothesis | strategy_only | forbidden
```

Acceptance:

- UserModel distinguishes verified fact, user-stated claim, inferred
  hypothesis, contradiction, and unknown.
- UserModel can be saved and loaded as plain JSON without reading prompt
  text or trace dumps.
- A single user utterance never becomes a stable personality fact unless
  multi-turn evidence and band-promotion thresholds are met.

### 2. Add UserPredictionLedger

`segmentum/user_model/prediction_ledger.py`. Append-only.

Minimum prediction types:

```text
intent_prediction
preference_prediction
reaction_prediction
claim_reliability_prediction
relationship_state_prediction
needed_memory_prediction
```

Minimum entry fields:

```text
prediction_id
turn_id
prediction_type
predicted_value_summary
confidence_band                      # low | med | high
evidence_refs
validation_status                    # pending | confirmed | violated | uncertain
observed_outcome_summary
calibration_need_band                 # low | med | high (LLM-judged surprise proxy)
```

Prediction proposals for future turns are explicit records, not implicit
side effects. They may be generated deterministically from high-confidence
UserModel state, or proposed by the extractor and accepted by deterministic
gates.

Minimum prediction proposal fields:

```text
proposal_id
proposed_prediction_type
predicted_value_summary
confidence_band
source_hypothesis_ids
source_judgment_ids
expires_after_turns
accepted                         # deterministic gate result
rejection_reason
```

Proposal admission rules (deterministic, all constants in `hyperparams.py`):

- A per-turn proposal quota bounds how many proposals can be admitted
  per turn. Excess proposals are rejected with
  `rejection_reason = "proposal_quota_exceeded"` and never reach
  acceptance. This prevents the LLM from using the ledger as a scratchpad.
- A proposal whose `source_hypothesis_ids` or `source_judgment_ids`
  reference ids not present in the bounded snapshot is rejected with
  `rejection_reason = "unknown_source_id"`.
- A proposal whose `expires_after_turns` has elapsed without a matching
  judgment is closed with `validation_status = "uncertain"` and
  `rejection_reason = "expired"`.
- The deterministic gate's inputs are restricted to enums, counts,
  ledger state, and `hyperparams.py`. The gate must not read raw turn
  text or extractor explanations.

Acceptance:

- The ledger can answer "which predictions about user X were confirmed,
  violated, or still pending in the last N turns".
- The ledger can show which next-turn predictions were proposed, accepted,
  rejected, or expired, with the rejection_reason for each rejected
  proposal.
- Ledger entries reference turns and quote ids, never store full transcripts.
- Ledger updates are pure functions over structured extractor output,
  existing UserModel state, existing ledger state, and hyperparameters.
- The per-turn proposal quota, source-id validation, and expiration
  closure are reproducible across runs given the same extractor
  fixture.

### 3. Add SourceReliabilityLedger

`segmentum/user_model/reliability_ledger.py`. **The only durable latent-trust
float producer in M11.**

`value_composer.py` may produce derived `value_score` floats for write
decisions, diagnostics, and acceptance artifacts. Those scores are not a second
long-term trust state and must be reproducible from ledger state plus
hyperparameters.

Minimum state per (user, domain):

```text
domain
alpha                              # confirmed-equivalent count incl. prior
beta                               # violated-equivalent count incl. prior
reliability                        # alpha / (alpha + beta), bounded by max delta
last_updated_turn
update_reason
```

Domain examples:

```text
self_reported_preferences
self_reported_history
task_requirements
emotional_state
technical_claims
social_relationship_claims
```

Update rules (deterministic, no text inspection):

- Confirmed judgment in domain D: `alpha += confirm_weight`
- Violated judgment in domain D: `beta  += violate_weight` (asymmetry: lies cost more)
- No-event domain at turn t: time-decay `alpha, beta` toward prior using
  configured `half_life_turns`
- New `reliability` is clamped so single-turn change never exceeds
  `max_delta_per_turn`
- `modality in {roleplay, joke, hypothetical}` events do NOT update the
  factual reliability ledger
- Reliability is domain-scoped; events in domain A never update domain B

All constants live in `hyperparams.py` with per-field docstrings explaining
operational meaning. Magic numbers inside update functions are forbidden.

Acceptance:

- Repeated confirmed claims in a domain monotonically raise that domain's
  reliability (subject to per-turn cap).
- A single violated claim drops reliability by at most `max_delta_per_turn`.
- Roleplay or joke modality cannot damage factual-domain reliability.
- Domain isolation is provable: events in one domain do not move another.
- Long silence causes reliability to decay toward prior, not freeze at peak.

### 4. Add LLM Extractor With Strict Output Schema

`segmentum/user_model/llm_extractor.py`. Single LLM call per turn.

Strict output JSON schema (the LLM MUST conform; non-conforming output is
rejected and the turn falls back to a no-op extraction):

```json
{
  "claims_made": [
    {
      "id": "string",
      "domain": "self_reported_preferences | self_reported_history | task_requirements | emotional_state | technical_claims | social_relationship_claims",
      "modality": "factual | roleplay | joke | hypothetical | request | command",
      "content_summary": "string, <= 120 chars",
      "evidence_quote_ids": ["string"],
      "confidence_band": "low | med | high"
    }
  ],
  "prediction_judgments": [
    {
      "prediction_id": "string",
      "status": "confirmed | violated | uncertain",
      "evidence_quote_ids": ["string"]
    }
  ],
  "prediction_proposals": [
    {
      "id": "string",
      "prediction_type": "intent_prediction | preference_prediction | reaction_prediction | claim_reliability_prediction | relationship_state_prediction | needed_memory_prediction",
      "predicted_value_summary": "string, <= 120 chars",
      "confidence_band": "low | med | high",
      "source_hypothesis_ids": ["string"],
      "source_judgment_ids": ["string"],
      "expires_after_turns": 1
    }
  ],
  "hypothesis_activations": [
    {
      "hypothesis_id": "string",
      "relation": "activates | contradicts | irrelevant"
    }
  ],
  "contradiction_detections": [
    {
      "claim_id": "string",
      "conflicts_with_memory_id": "string",
      "severity_band": "minor | major"
    }
  ],
  "calibration_need_band": "low | med | high",
  "memory_value_band": "low | med | high",
  "surprise_explanation": "string, <= 200 chars"
}
```

The extractor prompt must:

- pass only a bounded snapshot: current turn + last K turns + active
  hypotheses (id + summary, not full evidence) + open predictions
- forbid the LLM from inventing prediction_ids or hypothesis_ids that are
  not in the snapshot
- allow new `prediction_proposals[*].id`, but require every proposal source id
  to reference an existing hypothesis or judgment from the bounded snapshot
- reject any output containing fields not in this schema
- reject any numeric float anywhere in the output

The extractor never outputs floats. The extractor never decides
write/promote/decay. Those decisions belong to the deterministic layer.

`calibration_need_band` is not a literal free-energy calculation. It is an
LLM-extracted ordinal cue that an event may be useful for model calibration.
The reproducible free-energy proxy for M11 comes from deterministic ledger
state: prediction violations, contradictions, reliability movement, and
accepted/rejected prediction proposals.

### 5. Add Memory Value Composer

`segmentum/user_model/value_composer.py`. Deterministic, no text inspection,
no token sets.

Inputs:

- `memory_value_band` (LLM enum)
- `source_reliability` (deterministic float from ledger)
- `recency_weight` (deterministic, function of turn distance)
- `contradiction_unresolved` (boolean from ledger)
- `privacy_or_safety_flag` (boolean from policy)

Output:

```text
value_score                # bounded float, fully reproducible
write_target               # none | short_term | user_model_patch | long_term_user_model
gating_reasons             # list of strings explaining each contribution
```

Composition rule (illustrative, hyperparams in `hyperparams.py`):

```text
band_prior   = {low: 0.2, med: 0.5, high: 0.8}[memory_value_band]
trust_factor = 0.5 + 0.5 * source_reliability      # in [0.5, 1.0]
penalty      = 0.4 if contradiction_unresolved else 0.0
penalty     += 0.6 if privacy_or_safety_flag else 0.0
value_score  = clamp(band_prior * trust_factor * recency_weight - penalty, 0, 1)

write_target = (
  "long_term_user_model" if value_score >= long_term_threshold and confidence_band == "high"
  else "user_model_patch"  if value_score >= patch_threshold
  else "short_term"        if value_score >= short_term_threshold
  else "none"
)
```

All thresholds and weights live in `hyperparams.py` with rationale.

Acceptance:

- Same inputs produce byte-identical `value_score` across runs.
- A stable preference whose band stays "med" but whose source reliability
  rises across turns sees its `value_score` rise monotonically with
  reliability, without text changes.
- Privacy or safety flag dominates: even with band="high" and reliability=1,
  privacy flag prevents long_term promotion.

### 6. Provide UserModel Evidence Cards To Generation

`segmentum/user_model/evidence_cards.py`.

Convert active UserModel hypotheses + recent confirmed/violated judgments
into cards compatible with `segmentum/memory_evidence.py::MemoryEvidence`.

Each card exposes only:

```text
memory_id                    # hypothesis_id or judgment_id
memory_class                 # one of the existing MemoryEvidence classes
content_summary              # bounded
confidence_band              # low | med | high (also mirrored to confidence float for compat)
source_reliability           # float, but exposed as ordinal label in any prompt-facing surface
permitted_use
contradiction_status
why_retrieved                # short string, NOT internal scores
```

Acceptance:

- Generation receives compact UserModel cards, not raw hypothesis dumps,
  not ledger numbers, not LLM extractor JSON.
- Cards are ordered by `permitted_use` priority, then `confidence_band`,
  then `source_reliability`.
- Cards never expose the LLM's `surprise_explanation` verbatim; it is
  diagnostic only.

### 7. Connect Value Dynamics To Reply Policy

UserModel state must causally affect reply policy. Examples:

- If user reliably dislikes long answers (preference hypothesis confirmed
  multiple turns, reliability above threshold), prefer a shorter reply.
- If user's intent prediction was violated last turn, prefer a clarifying
  question this turn.
- If `social_relationship_claims` reliability is below threshold, soften
  evidence language for that domain.
- If a relationship-repair cue is the strongest active hypothesis,
  prioritize repair over explanation.
- If user-model confidence is high and recent outcomes confirm it, reduce
  unnecessary clarification.

Acceptance:

- The reply policy effect is visible in a diagnostic trace (which
  hypothesis or judgment caused which adjustment).
- The reply itself never exposes hidden scores, JSON, or internal labels.
- Reply policy adjustments are pure functions of UserModel state +
  composer output, not of raw LLM extractor JSON.

## LLM Use Boundary

**The LLM is allowed to produce:**

- enum-typed classifications (modality, domain, status, relation, severity)
- bounded structured lists (claims, judgments, prediction proposals,
  activations, contradictions)
- ordinal bands (low | med | high) for confidence, calibration_need,
  memory_value
- short string explanations for diagnostics (`surprise_explanation`)
- evidence quote ids referencing turn content

**The LLM is forbidden to produce:**

- continuous floats anywhere in any field of any output
- `value_score`, `source_reliability`, `prediction_error`,
  `future_path_utility`, `expected_free_energy_delta`, `retention_pressure`,
  `retrieval_score`, or any other float consumed by ledgers or composer
- decisions about promotion / quarantine / decay
- write_target selection
- retrieval ranking
- update rules, threshold tuning, or hyperparameter values

The deterministic layer must run end-to-end on stubbed extractor output
without invoking any LLM. This is mandatory; it is what makes M11 testable
and reproducible.

## Determinism Reliability Contract

The deterministic layer (`reliability_ledger`, `prediction_ledger`,
`value_composer`, `evidence_cards`) must satisfy:

1. **Pure functions over structured inputs only.**
   - Inputs are enums, counts, ledger state, hyperparams.
   - No raw text inspection, no regex, no keyword sets.
   - Property test: identical structured inputs with different
     `evidence_text` produce identical outputs.

2. **All constants live in `hyperparams.py`** with per-field docstrings
   stating the operational meaning. No inline magic numbers.

3. **Every update is one of:**
   - count or ratio over enum events
   - bounded EMA with documented half-life
   - Bayesian posterior (Beta or Dirichlet) over enum events
   - bounded clamp with documented per-turn delta cap

4. **Property tests required per dynamics function:**
   - boundedness in declared range
   - monotonicity under monotone enum input
   - domain isolation
   - per-turn delta bound
   - no-text-dependency
   - decay-under-silence

5. **Replay fixtures per dynamics function:**
   - committed pairs of (input judgments JSON, expected state trace JSON)
   - any change to a function that alters a trace requires the fixture to
     be updated and a `rationale.md` entry to be added

6. **Calibration audit:**
   - at least 5 hand-authored multi-turn scenarios:
     stable_preference, detected_lie, roleplay_isolation,
     contradiction, long_silence
   - documented per-scenario success thresholds
   - audit report shipped as part of M11 acceptance artifact

The deterministic layer is "reliable" iff (1)-(5) pass and (6) meets the
documented thresholds. M11 acceptance must cite the audit report.

## Non-Interference Guarantee

M11 must not change observable behavior of the existing chat memory path
when the M11 module is disabled or the UserModel for a user does not yet
exist.

Required:

- A configuration flag (default off in legacy personas, default on in new
  personas) gates all M11 reads, writes, and evidence-card emission.
- When the flag is off, the runtime behaves exactly as before M11.
- When the flag is on but no UserModel state exists for the user, the
  runtime initializes a clean UserModel and proceeds without altering
  any pre-existing chat memory rows.
- M11 never deletes, edits, or re-scores any row owned by
  `MemoryStore`, `LongTermMemory`, or the existing anchored memory layer.

Acceptance:

- An existing MVP persona with M11 flag off produces byte-identical chat
  outputs to its pre-M11 baseline on a fixed transcript when run with a
  rule-mode generator, stub generator, or recorded LLM fixture.
- A new persona with M11 flag on produces UserModel state plus the same
  chat baseline plus added evidence cards, but never modifies legacy
  memory rows.

## Non-goals

- Do not import or wrap any function from `segmentum/memory_dynamics.py`,
  `segmentum/value_memory.py`, or `segmentum/memory_retrieval.py`.
- Do not change retrieval ranking for the existing chat memory path.
- Do not migrate or rewrite legacy memory rows.
- Do not let the LLM produce any continuous float in any output field.
- Do not let the LLM decide promotion, quarantine, decay, or retrieval.
- Do not build a surveillance profile or infer sensitive traits without
  user-provided evidence and permitted use.
- Do not make global moral judgments about the user from isolated events.
- Do not treat high free energy as truth.
- Do not promote LLM speculation to stable user facts.
- Do not leak hidden diagnostics, raw memory dumps, source reliability
  numbers, or LLM extractor JSON into user-visible replies.
- Do not revive the old M11 conscious-projection scope in this milestone.

## Completion Criteria

M11.0 is complete when:

- the chat-facing digital persona maintains a bounded, auditable user
  generative model whose updates, value composition, and reply policy
  are driven by deterministic accumulation over LLM-extracted enums
- the deterministic layer satisfies the Determinism Reliability Contract
- the LLM Use Boundary is enforced in code, not only in documentation
- the Non-Interference Guarantee holds against a baseline persona
- the calibration audit report passes documented thresholds for all
  five scenarios
- no symbol from `memory_dynamics.py`, `value_memory.py`, or
  `memory_retrieval.py` appears in any M11-owned import statement
