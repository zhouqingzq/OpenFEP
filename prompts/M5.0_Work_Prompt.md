# OpenFEP M5.0 Implementation Prompt

## 里程碑名称

M5.0: Chat Data Pipeline（聊天数据管线）

## 目标

构建一条从原始聊天日志到结构化、按用户组织的对话数据集的处理管线。
这是纯数据工程里程碑，不涉及人格建模、FEP 集成或任何认知机制。
产出物必须干净、可检查、可版本化，供后续 M5.1-M5.2 使用。

## 工作原则

- 不引入任何认知/人格/FEP 依赖，这是一个独立的数据处理模块
- 必须处理真实数据的脏情况：混合繁简体、单字消息、广告、PII
- ultra-short 消息（"嗯"、"哈哈"）标记但不删除——它们携带行为信号
- 输出格式为结构化 JSON，per-user 组织
- 必须支持 deterministic replay（同输入同输出）
- 必须保持现有测试兼容，不修改任何现有模块

## 输入数据格式

```
2023-10-22-12:42:01 INFO   MessageSender::OnData message type: 0, sender uid: 2653466, reciever uid: 4172441, body: 嗯
2023-10-22-12:42:02 INFO   MessageSender::OnData message type: 0, sender uid: 2646798, reciever uid: 5363609, body: 哈哈
```

字段：
- 时间戳: `YYYY-MM-DD-HH:MM:SS`
- 日志级别: `INFO`
- 来源: `MessageSender::OnData`
- message type: 整数（0 = 文本）
- sender uid: 发送者 ID
- reciever uid: 接收者 ID（注意原始数据拼写为 reciever）
- body: 消息正文（可能包含换行、URL、表情符号）

数据规模：约 1000 万行。

    receiver_uid: int

    raw_line: str  # 保留原始行用于调试

def parse_line(line: str) -> ChatMessage | None:
    """解析单行日志。格式不匹配返回 None。"""

def parse_file(path: Path, *, encoding: str = "utf-8") -> Iterator[ChatMessage]:
    """流式解析整个日志文件。跳过解析失败的行并记录警告。"""
```

要求：
- 正则匹配要兼容时间戳中的不同分隔符
- body 可能包含逗号、冒号等需要特殊处理的字符
- 解析失败的行跳过但计入统计（报告解析成功率）
- 支持流式处理，不要一次性加载全部 100w 行到内存

class ConversationSession:

    metadata: dict[str, object]  # 包含 turn_count, duration_seconds 等

def build_sessions(
    messages: Iterable[ChatMessage],
    *,
    gap_threshold_minutes: int = 30,
) -> dict[tuple[int, int], list[ConversationSession]]:
    """
    将消息按 (uid_a, uid_b) 配对分组，按时间排序，
    以 gap_threshold 切分为独立会话。
    """
```

要求：
- uid 对必须规范化为 `(min, max)` 避免重复
- 时间排序必须稳定
- 每个 session 记录元数据：轮次数、持续时间、双方消息数比

### 3. 质量过滤器 `segmentum/chat_pipeline/quality_filter.py`

```python
@dataclass
class FilterResult:
    message: ChatMessage
    kept: bool
    tags: list[str]  # e.g., ["ultra_short", "spam", "pii_redacted"]
    redacted_body: str  # PII 脱敏后的正文

class QualityFilter:
    def __init__(
        self,
        *,
        spam_url_threshold: int = 2,
        spam_template_patterns: list[str] | None = None,
        normalize_chinese: str | None = "simplified",  # or "traditional" or None
    ): ...

    def filter_message(self, msg: ChatMessage) -> FilterResult: ...
    def filter_session(self, session: ConversationSession) -> ConversationSession: ...
```

过滤规则：
- **spam**: body 含 >= 2 个 URL，或匹配预定义推广模板
- **ultra_short**: body 长度 <= 2 且为常见语气词（标记，不删除）
- **PII**: 正则匹配手机号（11位数字）、邮箱、URL，替换为 `[PHONE]`/`[EMAIL]`/`[URL]`
- **繁简转换**: 使用 `opencc` 或等价库
- 过滤后的 session 如果有效轮次 < 2 则整个丢弃

## 验收标准

1. **解析完整性**: 100w 行日志解析成功率 >= 99.5%，解析失败的行被记录且不中断流程。
2. **会话重建正确性**: 人工抽检 20 个随机会话，100% 时间排序正确、uid 配对正确。
3. **质量过滤有效性**: 在人工标注的 200 条消息样本上，spam 召回率 >= 80%，非 spam 误杀率 < 5%。
4. **PII 脱敏**: 手机号、邮箱等在输出中不出现原始值（抽检 100 条含数字的消息）。
5. **用户过滤**: 不合格用户（< 200 条消息或 < 3 个伙伴）被排除，报告中注明数量。
6. **内存效率**: 100w 行处理峰值内存 < 2GB（流式处理，不全量加载）。
7. **确定性**: 同输入同参数，两次运行输出完全相同（byte-identical JSON）。
8. **独立性**: `chat_pipeline` 模块不 import segmentum 中任何其他模块。
9. **管线报告**: 生成包含以下统计的 JSON 报告：
   - 总行数、解析成功/失败数
   - 总消息数、总会话数、总用户数
   - 合格用户数、被过滤用户数
   - 每种过滤标签的消息数
   - 处理耗时
