# OpenFEP M9.0 Work Prompt

## Milestone

M9.0: Memory Dynamics Integration

## Goal

Integrate memory dynamics with the MVP cognitive loop so memory is not just storage and retrieval. Memory must become a bounded, value-weighted, cue-triggered state backend that influences cognition through evidence, interference, decay, consolidation, and auditable state proposals.

M9 builds on:

- M6 cognitive loop
- M7 bounded causal control
- M8 anchored memory contract
- M8.9 architecture hardening

## Core Contract

Memory may affect current cognition, but it must not directly overwrite current self-cognition.

Required path:

```text
cue / gap / self-thought / outcome
-> memory retrieval or consolidation
-> MemoryRecallEvent / MemoryInterferenceEvent / MemoryWriteResultEvent
-> CognitiveLoop
-> StatePatchProposal when self-cognition should change
-> StateCommitEvent if accepted by the proper owner
```

## Required Work

### 1. Unify Memory Evidence Surfaces

Create one compact memory evidence structure consumed by cognitive state and generation:

```text
memory_id
memory_class
source_turn_id
source_utterance_id
speaker
status
confidence
retrieval_score
cue_match
salience
value_score
decay_state
conflict_status
permitted_use
content_summary
```

Acceptance:

- Anchored facts, retrieved episodic memory, semantic memory, and inferred hypotheses can all be represented without losing source or confidence.
- Generation can distinguish asserted user statement from verified fact.

### 2. Implement Value-Based Retention And Decay Integration

Memory retention should use value and cost:

```text
retention_pressure =
  identity_continuity_value
+ relationship_continuity_value
+ future_prediction_value
+ affective_salience
+ user_emphasis
- privacy_or_safety_penalty
- contradiction_penalty
- low_confidence_penalty
- maintenance_cost
```

Acceptance:

- High-value memories decay slower or are protected.
- Low-value memories decay faster or are excluded from retrieval.
- Decay decisions are logged with reasons.

### 3. Make Cue-Based Recall Explicit

Long-term memory must require cues. The cue may come from:

- current user input
- current task/gap
- self-thought event
- outcome feedback
- active relationship/entity
- explicit user request

Acceptance:

- No-cue retrieval returns no factual detail or an unknown stance.
- Cue-triggered recall includes cue trace and confidence.

### 4. Add MemoryInterference And Overdominance Feedback

Memory conflict should affect:

- memory retrieval gain
- caution
- assertiveness
- clarification bias
- response evidence contract

Acceptance:

- Conflicting memories reduce overconfident generation.
- Memory overdominance does not silently steer the reply without trace.

### 5. Add StatePatchProposal / StateCommitEvent MVP

Define:

```text
StatePatchProposal:
  proposal_id
  target_state
  operation
  field_path
  value_summary
  source_event_id
  reason
  confidence
  ttl

StateCommitEvent:
  commit_id
  proposal_id
  accepted
  owner
  reason
  committed_summary
```

Use first for durable updates involving `SubjectState` or selected `SelfModel` fields.

Acceptance:

- Durable self-cognition changes are no longer opaque side effects.
- Rejected or deferred patches are visible in trace.

## Non-goals

- Do not build self-initiated exploration agenda yet.
- Do not make every memory a prompt fact.
- Do not let memory bypass the response evidence contract.
- Do not claim subjective consciousness from memory projections.

## Suggested Tests

```text
test_memory_evidence_unifies_anchored_retrieved_and_hypothesis_sources
test_long_term_memory_requires_cue_for_specific_detail
test_high_value_memory_decays_slower_than_low_value_memory
test_memory_conflict_reduces_assertiveness_and_memory_reliance
test_memory_patch_proposal_requires_commit_owner
test_generation_unknown_when_memory_not_recalled
```

## Completion Criteria

M9.0 is complete when memory dynamics can causally influence cognition through events and patch proposals while preserving source, confidence, value, cue, and conflict boundaries.
