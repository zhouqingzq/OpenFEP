# OpenFEP M13.1 Work Prompt

## Milestone

M13.1: MVP-Local Boredom And Exploration Bias

## Goal

Add a bounded boredom signal to the MVP chat runtime. In this system,
boredom is not "nothing to do." It is a resource-constrained signal that the
current candidate behavior set lacks enough useful gradient:

```text
low novelty
+ low expected information gain
+ high environmental predictability
+ repeated action pattern
+ weak candidate margin
+ weak progress signal
-> boredom_level
-> exploration_bias
-> ask / reframe / switch angle / propose small next target
```

M13.1 should make Hu Tao less passively reactive when the current path is
stale, while staying bounded, auditable, and prompt-safe.

## Dependencies

M13.1 builds on M13.0:

- `m13_drive_state`
- `M13DriveEvaluationEvent`
- prompt-safe `drive_guidance`
- post-turn audited patch path

If M13.0 is not present, M13.1 may introduce the shared M13 state file and
event helpers, but it must not skip behavioral-pull tests.

## Architecture Position

M13.1 remains MVP-local:

```text
MVPDialogueRuntime.run_turn
-> conscious plan
-> memory_dynamics / retrieval / evidence judgment
-> M13DriveEvaluator
-> M13BoredomEvaluator
-> M13DriveEvent + M13BoredomEvent
-> exploration control guidance
-> thinking prompt
-> post-turn state patch
```

The boredom evaluator must not directly generate visible text. It only
proposes exploration control.

## Non-Goals

- Do not implement asynchronous UI initiative. That is M13.3.
- Do not simulate real emotion or claim subjective boredom.
- Do not make the persona visibly say "I am bored" unless the user explicitly
  asks about the internal diagnostic surface.
- Do not force topic changes when the user has an explicit task.
- Do not use boredom to ignore direct user requests.
- Do not run extra unbounded LLM calls just to decide boredom.

## Required Work

### 1. Extend M13 State With Boredom Fields

Add or refine:

```text
m13_drive_state:
  boredom:
    boredom_level
    novelty_baseline
    recent_topic_fingerprints
    last_exploration_target
    exploration_cooldown
    stale_turn_count
    last_progress_signal
```

All fields are engineering proxies. They must be labeled as such in
diagnostics and tests.

M13.1 must reuse the canonical `m13_drive_state.recent_action_trace` field
defined by M13.0 as the single recent-action sequence. Do not introduce a
parallel `recent_action_sequence` or any other action-history field; if more
metadata per action is needed, extend `recent_action_trace` rows in
coordination with M13.0.

Hard bounds inherited from M13.0 apply. In addition:

```text
recent_topic_fingerprints: 24 rows
stale_turn_count: capped at 12
exploration_cooldown: capped at 5 turns
```

### 2. Compute Novelty And Information-Gain Proxies

Use deterministic, bounded proxies first. Use the shared M13 topic fingerprint
helper from M13.0. M13.1 must not add an LLM call for topic identity, novelty,
or information gain.

Novelty decreases when:

- current semantic terms overlap with recent topic fingerprints
- the same reply action repeats across recent turns
- retrieved memory ids are mostly the same as recent retrieval ids
- user text is short and does not introduce new task, entity, or affect cues

Information gain increases when:

- `conscious_plan.memory_search_keywords` contains new terms
- evidence judgment is uncertain but has candidate evidence
- entity binding detects a new target person or alias correction
- M11/M12/M12.2 outputs indicate unresolved uncertainty or clarifying
  candidates
- the user asks for analysis, design, correction, or a new decision

Progress signal increases when:

- an open item closes
- expectation result is confirmed
- a high-uncertainty evidence stance becomes clearer
- a user correction is acknowledged and resolved

### 3. Compute Boredom Level

Suggested component shape:

```text
boredom_level =
  predictability
+ repetition_pressure
+ low_novelty
+ low_information_gain
+ low_progress
- explicit_task_pressure
- conflict_or_repair_pressure
- user_need_salience
```

The output must be clamped to `0.0..1.0`.

Bands:

```text
low:    0.00-0.34
medium: 0.35-0.66
high:   0.67-1.00
```

High boredom should not mean "stop answering." It means "the current path
needs a better gradient."

### 4. Derive Exploration Bias

Exploration actions may include:

```text
ask_clarifying_question
offer_new_angle
summarize_and_choose_next_target
retrieve_specific_memory
mark_uncertainty
propose_small_goal
shift_from_repetition_to_progress
```

The evaluator should produce:

```text
exploration_bias
preferred_exploration_mode
exploration_target
suppressed_repetition_actions
cooldown
ordinary_language_hint
```

Prompt-facing exploration guidance must be compact:

```text
maximum_drive_guidance_lines: 6
maximum_line_length: 160 characters
```

Do not dump the full boredom event, full M13 state, or numeric component table
into the thinking prompt.

Prompt-safe example:

```text
"The current exchange is starting to repeat. Prefer a small new angle,
clarifying question, or concrete next step if it fits the user's request."
```

Blocked example:

```text
"boredom_level is high; say you are bored."
```

### 5. Add M13 Boredom Event And Patch

Minimum event:

```text
M13BoredomEvaluationEvent:
  event_id
  turn_id
  turn_index
  boredom_level
  boredom_band
  novelty_proxy
  information_gain_proxy
  repetition_pressure
  progress_signal
  exploration_bias
  exploration_target
  evidence_refs
  prompt_safe_summary
```

Patch updates should write only compact boredom state and cooldown fields.

### 6. Preserve Direct Task Priority

Boredom must be suppressed or softened when:

- the user asks a direct factual or implementation question
- there is high conflict, repair, or safety pressure
- identity/entity binding needs clarification
- evidence judgment says assumptions are forbidden
- the current turn is the first turn

The reply should still answer the user first. Exploration may appear as a
short next-step suggestion, clarifying question, or reframing only when useful.

M13.1 inherits the M13.0 merge precedence. Exploration bias is lower priority
than safety, evidence boundaries, identity constraints, direct task handling,
and hard reply-contract rules.

## Suggested Tests

```text
test_boredom_low_on_first_turn_with_direct_task
test_boredom_rises_for_repeated_action_and_same_topic
test_new_information_gain_reduces_boredom
test_high_boredom_adds_exploration_guidance_not_visible_diagnostic
test_boredom_suppressed_under_repair_or_conflict_pressure
test_exploration_cooldown_prevents_every_turn_topic_shift
test_boredom_event_and_patch_are_auditable
test_boredom_multiturn_same_topic_raises_exploration_bias
test_drive_guidance_is_capped_to_six_prompt_safe_lines
```

The multi-turn boredom scenario should run 8-10 same-topic low-novelty turns
and assert that exploration bias rises, then drops or cools down when a new
topic fingerprint or high-information-gain turn appears.

## Acceptance Criteria

M13.1 is complete when the MVP chat path can detect stale low-gradient
interaction states, produce a bounded boredom proxy, convert it into
prompt-safe exploration guidance, and persist only compact audited boredom
state without overriding direct user tasks or exposing internal terminology.
