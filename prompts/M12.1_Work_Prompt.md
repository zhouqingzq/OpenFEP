# OpenFEP M12.1 Work Prompt

## Milestone

M12.1: Mechanistic Personality Model And Plain-Language Report

## Goal

Add a slow, trigger-driven personality / cognitive-style modeling layer on
top of the M12.0 identity-continuity module and the M11 user generative
model. The digital persona must be able to maintain, per `user_id`, a
bounded **mechanistic** account of how that user's prediction-and-strategy
system tends to run, and to produce a structured eight-step report when
explicitly requested or when accumulated evidence justifies a refresh.

M12.1 must:

- maintain a per-`user_id` `PersonalityProfile` that explains *how* this
  user tends to allocate attention, react emotionally, and choose actions
  under bounded resources, **mechanistically, not as labels**
- produce a `PersonalityReport` strictly following the eight-step flow
  defined below, with one structured field per step and a deterministic
  assembler
- enforce the seven analysis principles defined below in code, including
  a plain-language linter that blocks engineering jargon, clinical
  labels, moral verdicts, and chicken-soup filler from any user-facing
  surface
- fall back to `insufficient_evidence` for any step where the available
  material is too thin, rather than fabricate for the sake of completeness
- expose personality state to generation only through bounded evidence
  cards, never as raw report text

M12.1 must not:

- run the eight-step pipeline every turn (it is slow and trigger-driven)
- diagnose mental illness or assign personality-disorder labels
- emit moral judgements, chicken-soup prose, or coverage-padding
  inventions
- expose engineering vocabulary (prediction, model, prediction error,
  free energy, Bayesian, posterior, prior update) on any user-facing
  surface
- mutate M11 ledgers, M12.0 ledgers, anchored memory, or legacy memory
  rows
- bypass the cognitive loop, MemoryWriteIntent, or response evidence
  contract
- inherit, wrap, or recompose any function from
  `segmentum/memory_dynamics.py`, `segmentum/value_memory.py`, or
  `segmentum/memory_retrieval.py`
- become a second autonomous chat agent or run unbounded background
  reflection

## Stand-Alone Posture

M12.1 is a **stand-alone adjacent module** built on top of M12.0 and beside
M11. It reads M12.0 and M11 state read-only and writes only into its own
`segmentum/user_personality/` ledgers and reports.

Composition direction is strict:

```text
M11 produces per-user hypotheses, predictions, reliability, value cards.
M12.0 produces IdentityProfile, ContinuityCue, ConflictRecord,
       StrangenessSignal, EntityBindingContext.
M12.1 reads them and adds a mechanistic personality account and an
       eight-step report on top.
M12.1 NEVER writes back into M11 or M12.0 ledgers.
```

If M12.1 wants a behavior that lives in M11 or M12.0, copy a clean,
read-only adapter into `segmentum/user_personality/` instead of mutating
or composing across the boundary.

## Core Contract

M12.1 runs on **trigger**, not on every turn. Triggers are listed in
"Trigger And Scheduling Policy" below. On each trigger:

```text
trigger fires
-> M12.1 reads bounded read-only snapshots from M11 and M12.0
   (UserModel hypotheses, prediction_ledger summary, reliability_ledger,
    IdentityProfile, ContinuityCue list, StrangenessSignal history,
    relevant turn transcript window)
-> 8 step LLM extractor calls (one per step, strict schemas, NO floats)
   any step is allowed to return `insufficient_evidence`
-> deterministic PersonalityProfile patch composition
   (incremental update over existing per-user PersonalityProfile)
-> deterministic PersonalityReport assembly
   (pure function of PersonalityProfile + report-time hyperparams)
-> plain-language linter pass / fail
   (rejects engineering jargon, clinical labels, moral verdicts,
    chicken-soup filler, and DSM/ICD names)
-> if linter passes:
     emit M12.1 evidence cards to the generation pipeline
     emit PersonalityProfileUpdateEvent on the bus (informational only)
-> if linter fails:
     mark report `linter_failed`, do not surface to any prompt,
     log diagnostic for the developer
```

The system must keep these things separate at the boundary:

```text
evidence_quote_ref            # turn_id + quote_id from real turns
continuity_cue_ref            # ContinuityCue id from M12.0
m11_hypothesis_ref            # UserHypothesis id from M11 (read-only)
mechanism_claim               # short plain-language sentence + confidence_band
core_belief                   # one of {about_self, about_others, about_world}
defense_kind                  # enum (see Step 5 list)
loop_stage                    # one of the six core-loop stages
report_section                # one of step_1 .. step_8
report_status                 # draft | linter_failed | ready | stale | superseded
```

A `PersonalityProfile` is **not a fact** about the user. Every claim
inside it carries an evidence-ref chain and a confidence band, and is
labeled as `inferred_hypothesis` or `insufficient_evidence` unless
explicitly confirmed by the user with traceable evidence.

## Analysis Principles (Report Contract)

Every personality / cognitive-style output produced by M12.1 (the
PersonalityReport, every PersonalityProfile section, every evidence
card, every reply-policy reason) must obey the following principles.
Code paths that surface text MUST enforce them as deterministic checks
where feasible.

1. The analysed subject is a system running under limited resources.
   Its core driver is to keep narrowing the gap between what it
   internally expects and what it actually encounters, under bounded
   compute.
2. "Personality" is the **stabilised prior structure** this resource-
   bounded system has solidified through long interaction with its
   environment. These priors decide:
   - what the person tends to pay attention to
   - which situations make them anxious vs energised
   - whether they tend to explore or avoid
   - how they explain events, where they put their attention, how they
     protect themselves, how they handle relationships
3. Output must give a **mechanistic account** ("why does he act this
   way") rather than a **label** ("he is an introvert").
4. No chicken soup, no moral judgement, no vague filler like "this is
   a complex person".
5. **No clinical diagnoses.** Phrasings like "mechanistically similar
   to ..." or "there is a tendency toward ..." are allowed; pretending
   to be a clinician is not. DSM / ICD names are banned.
6. If evidence is insufficient, the output is literally
   `insufficient_evidence`. Do not invent for the sake of completeness.
7. **User-facing language constraint.** Anything that crosses into a
   prompt, a card's `content_summary`, a reply-policy reason, or a
   generated reply MUST use ordinary social language and MUST NOT use
   engineering vocabulary such as "predict", "prediction", "model",
   "prediction error", "free energy", "Bayesian", "posterior", "prior
   update", "predictive system", or their Chinese equivalents
   ("预测", "模型", "误差", "预测误差", "自由能", "贝叶斯", "后验",
   "预测系统"). Use ordinary words like 认为 (thinks), 不确定因素
   (uncertain factors), 过往经验 (past experience), 倾向于 (tends to),
   在意 (cares about), 害怕 (fears). Internal engineering docs,
   ledger code, log lines, and acceptance artifacts may keep technical
   vocabulary; the constraint applies only to user-facing surfaces.

## Analysis Flow (Eight-Step Report Pipeline)

The personality report generator MUST follow this eight-step flow. The
flow is a single deterministic orchestration: each step is one bounded
LLM extractor call producing a structured object, and the final report
is a pure function of those eight objects plus `hyperparams.py`.

### Step 1: Overall Personality Summary

A short plain-language summary that acts as the navigation map for the
rest of the report. It must answer:

- What kind of inner world does this person live in by default? How do
  they assume the world works?
- What core strategies have they developed to get along inside that
  world?
- When those strategies work well, what does that look like? When they
  fail or "glitch", what does that look like?
- What is this person's central tension? (e.g. wants closeness but
  fears being hurt; wants approval but refuses to look weak.)

Plain-language constraint (Principle 7) applies. The output of this
step is a single bounded string field plus `evidence_quote_refs`.

### Step 2: Evidence Extraction

Pull key signals from the available material:

- recurring emotions, vocabulary habits, shifts in tone
- default attributions for other people's intent
- behaviour choices under conflict or pressure
- topics that trigger sensitivity vs avoidance
- preferred relational position (lead / submit / avoid / please / observe)

List core evidence as bounded structured items with short quote
references. Exhaustive enumeration is not required; the goal is to
surface the main tensions.

### Step 3: How This Person's Prediction System Runs

In plain language (Principle 7), answer:

- What state does this person most want to maintain? (being loved,
  respected, safe, in control, free, understood, ...)
- What do they most fear? (being abandoned, denied, losing control,
  being seen through, meaninglessness, being used, ...)
- What signals are they hypersensitive to? What signals do they
  selectively ignore?
- When something happens, what is their first interpretive reflex?
  (assume the worst, rationalise, blame self, blame outside, ...)

The internal field name may stay technical, but the surface text must
not contain the words "predict / model / error / free energy" or their
Chinese equivalents.

### Step 4: Core Beliefs

Three short sentences capturing this person's default assumptions:

- About themselves: "I am ..."
- About others: "Other people usually ..."
- About the world: "The world is ..."

Each belief carries:

```text
content_summary               # plain language, <= 80 chars
evidence_quote_refs           # bounded list
confidence_band               # low | med | high
```

### Step 5: Emotion Pattern And Defenses

- `dominant_emotional_baseline` (enum or plain-language token)
- typical response under threat or conflict

Allowed `defense_kind` enums (pick those that fit; do not force coverage):

```text
intellectualisation
avoidance
pleasing
attack
self_deprecation
rationalisation
control
projection
cold_detachment
```

For each chosen defense, output a structured item:

```text
defense_kind
protects_what
short_term_benefit
long_term_cost
evidence_quote_refs
confidence_band
```

### Step 6: Relationship Patterns

- typical role in close relationships and recurring loops
- typical conflict style (enum from `confront | yield | cold_war | argue | flee`)
- type of person they are most drawn to, and why
- type of person they most often clash with, and why

Each item is structured and carries `evidence_quote_refs` and
`confidence_band`.

### Step 7: Core Loop

A single closed loop with exactly six stages:

```text
trigger_event
-> interpretation
-> emotion
-> action
-> outcome
-> belief_reinforcement
```

Each stage is a bounded plain-language string with at least one
`evidence_quote_ref`. The assembled loop is what makes "why does this
person keep meeting the same kind of problem" visible.

### Step 8: Growth Hints

Only if evidence is sufficient:

- `stable_parts` (short-term unlikely to shift)
- `fragile_spots` (collapse easily when touched)
- `soft_spots` (most likely to update in a safe environment)
- `communication_styles_likely_accepted`
- `communication_styles_that_trigger_defenses`

If evidence is insufficient, the output is literally the token
`insufficient_evidence` plus a one-line `reason` string. Do not
fabricate growth lines to fill the section.

Insufficient-evidence policy (cross-step):

- Each step is allowed to return its own `insufficient_evidence` marker
  if the bounded snapshot does not support that step.
- A report whose Step 1 returns `insufficient_evidence` is marked
  `report_status = stale` and is not surfaced to generation.
- A report whose Steps 4 or 7 return `insufficient_evidence` may still
  be surfaced, but its evidence cards must carry `permitted_use =
  strategy_only` rather than `cautious_hypothesis`.

## Trigger And Scheduling Policy

M12.1 is slow on purpose. The eight-step pipeline runs only when one of
these triggers fires, gated by `hyperparams.py`:

```text
explicit_request                  # developer / operator / sleep-consolidation hook
turn_count_cadence                # every N turns since last successful report
new_evidence_threshold            # M new bounded ContinuityCues since last report
identity_state_change             # corroborated <-> conflicted on a high-band binding
strangeness_followup              # M12.0 strangeness_band stayed `high` for K turns
session_boundary                  # end-of-session / inactivity flush
```

Hard rules:

- The pipeline never runs more than `max_personality_runs_per_hour`
  for the same `user_id`.
- A failed plain-language linter run does not count toward the cadence
  counter (so a single bad output cannot starve the next attempt).
- Two consecutive `insufficient_evidence` Step-1 results suspend the
  cadence trigger for that `user_id` until new evidence arrives.

## Required Work

### 1. Define PersonalityProfile

Create `segmentum/user_personality/personality_profile.py`. Bounded,
serialisable, explicit. Per `user_id`.

The profile has **one structured field per analysis step**, each
independently versioned:

```text
user_id
display_name_hint
step_1_summary                       # PersonalitySummary or insufficient_evidence
step_2_evidence                      # list[EvidenceItem]
step_3_prediction_system_account     # PredictionSystemAccount or insufficient_evidence
step_4_core_beliefs                  # CoreBeliefSet or insufficient_evidence
step_5_emotion_and_defenses          # EmotionAndDefenses or insufficient_evidence
step_6_relationship_patterns         # RelationshipPatterns or insufficient_evidence
step_7_core_loop                     # CoreLoop or insufficient_evidence
step_8_growth_hints                  # GrowthHints or insufficient_evidence
section_freshness                    # per-step turn_id of last successful update
hyperparams_version
last_full_report_turn_id
last_full_report_status              # draft | linter_failed | ready | stale | superseded
```

Each section dataclass carries `evidence_refs`, `confidence_band`,
`hyperparams_version`, and `last_updated_turn_id`.

Acceptance:

- PersonalityProfile distinguishes a section that has never been
  computed from a section that returned `insufficient_evidence`.
- PersonalityProfile can be saved and loaded as plain JSON without
  reading prompt text or trace dumps.
- A single transcript snippet never promotes a section confidence band
  from `low` to `high` in one shot; band promotion requires
  multi-turn evidence and band-promotion thresholds in `hyperparams.py`.

### 2. Define PersonalityReport

`segmentum/user_personality/personality_report.py`.

The report is a **view object** assembled from PersonalityProfile plus
report-time hyperparams. It is not a separate source of truth.

Minimum fields:

```text
report_id
user_id
turn_id
hyperparams_version
report_status                        # draft | linter_failed | ready | stale | superseded
sections                             # ordered list of section views (step_1 .. step_8)
linter_findings                      # list of LinterFinding (token, location, reason)
trigger_kind                         # enum from Trigger And Scheduling Policy
prior_report_id                      # for traceability
```

Acceptance:

- Two report assembles from the same PersonalityProfile and the same
  hyperparams_version produce byte-identical reports.
- A section that contains `insufficient_evidence` is rendered as an
  explicit marker, not as a blank or filler paragraph.
- A `linter_failed` report is never returned in the "ready" channel
  and never becomes input to any prompt.

### 3. Implement Eight-Step Orchestrator

`segmentum/user_personality/personality_orchestrator.py`. Deterministic.

The orchestrator runs steps 1..8 in order, building a bounded snapshot
for each step from:

- the current trigger context (turn_id, user_id, trigger_kind)
- a bounded turn-window of transcript references (turn_ids + quote_ids
  only; no full transcripts in the snapshot beyond the configured cap)
- the active M12.0 IdentityProfile, the K most recent ContinuityCues,
  and the most recent ConflictRecord summary (read-only)
- the active M11 UserModel hypotheses (id + content_summary +
  confidence_band) and the most recent reliability_ledger summary
  (read-only)
- the existing PersonalityProfile section (so each step can do an
  **incremental** update rather than a from-scratch rewrite)

Each step calls its corresponding `llm_step_extractors.py` function,
validates the strict schema, and either:

- updates the PersonalityProfile section deterministically, or
- marks the section as `insufficient_evidence` if the extractor returns
  that token, or
- aborts the run with `report_status = linter_failed` if any step's
  output violates the linter.

Acceptance:

- Replaying the same recorded step-extractor fixtures yields a
  byte-identical PersonalityProfile patch and a byte-identical
  PersonalityReport.
- A step that returns `insufficient_evidence` does not corrupt the
  prior section state; the prior section remains accessible.
- The orchestrator never reads files outside the bounded snapshot it
  was given.

### 4. Implement LLM Step Extractors With Strict Schemas

`segmentum/user_personality/llm_step_extractors.py`. One function per
step. Each function:

- accepts a bounded snapshot (turn-window quote ids, M11 hypothesis
  summaries, M12.0 cue summaries, current PersonalityProfile section)
- issues exactly one LLM call with a strict JSON schema for that step
- forbids floats anywhere in the output
- forbids any token from the broader user-facing token list (see
  `plain_language_linter.py`)
- forbids DSM-style labels and moral language
- requires `evidence_quote_refs` for every claim
- requires `confidence_band` for every belief / hypothesis
- accepts a single sentinel return: `insufficient_evidence` with a
  short `reason` string

Step-specific schemas (sketch):

- Step 1: `{ summary: string, evidence_quote_refs: [string], confidence_band: ... }`
- Step 2: `{ evidence_items: [{ kind, content_summary, evidence_quote_refs, confidence_band }] }`
- Step 3: `{ wants, fears, hypersensitive_to, ignores, default_interpretation, evidence_quote_refs, confidence_band }`
- Step 4: `{ about_self, about_others, about_world }` each with `content_summary, evidence_quote_refs, confidence_band`
- Step 5: `{ dominant_emotional_baseline, threat_response, defenses: [{ defense_kind, protects_what, short_term_benefit, long_term_cost, evidence_quote_refs, confidence_band }] }`
- Step 6: `{ close_relationship_role, recurring_loop_summary, conflict_style, drawn_to: { kind, why }, clashes_with: { kind, why }, evidence_quote_refs, confidence_band }`
- Step 7: `{ trigger_event, interpretation, emotion, action, outcome, belief_reinforcement, evidence_quote_refs, confidence_band }`
- Step 8: `{ stable_parts, fragile_spots, soft_spots, communication_styles_likely_accepted, communication_styles_that_trigger_defenses, evidence_quote_refs, confidence_band }` or `insufficient_evidence`

The extractor prompts must:

- pass only the bounded snapshot
- forbid the LLM from inventing turn ids, quote ids, hypothesis ids,
  or continuity-cue ids that are not in the snapshot
- reject any output containing fields not in the schema
- reject any numeric float anywhere in the output

The extractor never decides write / promote / decay / surface. Those
decisions belong to the deterministic layer.

### 5. Implement Plain-Language Linter

`segmentum/user_personality/plain_language_linter.py`.

This linter extends the M12.0 prompt-safety linter with the broader
user-facing token list defined in `hyperparams.py`:

```text
forbidden_user_facing_tokens_extra = [
  "predict", "prediction", "model", "prediction error",
  "free energy", "Bayesian", "posterior", "prior update",
  "predictive system", "compress error",
  "预测", "模型", "误差", "预测误差", "自由能", "贝叶斯", "后验",
  "预测系统",
]

forbidden_clinical_label_tokens = [
  "narcissist", "narcissistic personality",
  "borderline", "borderline personality",
  "autistic", "schizoid", "schizotypal",
  "bipolar", "ADHD", "PTSD", "OCD",
  "depression disorder", "anxiety disorder",
  "自恋型", "边缘性", "自闭", "抑郁症", "焦虑症",
  "双相", "强迫症", "PTSD", "ADHD",
]

forbidden_moral_or_chicken_soup_tokens = [
  "should",
  "everything will be fine",
  "needs to love himself",
  "needs to love herself",
  "应该", "本该", "都会好起来", "需要爱自己",
]
```

The linter operates on any field of the report or evidence card that
crosses into a prompt or a generated reply (`content_summary`,
`reason`, `why_retrieved`, etc). It does not operate on internal log
lines, ledger keys, schema enum names, or acceptance-artifact files.

A linter finding is structured:

```text
linter_finding
  token
  section                            # step_1 .. step_8, or `card.content_summary`
  rule                               # engineering_jargon | clinical_label | moral | chicken_soup | dsm_icd
  raw_quote                          # bounded
```

Acceptance:

- The linter catches every token in every forbidden list, in both
  English and Chinese variants.
- The linter never inspects internal log lines, acceptance artifacts,
  or schema enum names.
- A `linter_failed` report is never returned in the "ready" channel
  and never becomes input to any prompt.
- The linter is a pure function over text and the forbidden token sets
  in `hyperparams.py`.

### 6. Implement Trigger Policy

`segmentum/user_personality/trigger_policy.py`. Deterministic.

Takes (current turn_id, user_id, M12.0 state summary, last report
state, cadence counters) and returns either `no_run` or `run(kind)`.

Hard rules:

- never more than `max_personality_runs_per_hour` for the same `user_id`
- two consecutive `insufficient_evidence` Step-1 results suspend the
  cadence trigger for that user until new evidence arrives
- `explicit_request` always overrides cadence suppression but still
  respects the per-hour cap
- a run is gated to weekday / weekend windows only if configured in
  `hyperparams.py`; default is no calendar gating

Acceptance:

- Trigger decisions are pure functions of the listed inputs and
  `hyperparams.py`.
- Trigger decisions are byte-identical across runs given the same
  inputs.

### 7. Provide Evidence Cards To Generation

`segmentum/user_personality/evidence_cards.py`.

Convert the latest `ready` PersonalityProfile sections into cards
compatible with `segmentum/memory_evidence.py::MemoryEvidence`.

Each card exposes only:

```text
memory_id                     # PersonalityProfile section id
memory_class                  # one of the existing MemoryEvidence classes
content_summary               # bounded, plain-language, Principle-7 compliant
confidence_band               # low | med | high (mirrored to float for compat only)
permitted_use                 # explicit_fact | cautious_hypothesis | strategy_only | forbidden
section_kind                  # step_1 .. step_8
why_retrieved                 # short string, NOT internal scores
```

Hard rules:

- Cards never expose the raw `surprise_explanation`, raw step
  schemas, or raw linter findings; those are diagnostic only.
- Cards never expose DSM-style labels, moral verdicts, chicken-soup
  filler, or engineering vocabulary.
- A section in `insufficient_evidence` state never produces a card
  with `permitted_use = explicit_fact`; the maximum is
  `strategy_only`.
- A section in `linter_failed` state never produces a card.

Acceptance:

- Generation receives compact M12.1 cards, not raw PersonalityProfile
  dumps, not LLM extractor JSON, not linter diagnostics.
- Cards are ordered by `permitted_use` priority, then `confidence_band`,
  then `section_kind`.

### 8. Wire M12.1 Into The MVP Runtime

`segmentum/user_personality/m12_1_runtime.py`.

Per-tick responsibilities:

```text
1. Consult trigger_policy. If `no_run`, return immediately; M12.1
   contributes only previously-stored evidence cards (if any).
2. On `run(kind)`, build bounded snapshots from M11 and M12.0
   read-only state plus the configured turn window.
3. Run the eight-step orchestrator.
4. Update PersonalityProfile deterministically (incremental section
   patches; never wipe a section that just returned
   `insufficient_evidence`).
5. Assemble PersonalityReport. Run plain-language linter.
6. If linter passes: emit M12.1 evidence cards and publish
   `PersonalityProfileUpdateEvent` (informational only) on the bus.
7. If linter fails: mark report `linter_failed`, do not surface cards,
   log diagnostic.
8. Trace effects (which trigger fired, which sections updated, which
   linter findings, which cards were emitted).
```

Acceptance:

- M11 and M12.0 state remain read-only across this orchestration.
- All durable M12.1 writes are auditable (turn id, trigger kind,
  hyperparam version).
- Disabling M12.1 (config flag) yields byte-identical chat outputs to
  an `M11 + M12.0`-only baseline on a fixed transcript.

## LLM Use Boundary

**The LLM is allowed to produce:**

- enum-typed classifications (defense_kind, conflict_style, kind, ...)
- bounded structured lists per step schema
- ordinal bands (low | med | high) for confidence
- short plain-language strings that pass the linter
- evidence quote refs that already exist in the bounded snapshot
- the sentinel return `insufficient_evidence` with a short `reason`

**The LLM is forbidden to produce:**

- continuous floats anywhere in any field of any output
- decisions about promotion / quarantine / decay / cadence / surface
- card emission decisions
- trigger decisions
- DSM / ICD names, clinical diagnoses, or psychiatric labels
- moral verdicts ("should", "ought", "shouldn't", "本该", "应该")
- chicken-soup phrasing
- engineering vocabulary in any user-facing field (Principle 7)
- new turn ids / quote ids / hypothesis ids / cue ids that are not in
  the bounded snapshot

The deterministic layer (orchestrator, linter, trigger_policy,
profile patcher, report assembler, evidence_cards) must run end-to-end
on stubbed step-extractor outputs without invoking any LLM. This is
mandatory; it is what makes M12.1 testable and reproducible.

## Determinism Reliability Contract

The deterministic layer (`personality_orchestrator` patcher,
`personality_report` assembler, `plain_language_linter`,
`trigger_policy`, `evidence_cards`) must satisfy:

1. **Pure functions over structured inputs only.**
   - Inputs are enums, counts, section state, hyperparams, and
     read-only state snapshots from M11 / M12.0.
   - No raw text inspection in decision logic. The plain-language
     linter is the only path that inspects text, and it only operates
     on assembled report / card output, not on decisions about whether
     to run or surface.
   - Property test: identical structured inputs with different
     `evidence_text` produce identical decisions, identical patch
     outputs, identical linter findings (up to token presence), and
     identical card sets.

2. **All constants live in `hyperparams.py`** with per-field docstrings
   stating operational meaning. No inline magic numbers.

3. **Every update is one of:**
   - replace a section with a new section produced by its step
   - mark a section as `insufficient_evidence`
   - keep a section unchanged
   - bump `section_freshness` counters
   - bounded clamp with documented per-run delta cap on band promotion

4. **Property tests required per dynamics function:**
   - boundedness in declared range / enum set
   - per-`user_id` isolation (a run for user A never moves user B)
   - per-run delta bound on confidence band promotions
   - no-text-dependency in patch decisions (only the linter inspects text)
   - "insufficient_evidence resilience" (returning the sentinel from
     any step never corrupts prior sections)

5. **Replay fixtures per step:**
   - committed pairs of (input step-extractor JSON, expected section
     patch JSON, expected linter findings)
   - any change that alters a trace requires fixture update and a
     `rationale.md` entry

6. **Calibration audit:**
   - at least 6 hand-authored multi-turn scenarios:
     `rich_evidence_full_report`,
     `sparse_evidence_insufficient_at_step_8`,
     `engineering_jargon_caught_by_linter`,
     `dsm_label_caught_by_linter`,
     `moral_verdict_caught_by_linter`,
     `roleplay_does_not_destabilise_profile`
   - documented per-scenario success thresholds
   - audit report shipped as part of M12.1 acceptance artifact

The deterministic layer is "reliable" iff (1)-(5) pass and (6) meets
documented thresholds. M12.1 acceptance must cite the audit report.

## Non-Interference Guarantee

M12.1 must not change observable behavior of the existing chat memory
path, the M11 user-model path, or the M12.0 identity-continuity path
when the M12.1 module is disabled. When M12.1 is enabled, it may change
the visible reply only through an explicit M12.1 evidence-card effect.

Required:

- A configuration flag (default off in legacy personas, default on in
  new personas) gates all M12.1 reads, writes, runs, and card emission.
- When the flag is off, the runtime behaves exactly as `M11 + M12.0`
  alone would behave.
- When the flag is on but no PersonalityProfile exists for the user,
  the runtime initialises a clean profile and waits for a trigger;
  it does not produce a report on the first turn.
- M12.1 never deletes, edits, or re-scores any row owned by
  `MemoryStore`, `LongTermMemory`, the existing anchored memory layer,
  any M11 ledger, or any M12.0 ledger.

Acceptance:

- An existing persona with M12.1 flag off produces byte-identical
  chat outputs to its `M11 + M12.0`-only baseline on a fixed
  transcript when run with a rule-mode generator, stub generator, or
  recorded LLM fixture.
- A persona with M12.1 flag on produces PersonalityProfile state and
  evidence cards, never modifies legacy memory rows or M11 / M12.0
  ledger rows, and changes user-visible replies only when an active
  M12.1 evidence card is consumed by the existing evidence pipeline.

## Non-goals

- Do not run the eight-step pipeline every turn.
- Do not import or wrap any function from `segmentum/memory_dynamics.py`,
  `segmentum/value_memory.py`, or `segmentum/memory_retrieval.py`.
- Do not mutate any M11 or M12.0 module state from inside M12.1.
- Do not change retrieval ranking for the existing chat memory path.
- Do not migrate or rewrite legacy memory rows.
- Do not produce DSM / ICD names, clinical diagnoses, moral verdicts,
  or chicken-soup prose anywhere a user-facing surface can read it.
- Do not let the LLM produce any continuous float in any output field.
- Do not let the LLM decide cadence, triggers, promotion, decay, or
  card surfacing.
- Do not leak raw extractor JSON, linter findings, profile dumps, or
  engineering vocabulary into user-visible replies.
- Do not build a surveillance profile or infer sensitive traits
  without user-provided evidence and permitted use.
- Do not run unbounded background LLM reflection or a second
  autonomous agent.

## Completion Criteria

M12.1 is complete when:

- the digital persona maintains a bounded, auditable
  `PersonalityProfile` per `user_id` whose updates are produced by a
  deterministic eight-step orchestrator over LLM-extracted enums and
  bounded structured lists.
- `PersonalityReport` is a pure-function view of `PersonalityProfile`
  and replays byte-identically from recorded extractor fixtures.
- the plain-language linter (Principle 7), the clinical-label ban
  (Principle 5), the moral-verdict ban (Principle 4), and the
  chicken-soup ban (Principle 4) are enforced in code and block any
  failed report from reaching a user-facing surface.
- `insufficient_evidence` is a first-class output (Principle 6); no
  step fabricates content to fill empty material.
- the deterministic layer satisfies the Determinism Reliability
  Contract.
- the Non-Interference Guarantee holds against an `M11 + M12.0`-only
  baseline.
- the calibration audit report passes documented thresholds for all
  six scenarios listed above.
- no symbol from `memory_dynamics.py`, `value_memory.py`, or
  `memory_retrieval.py` appears in any M12.1-owned import statement,
  and no mutation method of any `segmentum/user_model/` or
  `segmentum/user_continuity/` module is called from any
  M12.1-owned module.
