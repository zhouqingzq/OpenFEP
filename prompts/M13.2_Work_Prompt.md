# OpenFEP M13.2 Work Prompt

## Milestone

M13.2: MVP-Local Affective Reward Proxy And Tolerance

## Goal

Add a bounded affective reward proxy to the MVP-local M13 layer. This is not
subjective pleasure. It is an engineering signal for how much a path currently
appears to reduce local free-energy-like tension after accounting for
prediction, baseline, adaptation, opponent pressure, and information gain.

The target model:

```text
external / social / task reward proxy
- predicted reward
- reward_baseline
- tolerance_discount
- opponent_pressure
+ information_gain
+ relief_from_uncertainty
= net_affective_reward_proxy
```

Repeated successful paths should become more predictable and therefore less
surprising. Their subjective-looking positive proxy can fall while their
behavioral pull remains high because habit precision and control-cost discount
remain high.

This milestone should make the system capable of the pattern:

```text
less fresh / less rewarding
but still easier to choose
```

without claiming addiction is fully implemented.

## Dependencies

M13.2 builds on:

- M13.0 behavioral pull
- M13.1 boredom and exploration bias
- existing MVP memory, habit, relationship value, and post-reply observer
  surfaces

M13.2 should also add next-turn settlement for prior action proxies. This
helps turn M13 from same-turn prompt guidance into a small feedback system.

## Non-Goals

- Do not claim real pleasure, real craving, clinical addiction, or subjective
  experience.
- Do not implement a mental-health diagnosis or user addiction classifier.
- Do not expose "reward", "tolerance", "opponent process", or similar
  engineering terms to visible replies.
- Do not let reward proxy override safety, privacy, evidence, or direct user
  request handling.
- Do not add background autonomy; that remains M13.3.

## Required Work

### 1. Extend M13 State With Reward Proxy Dynamics

Add:

```text
m13_drive_state:
  affective_reward_proxy:
    reward_baseline
    opponent_strength
    tolerance_by_path
    predicted_reward_by_path
    last_net_reward_proxy
    last_relief_proxy
    last_information_gain_proxy
    pending_settlements
```

Hard bounds:

```text
tolerance_by_path: 64 rows, evict oldest last_updated_turn
pending_settlements: 8 rows
pending settlement TTL: 5 turns
reward history window: 24 compact rows
reason_codes per settlement: 8
```

Per-path tolerance fields:

```text
path_id
action
topic_fingerprint
support_count
predicted_reward
tolerance
opponent_strength
last_net_reward_proxy
last_updated_turn
```

### 2. Define Reward Proxy Components

Positive proxy sources:

- expectation confirmed
- uncertainty reduced
- user correction handled cleanly
- evidence stance became clearer
- relationship value constraint satisfied
- reply validation unchanged
- post-reply observer did not require repair
- memory write accepted with real evidence
- user response in next turn indicates uptake, clarification, or continuity

Negative proxy sources:

- expectation violated
- user correction indicates prior miss
- safety rewrite occurred
- post-reply observer required repair
- evidence judgment blocked assumptions
- repeated same path with low novelty
- conflict or social risk increased

Each source should be bounded and named in diagnostics.

### 3. Add Next-Turn Settlement

At the start of `run_turn`, before conscious planning, settle pending M13
action traces from the previous turn.

Pending trace fields:

```text
pending_id
prior_turn_index
prior_action
prior_topic_fingerprint
prior_reply_summary
predicted_relief
predicted_reward
evidence_refs
expires_after_turns
```

Settlement uses current-turn signals:

- current `user_text`
- observation channels from `ChatInterface._send_mvp`
- explicit correction markers
- expectation results from conscious plan when available
- safety / reply validation from prior turn diagnostics if stored

Settlement output:

```text
settlement_id
pending_id
observed_reward_proxy
prediction_error_proxy
relief_proxy
information_gain_proxy
outcome_band
reason_codes
```

If signals are insufficient, write an `uncertain` settlement rather than
fabricating success or failure.

Startup / load behavior:

- clear stale settlements whose TTL has expired
- mark cleared items with a compact suppression or expiration reason in the
  audit log when possible
- never let an old pending settlement apply to an unrelated later session

### 4. Compute Net Affective Reward Proxy

Suggested shape:

```text
net_affective_reward_proxy =
  observed_reward_proxy
+ relief_proxy
+ information_gain_proxy
- predicted_reward
- reward_baseline
- tolerance
- opponent_strength
```

Update dynamics:

```text
predicted_reward_by_path moves toward observed_reward_proxy
reward_baseline moves slowly toward recent observed rewards
tolerance rises after repeated high-pull / low-novelty success
opponent_strength rises after repeated intense reward proxy or repeated repair cost
```

All rates must be small, bounded, and documented in constants.

Default update-rate constraints:

```text
maximum_single_turn_predicted_reward_delta: 0.05
maximum_single_turn_tolerance_delta: 0.04
maximum_single_turn_opponent_strength_delta: 0.04
minimum_settlement_confidence_for_positive_update: 0.60
```

The constants may be tuned, but tests must fail if a single turn can cause a
large jump without an explicit bound change.

### 5. Connect Reward Proxy To Behavioral Pull Without Collapsing Them

M13.2 must preserve the distinction:

```text
net_affective_reward_proxy  # freshness / relief / positive deviation
behavioral_pull             # habit precision / cue precision / low control cost
```

Rules:

- high reward proxy may increase future path confidence a little
- high tolerance may reduce reward proxy
- high habit precision may keep behavioral pull high even when reward proxy
  falls
- high opponent pressure should increase caution or exploration, not visible
  self-punishment language

This is the core acceptance pattern:

```text
same repeated path:
  reward proxy decreases
  behavioral pull can stay stable or increase
```

This pattern must be covered by a numerical multi-turn test. Run 8-10 repeated
same-path turns and assert:

```text
net_affective_reward_proxy decreases after the path becomes predicted
behavioral_pull for the same action does not decrease over the same window
tolerance or predicted_reward explains the reward-proxy drop in diagnostics
```

### 6. Prompt-Safe Control Guidance

Allowed guidance:

```text
affective_drive_guidance:
  path_feels_stale_proxy
  relief_seeking_tendency
  novelty_needed
  caution_about_repeating_easy_path
  ordinary_language_hint
```

Blocked visible/internal leakage:

```text
"pleasure decreased"
"tolerance rose"
"opponent process is active"
"I am addicted to this behavior"
```

UI-visible diagnostics and dashboard labels must also avoid these names:

```text
reward
tolerance
opponent process
addiction
craving
behavioral_pull
```

Developer logs and JSON traces may keep engineering names, but prompt-facing
and UI-facing surfaces must use ordinary language such as "path is getting
stale", "repeat tendency", "needs a fresher angle", or "easy path caution".

## Acceptance Criteria

M13.2 is complete when the MVP chat path can maintain a bounded affective
reward proxy, settle prior action outcomes, model tolerance-like adaptation,
and demonstrate through tests that repeated paths may become less positively
deviant while remaining behaviorally easier to choose.
